Deployment

What makes deploying a model hard, the patterns for rolling it out safely, and how to monitor and maintain it once live traffic is turned on.

1 · Why Deployment Is Hard

🚀 Two Categories of Challenge

Deploying a model is one of the most exciting moments of any project — but plenty can go wrong. The challenges split into two broad categories:

Category 1

Machine Learning / Statistical Issues

Will the model keep performing once the world changes? This is where concept drift and data drift live.

Category 2

Software Engineering Issues

How do you actually build the prediction service — real-time vs batch, cloud vs edge, latency, throughput, logging, security?

Deploying a system requires two broad sets of tasks: writing the software to run the model in production, and monitoring & maintaining the system over time — especially in the face of concept drift and data drift.
2 · Concept Drift & Data Drift

📉 When the Data Changes After Deployment

Loosely, drift means: what if your data changes after the system has already been deployed? A vision model trained to detect scratches under one set of factory lighting conditions degrades when the lighting changes. A speech system trained on purchased data plus opt-in historical user data — validated on the last few months of audio — can still degrade once a new phone with a different microphone appears, or as language slowly evolves.

Change can be gradual (English vocabulary shifts slowly) or sudden (a shock to the system). When COVID-19 hit, purchase patterns changed overnight — people who rarely shopped online suddenly did — and many credit-card anti-fraud systems broke, forcing teams to scramble for new data and retrain.

Data Drift

The input distribution X changes

A new politician or celebrity suddenly gets mentioned far more often; people start building larger or smaller houses, so the distribution of house sizes shifts. The mapping from X to Y is unchanged.

Concept Drift

The desired mapping X → Y changes

The same surprising purchases that meant fraud before COVID-19 are no cause for alarm after. Or: due to inflation, the same size house now commands a higher price.

When you deploy, one of the most important tasks is to detect and manage both concept drift (the definition of Y given X changes) and data drift (the distribution of X changes, even if the mapping does not).
3 · The Software Engineering Checklist

🛠️ Design Choices for a Prediction Service

A prediction service takes queries x and outputs predictions y. There are many ways to implement it — work through this checklist to make the right choices for your application.

Real-time or batch?

Speech recognition needs a response in well under a second. A hospital health-records analysis can run as an overnight batch process.

Cloud or edge?

Cloud gives more compute for accuracy; edge keeps working when the network is down (cars, factories, offline mobile, in-browser).

Compute resources

A model trained on a powerful GPU may need compressing if you cannot afford equally powerful GPUs (or enough CPU/memory) for serving.

Latency & throughput (QPS)

A 500 ms budget might allow only 300 ms for recognition. Throughput is the queries per second you must handle given your compute.

Logging

Log as much data as possible for analysis, review, and as future training data to retrain the algorithm.

Security & privacy

Required levels vary enormously. Patient records demand high security; design the appropriate level based on data sensitivity and regulation.

The practices for the very first deployment differ from those for updating or maintaining a system already in production — but the first deployment means you are only about halfway there.
4 · Common Deployment Patterns

🐤 Rolling Out Without Hoping for the Best

The best way to deploy is rarely to switch the model on and hope. Use cases fall into three buckets — offering a new product/capability, automating or assisting a task a person already does, and replacing a previous ML system with a better one. Across all of them, two recurring themes appear:

  • Gradual ramp up with monitoring — send a little traffic to an unproven algorithm, watch it, then increase the percentage as confidence grows.
  • Rollback — if the algorithm misbehaves, revert to the previous system (if one existed).

Shadow Mode

The learning algorithm runs in parallel with the existing decision-maker (a human inspector or an older system), but its output is not used for any real decision yet. A human inspecting phones for scratches still makes the call; the algorithm's predictions are logged alongside the human's. By sampling and comparing, you gather data to verify accuracy before letting it make any real decision.

Canary Deployment

When ready to let the algorithm make real decisions, roll it out to a small fraction of traffic (maybe 5% or less). Mistakes then affect only a small slice, giving you room to monitor and ramp up gradually as confidence grows. The name echoes the "canary in a coal mine" — an early warning before large consequences.

Blue / Green Deployment

A router sends images to a prediction service. The blue version is the old software; the green version is the new one. To switch, you reconfigure the router to send traffic from blue to green. Its advantage is easy rollback: point the router back at blue (kept running) the moment something goes wrong. Traffic can switch 100% at once, or be moved over gradually.

Camera Software

Collects phone images in the factory.

›
Router

Routes images to the blue or the green version.

›
Blue (old)

Previous prediction service — kept alive for rollback.

›
Green (new)

Newly implemented prediction service.

Each of these patterns takes substantial software to execute. MLOps tools can help implement them — or you can build it yourself.
5 · The Degree of Automation

🎚️ Deployment Is Not Zero-or-One

Rather than treating deployment as deploy / don't-deploy, design for the appropriate degree of automation. Visual inspection of phones spans a full spectrum from human-only to AI-only:

Least

Human Only

No automation — people make every decision.

Shadow

Shadow Mode

Algorithm predicts but its output is not used.

HITL

AI Assistance

Human decides; AI highlights regions to look at. UI design is critical.

HITL

Partial Automation

AI decides when confident; unsure cases go to a human.

Most

Full Automation

The algorithm makes every decision.

← only human decisionsonly AI decisions →

Projects often start on the left and gradually move right — but you do not have to reach full automation. Partial automation is a great design point when the algorithm is not good enough for full automation: confident predictions are taken automatically, and the small subset of uncertain cases go to a human, whose judgment is also valuable feedback to improve the algorithm.

AI assistance and partial automation are both human-in-the-loop deployments. Consumer internet (web search, online speech) usually needs full automation — you cannot have a person in the loop on every query. But outside consumer software, such as factory inspection, a human-in-the-loop design is often the best choice.

6 · Monitoring a Deployed System

📊 Dashboards, Metrics & Alarms

The most common way to monitor a system is a dashboard tracking how it does over time. To decide what to track, sit down with your team and brainstorm everything that could go wrong — then, for each, pick a statistic or metric that would detect it. Start with a relatively large set of metrics and prune the unhelpful ones over time.

Software Metrics

Memory, compute, latency, throughput, server load — they monitor the health of the software implementation. Many MLOps tools track these out of the box.

Input Metrics — has X changed?

  • Average input length (seconds of audio) and average input volume.
  • Number or percentage of missing values (very common for structured data).
  • Average image brightness, if lighting conditions might change.

Output Metrics — has Y (or what follows) changed?

  • How often the system returns null / the empty string (thinks nothing was said).
  • How often a user does two near-identical searches in a row (likely a misrecognition).
  • How often a user switches from speech to typing (a sign of frustration / degrading performance).
  • Coarse health signals like click-through rate (CTR).

Input and output metrics are application-specific, so MLOps tools usually need configuring to track them.

Just like modeling, deployment is iterative. The first deployment plus dashboards is only the start — real traffic enables performance analysis, which drives updates and better monitoring. It typically takes a few tries to converge on the right set of metrics.

Once metrics are chosen, set thresholds for alarms — e.g. notify the team if server load exceeds 0.91, or if the fraction of non-null outputs or missing values crosses a threshold. Adapt metrics and thresholds over time so they flag the most relevant concerns.

When something goes wrong: a software issue (server load too high) means changing the implementation; an accuracy issue means updating the model. Models can be retrained manually (an engineer retrains, runs error analysis, then pushes) or automatically. Manual retraining is far more common today — developers are reluctant to fully automate pushing new models — though automatic retraining does happen, especially in consumer internet.

7 · Monitoring Machine Learning Pipelines

🔗 When It Is More Than One Model

Many AI systems are not a single model but a pipeline of steps. Mobile speech recognition typically routes audio through a VAD (Voice Activity Detection) module that detects whether anyone is speaking — clipping the audio to just the speech and streaming only that to the cloud (saving bandwidth) — before the speech recognizer generates a transcript.

Audio

Raw audio stream from the microphone.

›
VAD Module

Detects speech and clips the audio.

›
Speech Recognizer

Generates the text transcript.

With two learning algorithms working together, changes to the first can degrade the second. A new phone's microphone could make the VAD clip audio differently — leaving more or less silence — which changes the recognizer's input and hurts its performance.

The same cascade appears with user profiles → recommender systems. Clickstream data builds a user profile (e.g. "does this user own a car?" → yes / no / unknown), which feeds a recommender. If the clickstream changes and the percentage of unknown tags rises, the recommender's input shifts and recommendation quality can drop.

Brainstorm metrics for each component of the pipeline as well as the whole — software, input, and output metrics — so you can spot drift (concept, data, or both) at multiple stages and act before quality degrades.

⏱️ How Quickly Does Data Change?

The rate of change is highly problem-dependent. Face recognition shifts slowly (hairstyles, fashion, camera resolution), while a factory switching to a new batch of material can change appearances in minutes. Two broad generalities (with exceptions):

Consumer / User Data

Usually changes slowly

It is rare for millions of users to all change behavior at once. Exceptions exist — COVID-19, a holiday, or a new movie spiking web-search trends.

Enterprise / B2B Data

Can shift quickly

A factory adopting a new coating, or a CEO changing how the business operates, can shift an entire dataset almost overnight.

The first deployment is only halfway to the finish line. The second half — feeding data back, monitoring, and updating the model in the face of concept and data drift — begins right after you go live.
