Data Definition and Baseline

How to define what your data should be — what is x, what is y — establish a baseline, and then obtain, label, pipeline, and split clean, consistent data for the modeling phase.

1 · Why Defining Data Is Hard

🦎 The Iguana Example

The third and final stage of the machine learning lifecycle is the data stage: how do you get data that sets up your training and modeling for success? But first — why is defining what data to use even hard?

Imagine collecting hundreds of forest photos and asking labelers to draw bounding boxes around iguanas. Three equally diligent, hardworking labelers can produce three very different conventions:

Labeler 1

tight box per iguana

Labeler 2

box extended along the long tail

Labeler 3

one big box over both

Any single convention could train a perfectly good iguana detector. What is not fine is mixing them — one third of the data labeled each way. Then the labels are inconsistent, and that is confusing to the learning algorithm.

The problem is rarely that one convention is "wrong." The problem is inconsistency — even slightly ambiguous instructions produce a mix of conventions that hurts the algorithm.

📱 Phone Defect Detection

The same effect shows up in real computer-vision problems. Asked to box "significant defects" on a phone, one labeler boxes only the big scratch, a second boxes both the scratch and a small pit mark, and a third draws one large box across the region. Probably the second labeler did the best job — but this is a textbook example of inconsistent labeling from slightly ambiguous instructions. If you can consistently label data with one convention, your learning algorithm will do better.

2 · More Label Ambiguity

🎙️ Speech Recognition

Consider an audio clip of someone on a busy roadside: "…nearest gas station…" followed by a car driving past. There are combinatorially many ways to transcribe this:

  • "Um" with one m or two — um vs umm.
  • A comma vs an ellipsis (...) to mark the pause.
  • Whether to write "unintelligible" for the noise after the last words.

Being able to standardize on one convention will help the speech recognition algorithm learn.

🔗 User ID Merge & Structured Data

A common task in large companies is user ID merge — deciding whether two data records (say, from a jobs-listing site and an acquired résumé-chat app) belong to the same person. One approach is supervised learning that takes two records as input and outputs 1 or 0. Ground-truth labels can come from users who explicitly link accounts, or from human labelers using judgment on filtered pairs. Because the truth is genuinely ambiguous, different people label inconsistently — so getting them to label a little more consistently still helps.

Only perform user ID merge in ways that respect the user's data, their privacy, and the permissions they have granted. User privacy is really important.

Other structured-data tasks are similarly ambiguous: is this a bot or spam account? Is this transaction fraudulent? Is this user looking for a new job right now? When the ground truth is ambiguous, giving labeling instructions that produce more consistent, less noisy labels improves the algorithm.

3 · Defining X and Y

🖼️ What Is the Input x?

When defining data, the first question is: what is the input x? For phone-defect photos, is the lighting good enough? The camera contrast? The resolution? If images are so dark that even a person can't see what's going on, the right move may not be to label them — it may be to go to the factory and improve the lighting so labelers can actually see the scratches.

For structured data, the features you include can have a huge impact. For user ID merge, a rough GPS location (with permission) can be a very useful cue for deciding whether two accounts belong to the same person.

If even a person can't look at the input and tell what's going on, improving the quality of the sensor or input x is an important first step to getting reasonable performance.

🏷️ What Is the Target Label y?

Beyond defining x, you must figure out the target label y. The recurring key question: how can we ensure labelers give consistent labels? The ambiguity can come from the labels themselves (the iguana boxes, the "um" spellings) or from an input x that isn't informative enough (an image too dark to read). The next sections put these issues into a systematic framework.

4 · Major Types of ML Problems

🔲 A Two-by-Two Framework

Best practices for organizing data differ sharply across problem types. A useful framework places problems on a two-by-two grid along two axes:

  • Unstructured vs structured data — humans are great at processing unstructured data (images, audio, text) but not at structured database records.
  • Small vs large dataset — using a (slightly arbitrary) threshold of ~10,000 examples, roughly the size beyond which it becomes painful to examine every example yourself.
Unstructured · Small

Visual inspection, 100 phones

Image data, small set. Data augmentation and human labeling both help.

Unstructured · Large

Speech recognition, 50M clips

Lots of data. Augmentation (e.g. background noise) still helps.

Structured · Small

Housing prices, 52 homes

Real-valued features, few examples. Hard to synthesize new houses.

Structured · Large

Recommendations, 1M users

Hard to synthesize new users; human labeling is harder too.

🧭 How the Quadrant Changes Practice

Unstructured data
  • Humans can label it, so you can often get more by labeling unlabeled x.
  • Data augmentation (synthesizing images, audio, even text) is effective.
  • May have a large pool of unlabeled examples (factory photos, self-driving footage).
Structured data
  • Harder to obtain more data — only so many users or houses exist.
  • Augmentation is difficult; you can't synthesize people who don't exist.
  • Human labeling is harder and often more ambiguous.

On the size axis: with small datasets, clean labels are critical and a small team can manually inspect and fix every example. With large datasets, the emphasis shifts to data processes — how you collect data and the labeling instructions you write for a large crowdsourced team, since you can't easily relabel everything afterward.

A practical tip: advice from someone who has worked in the same quadrant as your problem is usually more useful than advice from a different quadrant. One-size-fits-all rules like "always get 1000 labeled examples" rarely hold — ML is too diverse.
5 · Why Small Data Needs Clean Labels

🚁 The Helicopter Rotor Example

Suppose you map voltage applied to a helicopter rotor to its speed in RPM, from just five examples with noisy labels. It's nearly impossible to confidently fit a function — straight line? curve? rising then flat? With a ton of equally noisy data, the algorithm can average out the noise and fit confidently. But the interesting case is having only five examples with clean, consistent labels — then you can confidently fit a good function from very little data.

Computer-vision systems have been trained on just 30 images and worked fine. The key is almost always making sure the labels are clean and consistent.

📏 Agree on a Threshold

For phone-defect inspection, everyone agrees a giant scratch is a defect and a tiny ding is not — but there's a region of ambiguity (say, scratch length between 0.2 and 0.4 mm) where inspectors disagree. One option is to collect far more scratch images and hope the network figures it out — a lot of work. More fruitful: get the inspectors to sit down and agree that, say, 0.3 mm is the threshold above which a scratch becomes a defect. The labels become consistent, and the algorithm achieves higher accuracy even on a small dataset.

Note: the input to the algorithm is still the images, not the scratch length — but agreeing on the length threshold makes the image labels consistent.

📉 Big Data Has Small-Data Challenges Too

Large datasets with a long tail of rare events face small-data challenges:

  • Web search — huge query logs, but many individual queries are rare, so clickstream data per rare query is small.
  • Self-driving cars — millions of driving hours, but very few examples of a child running across the highway or a truck parked across it.
  • Product recommenders — millions of items, but each long-tail item has little interaction data.

Ensuring label consistency for these rare events still helps a lot. Label consistency is critical for small datasets and important even for big ones — it's just easier to achieve on smaller datasets.

6 · Improving Label Consistency

🔄 A General Process

If you're worried about inconsistent labels, follow an iterative process:

  1. Have multiple labelers label the same examples. You can also have one labeler re-label after a break (a "wash-out" period) to check self-consistency.
  2. Discuss disagreements. The ML engineer, the subject-matter expert (e.g. a manufacturing expert), and/or dedicated labelers agree on a more consistent definition of y.
  3. Document the agreement as updated labeling instructions for labeling new data or relabeling old data.
  4. Consider changing the input x if labelers say x lacks enough information — e.g. increase the lighting on dark phone images.
  5. Repeat. Label more data, check for disagreements, and iterate.

🧰 Common Techniques

  • Standardize the label definition — pick one convention for the audio clip (or any ambiguous case) so the data becomes consistent.
  • Merge classes — if "deep scratch" vs "shallow scratch" is unclear and the distinction isn't truly needed, merge them into a single scratch class to remove inconsistency.
  • Create a new class for uncertainty — instead of forcing defect / not-defect, add a borderline class. In speech, add an "unintelligible" tag for genuinely unclear audio rather than forcing a guess.

For small datasets, the small labeling team can sit together and drive to agreement on specific examples. For big datasets, define a consistent standard with a small group, then distribute the instructions to the larger labeling team.

Having multiple labelers vote (consensus labeling) can work, but it's overused. Prefer fixing the label definitions first to make each labeler less noisy, rather than collecting noisy labels and voting to average out the noise — use voting as a last resort.
7 · Human Level Performance (HLP)

📐 HLP as a Baseline

For inherently ambiguous outputs, human level performance (HLP) establishes a useful reference baseline. Its most important use is estimating Bayes error (irreducible error), especially on unstructured data, to drive analysis, prioritization, and to set realistic expectations.

Suppose a business owner demands 99% accuracy. Take a dataset with ground-truth labels and have a human inspector label the same data blinded to ground truth. If they agree on four of six examples, HLP is 66.7% — which lets you reply: even your own inspector is only 66.7% accurate, so 99% is unrealistic.

A rarely asked question: what exactly is this ground-truth label? When it was itself written by a human, HLP may just measure how well two people happen to agree — not what's truly possible.

Other uses of HLP: a respectable academic benchmark (beating HLP helps get papers published), and setting a realistic target when a product owner's demand is unreasonable.

8 · The Problem with Beating HLP

⚠️ An Unfair Advantage

Using "we beat HLP" to prove an ML system is superior to humans rarely works as a practical argument — businesses need more than high average test accuracy. Worse, the metric can give the algorithm an unfair advantage when labeling instructions are inconsistent.

Say for "nearest gas station," 70% of labelers use one (perfectly fine) convention and 30% use another. The chance two random labelers agree is:

Chance two labelers agree
0.72 + 0.32 = 0.49 + 0.09 = 0.58
0.7 fraction using convention A  ·  0.3 fraction using convention B

So measured HLP is 58%. But a learning algorithm can learn the statistics and always pick the more common convention, agreeing with ground truth 70% of the time — appearing 12% "better" than HLP. That 12% gain is meaningless: it's just consistently choosing one of two equally good options.

This fake advantage can mask real errors: a system that looks 12% better on this query type may actually produce worse transcripts on other, more important audio — so beating HLP can hide that the algorithm is genuinely worse than people.
9 · Raising HLP Instead of Beating It

🩺 Externally vs Human-Defined Ground Truth

HLP behaves very differently depending on where the ground truth comes from:

Externally defined truth
  • e.g. an X-ray diagnosis confirmed by a biopsy or medical test.
  • HLP genuinely estimates Bayes error — how well a doctor (or algorithm) predicts a real outcome.
  • Fewer problems; this use is very meaningful.
Human-defined truth
  • e.g. a doctor labeling an X-ray, or one inspector vs another.
  • HLP just measures how well one human predicts another human's label.
  • Low HLP often signals an ambiguous labeling convention.

⬆️ Raise HLP by Fixing Labels

Return to the inspector at 66.7%. If you compare scratch lengths and get both labelers to agree on the 0.3 mm threshold, several "disagreements" turn out to be simple convention mismatches that get corrected — raising HLP from 66.7% toward 100% on those examples.

Raising HLP to 100% makes it nearly impossible for the algorithm to "beat HLP" — which sounds terrible. But the real benefit is much cleaner, more consistent data, which ultimately lets the learning algorithm perform better.

When the label y comes from a human, an HLP well below 100% often just means the labeling instructions are ambiguous. Improving label consistency raises HLP and — more importantly — gives cleaner data that raises your algorithm's real-world performance.

Takeaway: HLP is useful and important. Measure it to gauge what's possible and to drive error analysis. But if HLP is much lower than 100%, ask whether the gap is due to inconsistent labeling instructions — if so, fixing them benefits both HLP and your model.

10 · How Much Time to Spend Collecting Data

⏱️ Get Into the Iteration Loop Fast

Machine learning is a highly iterative process: pick a model and hyperparameters, train on a dataset, run error analysis, and loop many times. If a first round of training plus error analysis takes only a couple of days each, then spending 30 days collecting data first delays the entire iteration loop by a whole month.

A better mindset is to ask: "What if I gave myself only two days to collect data?" Maybe two days is too short — but far too many teams take too long building an initial dataset before training their first model. The opposite mistake is rare.

After you train an initial model and run error analysis, there is plenty of time to go back and collect more data. Enter the loop quickly, then let error analysis tell you whether more data is even worth collecting.

A useful trick: tell the team "we will spend at most seven days collecting data — what can we do?" Posing the question this way pushes people toward creative, scrappy ways to get a lot of data quickly (while still 100% respecting user privacy and any regulatory constraints).

Exception: if you have worked on the problem before and know from experience that you need at least a certain training-set size, it can be fine to invest more up front. For a brand-new problem where the needed amount is unclear (and the literature rarely settles it), quickly collect a small amount, train, and use error analysis to decide whether more data is worthwhile.

11 · Inventory of Data Sources

📋 Brainstorm Sources and Trade-offs

Before collecting, take inventory of the possible data sources and their costs. For speech recognition, a brainstorm might look like this:

SourceCostTime
Owned data (100 hrs transcribed)$0Instant
Crowdsourced reading (pay people to read text)Low per hour~2 weeks engineering
Pay to transcribe unlabeled audio~$6,000 / 100 hrsWork to organize
Purchase from a commercial vendorVariesOften quick (PO process)
The column most often forgotten is time cost — how long it takes to actually execute and get the data. Owned data is instant; crowdsourced reading may need software integration; purchasing may just be a purchase-order process.

Many teams skip this inventory and just pick a random idea (e.g. crowdsourcing). Writing out all the sources and weighing the trade-offs — cost, time, data quality, and privacy/regulatory constraints — leads to better decisions. If you are pressed for time, the analysis might point you to use data you already own plus some purchased data, skipping the slower middle options.

12 · Getting Data Labeled

✍️ In-House vs Outsource vs Crowdsource

The three most common ways to get labels:

In-house

your own team labels the data

Outsource

a specialized labeling company

Crowdsource

a large crowd labels collectively

Having ML engineers label data is expensive, but doing it for just a few days to get a project going is usually fine — and it builds the engineer's intuition about the data. Beyond that point, shift to a more scalable labeling process.

🧑‍⚕️ Who Is Qualified to Label?

Different applications need different labelers:

  • Speech recognition — almost any reasonably fluent speaker can transcribe audio, giving a very large pool of potential labelers.
  • Factory inspection / medical imaging — needs a subject-matter expert (SME); a person off the street can't reliably read an X-ray or spot a phone defect.
  • Product recommendations — often no human can give good labels, so you rely on the user's own purchase behavior as the label.
Identifying which category your application falls into — and finding the right type of person to label — is an important step to ensuring high-quality labels.
13 · Scaling Up the Dataset

📈 The 10x Rule

Say you have 1,000 examples and decide you need more. A useful guideline: don't increase your data by more than 10x at a time. Go from 1,000 to maybe 3,000 or at most 10,000, then train again and run error analysis before deciding whether to grow further.

Once you increase data by 10x, so many things change that it becomes very hard to predict the effect. Growing by 10%, 50%, or 2x is also perfectly fine — 10x is only an upper bound on how much to invest in one step.

This guideline helps teams avoid over-investing in tons of data, only to discover that collecting that much wasn't the most useful thing they could have done.
14 · Data Pipelines

🔧 Multi-Step Data Processing

A data pipeline (or data cascade) is when data goes through multiple processing steps before reaching the final output. Example: predict whether a user is looking for a job. The raw user data first goes through preprocessing / data cleaning — spam cleanup (removing spam accounts) and user ID merge — before it's fed to the learning algorithm.

These cleaning steps might be plain scripts (explicit rules for what counts as spam or when to merge two IDs) or themselves be ML models, which makes the pipeline even harder to manage.

♻️ Replicability: POC vs Production

The big risk with pipelines is replicability. During development, preprocessing is often messy — hacked-up scripts spread across different team members' laptops. In production, new data must flow through the same preprocessing so the input distribution to the model matches. How much effort to invest in making this replicable depends on the project phase:

Proof of Concept (POC)
  • Goal: decide if the application is workable and worth building.
  • Focus on getting the prototype to work — manual preprocessing is OK.
  • Take extensive notes and comments so you can replicate it later.
  • Don't get bogged down in heavy replicability process yet.
Production
  • Replicating every preprocessing step now becomes critical.
  • Use more sophisticated, heavyweight tooling.
  • e.g. TensorFlow Transform, Apache Beam, Airflow.
  • Make the entire data pipeline reliably replicable.
15 · Data Provenance and Lineage

🧬 Tracking Where Data Comes From

Real pipelines get far more complex — e.g. train an anti-spam model on a spam dataset, apply it to produce de-spammed user data, train an ID-merge model, apply it to get cleaned user data, then finally train the job-seeking predictor. Arrows of computation chain model after model.

Now suppose months later you discover the purchased IP-address blacklist had mistakes — some addresses were wrongly blacklisted because many users on a corporate or university campus legitimately shared them. Updating the spam dataset changes the spam model, which changes everything downstream. Fixing this in a system spread across many engineers' laptops is very hard. Two concepts help:

  • Data provenance — where the data came from (e.g. who you purchased the blacklist from).
  • Data lineage — the sequence of processing steps that produced the end of the pipeline.
At minimum, extensive documentation lets you reconstruct provenance and lineage. For production-grade systems, tools like TensorFlow Transform help — but honestly, tooling for provenance and lineage is still immature across the ML community.
16 · Metadata

🗂️ Data About Your Data

Metadata is data about data. In manufacturing visual inspection, the data is the phone images and labels — the metadata is everything around them:

  • Time the picture was taken; which factory and line number.
  • Camera settings — exposure time, aperture.
  • The phone's serial number; the ID of the inspector who labeled it.

This pays off during error analysis. If you discover that line 17 in factory 2 produces far more errors, you can go investigate what's unusual about it — but only if you stored that metadata in the first place.

Just like you rarely regret commenting your code, you will rarely regret storing metadata. And just like comments, it's much harder to recapture metadata after the fact than to record it in a timely way.

For speech recognition, track things like the smartphone brand the audio came from, which labeler transcribed it, or the version number of the voice-activity-detection model used. If one VAD version causes much larger errors, recorded metadata dramatically increases the odds you'll spot it. Metadata helps with both error analysis and data provenance.

17 · Balanced Train / Dev / Test Splits

⚖️ Why Random Splits Hurt Small Data

We usually split a dataset randomly into train / dev / test. But with a small dataset, a balanced split can significantly improve development. Take 100 visual-inspection images with 30 defective (positive) and 70 good, split 60% / 20% / 20%.

A random split could easily land 21 positives in train, 2 in dev, and 7 in test:

SplitRandom (positives)% positiveBalanced
Train (60)2135%18 → 30%
Dev (20)210%6 → 30%
Test (20)735%6 → 30%

The dev set at only 10% positive is non-representative of the true 30%, making it an unreliable measure of performance. A balanced split forces each of train, dev, and test to hold exactly 30% positives, matching the true distribution.

With a large dataset, a random split is almost always representative, so this isn't a concern. But for small datasets, an explicit balanced split makes your dev and test sets far more reliable measures of your algorithm's performance.
