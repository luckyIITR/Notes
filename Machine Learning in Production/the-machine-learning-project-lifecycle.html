The Machine Learning Project Lifecycle

Why a trained model is only the beginning, and how the full lifecycle — scoping, data, modeling, deployment — frames every production ML project.

1 · From Model to Production

🚀 A Great Model Is Not the Finish Line

Knowing how to train a machine learning model is valuable, but a model only creates the maximum possible value once it is put into production to power real applications. This is also a common interview theme: how would you deploy and monitor a machine learning algorithm in production?

The goal here is to extend your skills — like training deep learning models — toward deployment: powering applications and products with the full machine learning project lifecycle, from scoping a project, to defining data, to training a model, to choosing and implementing the best deployment pattern.

Developing a great model is an important milestone — but the real question this course answers is what comes after: how to actually deploy and maintain it.
2 · A Concrete Example: Visual Defect Inspection

📱 Inspecting Phones on a Manufacturing Line

Imagine using computer vision to inspect phones rolling off a manufacturing line for defects. A good phone has no scratches; if there is a scratch or crack, the algorithm should detect it and draw a bounding box around it as part of quality control. This is called automated visual defect inspection.

With a dataset of scratched phones you can train a neural network to detect defects. But putting that into production requires far more than the model. A typical deployment uses an edge device — a device living inside the factory — running inspection software:

Camera

Takes a picture of each phone as it rolls off the line.

›
Inspection Software

Controls the camera and makes an API call with the image.

›
Prediction Server

Receives the image, decides defective or not, returns the prediction.

›
Control Decision

Keep the phone on the line, or push it aside as defective.

After training a learning algorithm to map input x (phone images) to output y (defective or not), you still must place the model in a prediction server, set up API interfaces, and write all the surrounding software. The prediction server can live in the Cloud or at the edge. Manufacturing leans heavily on edge deployments — a factory cannot halt every time Internet access goes down — while Cloud prediction servers are common for many other applications.

3 · What Could Go Wrong

⚠️ Doing Well on the Test Set Is Not Enough

Doing well on a holdout test set is worth celebrating — but reaching that milestone does not mean you are done. A frequent problem is that the production data differs from the training data. For example, if the factory's lighting conditions change, deployed images come back much darker than the training images, and performance degrades.

This is an example of concept drift or data drift. Spotting and handling these practical issues is part of the machine learning engineer's job — adjusting to the data distribution as it is, rather than as we wish it were.

A successful proof of concept in a Jupyter Notebook is a real success — but the gap from there to a reliable production deployment can still be another six months of work.

🧩 It Takes Much More Than ML Code

There has been amazing progress on the machine learning model itself — the neural network or algorithm that learns a mapping from input to output. But in a production system, the model code is only a small fraction of the whole: often just 5–10% (or less) of the code in a machine learning project.

This is a major reason for the POC-to-production gap — the sheer amount of additional software needed beyond the model. Adapted from a paper by D. Sculley and others, the supporting components include:

  • Data collection
  • Data verification
  • Feature extraction
  • Configuration and resource management
  • Serving infrastructure
  • Monitoring and analysis of returning data

Rather than tackling all these pieces at once, the most useful organizing framework is to systematically plan the lifecycle of the project.

4 · The Machine Learning Project Lifecycle

🔄 The Four Major Steps

Thinking through the lifecycle is an effective way to plan every step you need to work on — and to minimize surprises. The major steps are:

Step 1

Scoping

  • Define the project
  • Decide what to apply ML to
  • Define X and Y
Step 2

Data

  • Define data & baseline
  • Label and organize data
Step 3

Modeling

  • Select and train model
  • Perform error analysis
Step 4

Deployment

  • Deploy in production
  • Monitor & maintain

Machine learning is highly iterative. During error analysis you may go back to update the model, or back to the data phase to collect more data. Before deploying, a final check or audit confirms the system is reliable enough for the application.

When you deploy a system for the first time, you are only about halfway to the finish line — the second half of the lessons come only once live traffic is turned on.

After initial deployment, maintenance often means returning to error analysis, retraining the model, and feeding live data back into your dataset to update the data and retrain — until an improved model is ready for deployment. This framework applies broadly: computer vision, audio, structured data, and beyond.

5 · Walkthrough: Speech Recognition

🎙️ The Lifecycle on a Real Application

Deep learning has made speech recognition far more accurate, powering smart speakers and voice search. Walking through a speech system shows every lifecycle step in context.

Scoping

Decide to work on speech recognition (e.g. for voice search) and estimate key metrics — these are highly problem-dependent. For speech, important metrics include:

  • Accuracy — how accurate is the transcription?
  • Latency — how long to transcribe speech?
  • Throughput — how many queries per second can we handle?

If possible, also estimate the resources needed (time, compute, budget) and the timeline.

Data

A key challenge is data label consistency. Given the audio "Um, today's weather", transcriptionists might reasonably write it several ways — including or dropping the "Um", or labeling background sounds as noise. Any single convention is fine; the problem arises when transcriptionists disagree — say a third use each style. Inconsistent labels confuse the learning algorithm, because it cannot guess which convention a given clip used. Standardizing on one convention can significantly improve performance.

Other data-definition questions include how much silence to keep before and after each clip, and how to perform volume normalization (including clips that are loud and soft within the same recording).

Unlike fixed benchmark datasets, a production system lets you edit the training set or even the test set when needed to improve data quality — done systematically, not ad hoc.

Modeling

Three inputs produce a trained model: code (the algorithm / model architecture), hyperparameters, and data.

  • Research / academic style: hold the data fixed, vary the code and hyperparameters.
  • Product team style: often more effective to hold the code fixed (e.g. an open-source implementation) and optimize the data and hyperparameters.

Error analysis tells you where the model falls short. When it points to systematic, targeted ways to improve the data, that is far more efficient than blindly collecting more and more data, which is helpful but expensive.

Deployment

A common pattern runs software locally on a mobile phone (the edge device):

Microphone

Records what the user is saying.

›
VAD Module

Voice Activity Detection selects only the audio that contains speech.

›
Prediction Server

In the Cloud — returns the transcript and search results.

›
Frontend

Displays the transcript and results on the phone.

Voice Activity Detection (VAD) is usually a relatively simple algorithm that lets the phone send only speech audio to the server.

A real example of data drift: a speech system trained mostly on adult voices degraded as more young users — teenagers and younger — adopted it, because their voices sound different. The fix required collecting more data. Good monitoring spots such drift early, and timely fixes keep the deployment valuable.
6 · Course Roadmap & MLOps

🗺️ Working Backwards From Deployment

Although the lifecycle runs left-to-right, it is more efficient to learn it by starting at the end goal — deployment — and working backward to modeling, data, and scoping.

  1. Week 1 — Deployment: the most important ideas in taking a model to production.
  2. Week 2 — Modeling: a systematic, data-centric approach to efficiently improving model performance.
  3. Week 3 — Data: defining data, establishing a baseline, and labeling and organizing data systematically (plus an optional section on scoping).
  4. Final project (optional): the full lifecycle left-to-right, scoping through deployment.

MLOps (machine learning operations) is an emerging discipline of tools and principles that support progress through the lifecycle — especially the data, modeling, and deployment steps. Steps that were once done manually (okay, but slow) become much quicker with MLOps tooling.

The key idea of MLOps is systematic ways to think about scoping, data, modeling, and deployment — backed by software tools that support best practices.
