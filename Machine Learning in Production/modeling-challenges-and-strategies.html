Modeling Challenges and Strategies

The key challenges of building a production-ready model — and a data-centric playbook for selecting, baselining, and improving it efficiently.

1 · Modeling in the Lifecycle

🎯 Model-Centric vs Data-Centric AI

This part of the lifecycle focuses on the modeling step: how to select and train a model, how to perform error analysis, and how to use that analysis to drive improvements. Senior machine learning engineers tend to give remarkably consistent advice here — consistent enough that it can almost be captured as an if-then-else sequence of decisions.

A recurring theme is model-centric versus data-centric AI development. Historically, the field emphasized choosing the right model — for example, the right neural network architecture. For practical projects, it is often more useful to take a data-centric approach: focus not only on the architecture, but on feeding the algorithm high-quality data.

Data-centric development does not mean blindly collecting more data — that is slow and expensive. It means using tools to improve the data in the most efficient possible way.
2 · Why Model Development Is Hard

🧩 AI Systems Are Code + Data

A useful framework: every machine learning system comprises both code (the algorithm or model) and data. Decades of research grew up around downloading a fixed dataset and finding a model that does well on it. But in many real applications you have the flexibility to change the data if you do not like it.

For many projects the model is essentially a solved problem — something off GitHub will do well enough — so time is better spent improving the data, which is far more customized to your specific problem. Hyperparameters (learning rate, regularization, and a few others) matter too, but their space is relatively limited, so most attention goes to the code and the data.

🔄 A Highly Iterative Loop

Model development is highly iterative. You start with a model, hyperparameters, and data; you train; you carry out error analysis; and you use the results to decide how to change the model, the hyperparameters, or the data.

Model + Hyperparams + Data

Start with a reasonable choice for each.

Because machine learning is so empirical, getting through this loop many times, quickly is key to improving performance — as is making good choices each time about what to change. Once you reach a good model, a final, richer error analysis and an audit confirm the system works before pushing it to production.

🪜 Three Milestones Every Project Should Aim For

Most projects should aspire to clear three milestones, in order:

Milestone 1

Do well on the training set

If you cannot even fit the training data, you almost certainly will not do well on the dev or test set.

Milestone 2

Do well on dev / test set

Generalize to the holdout cross-validation set and the test set — the classic target of ML development.

Milestone 3

Do well on the business metrics

Meet the project's actual goals — not just average test accuracy, which is often not sufficient.

A high test set accuracy that fails the business metrics is a common source of friction between ML teams (who optimize the test set) and business teams (who care about real-world goals).
3 · When Test Set Accuracy Isn't Enough

⭐ Disproportionately Important Examples

A system can have low average test set error yet still be unacceptable if it fails on a set of disproportionately important examples. In web search, queries fall into different types:

  • Informational / transactional — "apple pie recipe", "wireless data plan". Users will forgive ranking the best result second or third, since many good results exist.
  • Navigational — "Stanford", "Reddit", "YouTube". The user has a very clear intent; anything other than returning the right site as the #1 result quickly loses their trust.

Average test set accuracy weights all examples equally, but navigational queries are disproportionately important. Simply re-weighting those examples can help in some applications, but in practice it does not always solve the whole problem.

⚖️ Performance on Key Slices & Fairness

Closely related is performance on key slices of the dataset. Consider a loan-approval model: it must not unfairly discriminate against applicants by ethnicity, gender, location, language, or other protected attributes. Many countries legally mandate this. High average accuracy is irrelevant if the system exhibits an unacceptable level of bias.

The same idea appears beyond fairness to individuals. An e-commerce site should treat all major user, retailer, and product categories fairly:

  • Giving irrelevant recommendations to all users of one ethnicity is unacceptable.
  • Always pushing products from large retailers while ignoring smaller brands can drive away small retailers and harm long-term business health.
  • Never recommending a whole category (e.g. electronics) would reasonably upset the retailers who sell it.

Later in the course you learn to run error analysis on key slices to spot and address problems like these.

🩺 Skewed Distributions & Rare Classes

With skewed data distributions, average accuracy can be deeply misleading. In medical diagnosis where 99% of patients are negative, a program that simply prints "0" achieves 99% accuracy — no learning required — yet is useless. (This really happened: a giant neural network was found to do exactly what print "0" does.)

Related is accuracy on rare classes. On chest X-ray diagnosis, a common condition like effusion (~10,000 images) reached high performance, while a rare condition like hernia (~100 images) performed much worse. A system could ignore hernia almost entirely with only a modest hit to average accuracy — but medically, missing an obvious hernia is unacceptable.

Andrew's advice when "but I did well on the test set!" meets "but it doesn't work for my application": don't get defensive. Doing well on the test set is worth celebrating, but the real job is to build a system that solves the actual business or application needs.
4 · Establishing a Baseline

📏 Why a Baseline Comes First

One of the most useful first steps is to establish a baseline level of performance. Only once a baseline exists do you have the tools to efficiently improve on it. Take a speech recognition system with four categories of audio:

CategoryCurrent accuracyHuman level (HLP)Potential gain
Clear speech94%95%~1%
Car noise89%93%~4%
People noise87%89%~2%
Low bandwidth70%70%~0%

Without a baseline you might rush to fix low-bandwidth audio, since 70% looks worst. But human level performance (HLP) reveals the low-bandwidth audio may be so garbled that even humans cannot transcribe it — leaving little headroom. The bigger opportunity is car noise, where there is a ~4% gap to close.

A baseline indicates what might be possible. For unstructured data, HLP also approximates the irreducible (Bayes) error — the best anyone could hope to achieve — which helps prioritize where to work.

🗂️ Unstructured vs Structured Data

Best practices for baselines differ by data type:

Unstructured data
  • Images, audio, natural language — things humans evolved to interpret well.
  • Humans are very good here, so HLP is a strong baseline.
Structured data
  • Giant databases / spreadsheets — transactions, inventory, prices.
  • Humans are not good at reading these, so HLP is a weaker baseline.

Beyond HLP, other ways to establish a baseline include:

  • Literature search / open source — see what state-of-the-art or published results report on similar data.
  • Quick-and-dirty implementation — a rough open-source build that hints at what is possible.
  • Older system — if you already have one running, its performance is a baseline to improve on.
If a business team demands a guaranteed accuracy (80%, 90%, 99%) before any baseline exists, push back and ask for time to establish a rough baseline first — it sets the team up for long-term success.
5 · Tips for Getting Started

🚦 Get Going Quickly, Then Iterate

To start the first lap of the iteration loop efficiently:

  1. Do a quick literature search. Skim courses, blogs, and open-source projects for half a day. For a production system (not research), don't obsess over the latest, greatest algorithm — pick something reasonable that lets you start fast.
  2. Favor reasonable algorithm + good data. A reasonable algorithm with good data often beats a great algorithm with mediocre data. A good open-source implementation also helps you establish a baseline more efficiently.
  3. Treat deployment constraints judiciously. If the baseline is set and the project is confirmed worth deploying, factor in compute and other constraints. If you are still just establishing a baseline to see what's possible, it can be okay to ignore them for now.

🔍 Run Quick Sanity Checks First

Before launching a long training run on all your data, run cheap sanity checks to find bugs fast:

  • Overfit a tiny dataset. Try to fit a very small training set — even a single example, especially when the output is complex. A speech system trained on one clip that outputs only "space, space, space…" is clearly broken; no point training overnight.
  • Try one example for complex outputs. For image segmentation, see if the model can overfit a single image before scaling to thousands.
  • Train on a small subset. Even with a million images, first train on 10–100 images. If the algorithm can't do well on 100, it won't do well on 10,000 — and you learn this in seconds or minutes.
Each sanity check costs seconds or minutes but can save hours or days — letting you find bugs quickly and iterate faster toward good performance.
6 · Error Analysis

🔬 The Heart of the Development Process

The first time you train a learning algorithm, you can almost guarantee it won't work — not the first time out. Error analysis is the heart of the development process: done well, it tells you the most efficient use of your time for improving performance.

The classic technique is wonderfully low-tech: open a spreadsheet (Google Sheets, Excel, Numbers) and listen to ~100 mislabeled examples from your dev set. For each example, tag it with attributes you hypothesize might explain the errors. In speech recognition, for instance, "stir fried lettuce recipe" might be transcribed as "stir fry lettuce recipe."

Example (ground truth → prediction)Car noisePeople noiseLow bandwidth
stir fried lettuce → stir fry lettuce✓
sweetened coffee → Swedish coffee✓
sail away song → sell away song✓
let's catch up → let's ketchup✓✓✓

Tags are not mutually exclusive, and you invent new ones as you go — noticing that several clips share a low-bandwidth connection, for example, prompts a new column you go back and re-check. This surfaces the categories that are the biggest sources of error, and therefore most worthy of attention.

Error analysis has traditionally been a manual process in a notebook or spreadsheet, but emerging MLOps tools (e.g. LandingLens for computer vision) now make tagging and slicing far easier than a spreadsheet.

🏷️ An Iterative Tagging Process

Like training and deployment, error analysis is itself iterative: tag a set of examples with an initial set of tags, reflect, propose new tags, then tag more examples. Tags can take many forms:

  • Class labels — does this phone have a scratch? a dent?
  • Image / signal properties — is the photo blurry? a dark vs light background? unwanted reflection? car noise vs people noise?
  • Metadata — phone model, factory, manufacturing line, or for recommendations: user demographics, product features, product categories.

The goal is to find a few categories where you could productively improve the algorithm — like speech with car noise, or "are we bad at recommending products to younger women?"

📊 Useful Numbers to Compute Per Tag

As you work through the tags, four questions help you decide what matters:

  • What fraction of errors have that tag? If 12% of your 100 clips have car noise, fixing all car-noise issues caps the gain at ~12%.
  • Of all data with that tag, what fraction is misclassified? If 18% of car-noise data is mistranscribed, that reveals how hard those examples are (requires tagging correct examples too, not just errors).
  • What fraction of all data has that tag? Tells you how important the category is relative to the whole dataset.
  • How much room for improvement is there? Often measured by HLP on data with that tag.
7 · Prioritizing What to Work On

🎚️ Weighting Gap by Category Size

The gap to HLP alone can mislead. Multiply each category's potential gain by the fraction of data it represents to estimate its impact on overall accuracy:

CategoryAcc / HLP (gap)% of dataOverall raise if closed
Clean speech94% / 95% (1%)60%0.60%
Car noise89% / 93% (4%)4%0.16%
People noise87% / 89% (2%)30%0.60%
Low bandwidth70% / 70% (0%)6%~0%
Car noise has the biggest gap, but it's only 4% of the data. Because people noise and clean speech each cover far more data, they offer larger total improvement — a richer analysis flips the priority.

🧭 Four Factors for Deciding

There is no mathematical formula for what to work on, but weigh these factors together:

  • Room for improvement — gap to HLP or some baseline.
  • Frequency — how often the category appears.
  • Ease of improvement — do you have ideas (e.g. data augmentation) for this category?
  • Importance — e.g. car-noise accuracy matters extra for hands-free, in-car map search.

Once you pick a category, a fruitful approach is to add data or improve its quality for that handful of categories — by collecting more targeted data or via data augmentation. This focus makes data collection far more efficient than gathering more data generically, which is slow and expensive.

8 · Skewed Datasets & the Confusion Matrix

⚠️ When Raw Accuracy Lies

Datasets where the ratio of positive to negative examples is far from 50/50 are skewed datasets. They appear everywhere: manufacturing (99.7% non-defective), medical diagnosis (99% disease-free), and wake-word detection (one dataset was 96.7% negative, 3.3% positive). On such data, print "0" scores extremely high accuracy while being useless.

9 · Precision, Recall & F1

🎯 Precision and Recall

Precision asks: of everything the algorithm called positive, what fraction was right? Recall asks: of everything that was actually positive, what fraction did it catch?

Precision & Recall
Precision = TP / (TP + FP) = 68 / (68 + 9) = 88.3%
Recall = TP / (TP + FN) = 68 / (68 + 18) = 79.1%
TP true positives  ·  FP false positives  ·  FN false negatives

These metrics expose the print "0" trick: an all-negative predictor produces 0 true positives, so recall = 0/(0+86) = 0% (and precision is the undefined 0/0). Recall instantly flags that it detects no useful positives. A high precision with near-zero recall is usually not useful.

⚖️ Combining Them: the F1 Score

One model may win on precision while another wins on recall. The F1 score combines them, deliberately emphasizing the worse of the two — it's the harmonic mean, an average that leans toward the lower number.

F1 Score
F1 = 2 · (P · R) / (P + R)
P precision  ·  R recall  ·  punishes a model that is weak on either one

A model with balanced P and R might score F1 = 83.4%, while a model with very poor recall gets a low F1 — making it clear the first model is superior. F1 is just one common way to combine the two; your application may warrant a different weighting.

🔢 Multi-Class Rare Defects

Precision/recall extend naturally to multi-class problems with several rare classes. For smartphone defect detection — scratches, dents, pit marks, discoloration — compute precision and recall for each defect individually:

DefectPrecisionRecallF1
Scratch82.1%99.2%89.8%
Dent—high—

Many factories favor high recall — they'd rather flag a good phone for human re-inspection than let a defective one ship. Because all four defects are rare, accuracy would be misleadingly high; F1 per class gives a single metric per defect to benchmark against HLP and to prioritize the most fruitful defect to work on.

10 · Performance Auditing

🛡️ One Last Check Before Production

Even when the algorithm does well on accuracy or F1, it's worth a final performance audit before deployment — it can save you from significant post-deployment problems. The framework:

  1. Brainstorm ways the system might go wrong — performance across ethnicities/genders, problematic false positives or negatives, accuracy on rare and important classes.
  2. Establish metrics against those issues — typically evaluated on slices of the data (e.g. one gender, one ethnicity, or all scratch-defect examples) rather than the whole dev set. Tools like TensorFlow Model Analysis (TFMA) can auto-compute sliced metrics for each model.
  3. Get buy-in from the business / product owner that these are the right problems and metrics — and fix any issue you find before shipping.

🗣️ Example & Standards

For a speech system you might audit accuracy by gender, ethnicity, perceived accent, and device (different microphones), plus the prevalence of rude mis-transcriptions. A real case: DeepLearning.AI's GANs lectures were mistranscribed as "gun" and "gang," making subtitles look full of gun violence — so swear-word/offensive mis-transcriptions get special attention, as they're perceived far more negatively than neutral errors.

Standards for fairness and acceptable bias vary by industry and are still evolving. Search your industry's norms, and for high-stakes applications have a team or external advisors brainstorm what could go wrong — get ahead of problems proactively rather than being surprised after deployment.
11 · A Data-Centric Approach

🔁 Hold the Data Fixed vs Hold the Code Fixed

Once error analysis points you at a category to improve — say speech with car noise — the next question is how to improve it. This is where the contrast between the two development philosophies becomes concrete:

Model-centric view
  • Hold the data fixed, iteratively improve the code / model.
  • Drove most academic research: download a benchmark dataset (a fixed quantity) and work hard to do as well as possible on it.
  • Still an important role to play in inventing better models.
Data-centric view
  • Hold the code fixed, iteratively improve the data.
  • Treats data quality as paramount; uses tools like error analysis and data augmentation to improve it systematically.
  • Often more useful in practice — if the data is good enough, many models will do just fine.
If you've spent most of your ML experience thinking model-first, try adding the data-centric question to your toolkit: instead of only asking "how can I improve the model?", ask "how can I make my dataset even better?" One of the most important answers is data augmentation.
12 · The Rubber Sheet Analogy

🎚️ Predicting the Effect of More Data

A helpful mental picture for deciding whether to collect or augment data. Imagine the horizontal axis as the space of possible inputs — for speech, the different noise types laid out so that similar noises sit near each other:

  • Mechanical noises cluster together — car, plane, train sound alike; machine noise (washing machine, loud AC) is a bit further away.
  • People / ambient noises cluster separately — cafe, library, food-court noise are similar to one another.

The vertical axis is performance (e.g. accuracy). Your model traces one curve across this space, and human level performance traces a higher curve; the gap between them is the opportunity for improvement. Think of the model's curve as a flexible rubber sheet.

When you add data for one category (e.g. lots more cafe-noise audio), you grab that point of the rubber sheet and pull it up. Nearby points (similar noises) get pulled up a lot; far-away points may rise a little. For unstructured data, pulling one part up almost never makes a distant part dip far down.

After you pull up one region, the biggest gap shifts elsewhere. Error analysis then tells you the location of the new biggest gap — the next piece worth pulling up. Iterating this "pull up one piece at a time" is an efficient way to march the whole curve toward human level performance.

13 · Data Augmentation

🎧 Creating Realistic New Examples

Data augmentation is an efficient way to get more data, especially for unstructured problems (images, audio, text). For speech you can literally sum two waveforms — a clean clip of "AI is the new electricity" plus background cafe noise (or background music) — to synthesize an example that sounds like it was recorded in a noisy cafe. But you must choose what noise to add and how loud it should be relative to the speech.

The guiding goal: create realistic examples the algorithm does poorly on, while a human (or some baseline) can still do well on them. Examples that are too easy give the algorithm nothing to learn; examples so noisy that no one can understand them aren't helpful either.

✅ A Checklist Instead of Retraining

A naive loop — generate augmented data, retrain, check the dev set, tweak parameters, retrain again — is slow, because every parameter change costs a full training run. Instead, sanity-check generated data before training with three quick questions:

  1. Does it sound / look realistic? The data should resemble the real audio or images you ultimately want to perform well on.
  2. Is the x→y mapping clear? Can a human still recognize what was said (or see the defect)? If yes, the label is valid.
  3. Is the algorithm currently doing poorly on it? If so, there is genuine signal left to learn from.

Data meeting all three gives a high chance that adding it to the training set will successfully "pull up the rubber sheet" — without spending hours or days verifying by training.

📱 Image Example: Scratched Phones

With a small set of scratched-phone images you can augment by:

  • Horizontal flip — buttons move to the other side, but it stays a realistic, useful example.
  • Contrast / brightening — makes the scratch more visible while remaining realistic.
  • Excessive darkening — fails the checklist: if even a human can't tell whether a scratch is present, the x→y mapping is no longer clear.

Favor schemes that produce many "passes the checklist" examples and few failures. More sophisticated options exist — using Photoshop to draw a realistic scratch onto a clean phone, or even GANs to synthesize scratches — but GANs are often overkill: simpler, faster techniques frequently work just as well.

14 · The Data Iteration Loop

🔄 Iterating on Data, Not Just the Model

Model iteration means repeatedly training a model, doing error analysis, and deciding how to improve the model. A data-centric approach swaps the focus: in the data iteration loop you take the data and model, train, do error analysis, and then focus on how to add data or improve its quality.

Train Model

On the current dataset.

›
Error Analysis

Find the category with the biggest gap.

›
Improve the Data

Augment / collect data for that tag.

↺
Repeat

Plus a robust hyperparameter search.

Paired with a robust hyperparameter search, this loop often yields faster improvements than model iteration. If error analysis shows the algorithm struggles on cafe-noise speech, generating more cafe-noise data is an efficient fix.

15 · Can Adding Data Hurt?

🧮 Usually No — With Caveats

Augmentation changes your input distribution P(x): maybe cafe noise jumps from 20% to 50% of the data, so the training set no longer matches the dev/test distribution. For unstructured problems, accurately labeled extra data rarely hurts accuracy — provided two conditions hold:

When it's safe
  • Large, low-bias model — a high-capacity network has enough resources to do well on cafe-noise and non-cafe-noise data at once.
  • Clear x→y mapping — given only x, a human can predict y accurately.
When it can hurt
  • Small model — limited capacity may get over-spent modeling the over-represented category, hurting the rest.
  • Ambiguous x→y mapping — when the true label is genuinely unclear from x.

The rare failure case comes from Google Street View house-number recognition. The digit 1 and the letter I can be visually identical, but house numbers almost never contain an "I". If you over-add ambiguous "I" examples, you skew the dataset so the model stops exploiting the prior that "1" is far more common — and it guesses worse on truly ambiguous images. This contradicts the "clear mapping" condition and is a genuine corner case.

This is a rare, almost corner-case scenario. So long as the model is big enough to learn from diverse sources, you can use augmentation comfortably — even when it makes the training distribution differ from dev/test.
16 · Adding Features for Structured Data

🍽️ When You Can't Make New Examples

For structured data, creating brand-new training examples is usually hard — a restaurant-recommendation app has a fixed pool of users and restaurants, so you can't synthesize new people. The more fruitful move is to add useful features to the examples you already have.

In a real case, error analysis showed the system kept recommending meat-only restaurants to clearly vegetarian users. Rather than augmenting users, the fix was to add features:

  • On the user side — a (soft) feature for how likely the person is vegetarian, e.g. the percentage of past orders that were vegetarian.
  • On the restaurant side — whether the restaurant has good vegetarian options based on its menu.

Features like these can be hand-coded, generated by a learning algorithm (e.g. classifying menu items as vegetarian), or labeled by people. The same idea fixes "users who only ever order tea/coffee" or "users who only order pizza" — enrich their features so the model recommends better. Features are themselves a form of data, so modifying features driven by error analysis is a legitimate data iteration loop.

🧲 Collaborative vs Content-Based Filtering

Adding features supports the industry shift from collaborative to content-based recommendations:

ApproachHow it recommendsWeakness
Collaborative filteringFinds users similar to you, recommends what people like you liked.Needs many people to rate a product first.
Content-based filteringLooks at you and at the restaurant's description / menu to judge fit.Requires good features for each item.

Content-based filtering tackles the cold-start problem: it can recommend a brand-new restaurant almost no one has rated, because it reads the description rather than relying on others' ratings. The key enabler is capturing good features for the things you want to recommend.

Deep learning largely removed hand-designed features for unstructured data (images, audio, text). But for structured data — and any time the dataset isn't massive — designing features driven by error analysis is still a very important driver of performance.
17 · Experiment Tracking

📒 What to Track

Running dozens or hundreds of experiments, it's easy to forget what you've already tried. Robust experiment tracking makes you more efficient at deciding what to change next. For each trained model, track:

  • Algorithm & code version — so you can replicate a run from weeks ago whose details you've forgotten.
  • Dataset used — exactly which data fed the model.
  • Hyperparameters — all the settings.
  • Results — at least high-level metrics (accuracy, F1), and ideally a saved copy of the trained model.

🛠️ Tools & What to Look For

Tracking tools scale with team size:

  • Text files — fine for a solo developer's small experiments; doesn't scale.
  • Shared spreadsheets — columns per tracked item; scale surprisingly far for teams.
  • Formal systems — Weights & Biases, Comet, MLflow, SageMaker Studio, LandingLens; the space is still evolving rapidly.

When choosing a tool, look for: enough information to replicate results (watch out — data pulled off the internet can change and break replicability); features to quickly understand a run via summary metrics and analysis; and nice-to-haves like resource monitoring (CPU / GPU / memory), model visualization, and deeper error analysis.

The single biggest takeaway: have some system — even just a text file or spreadsheet — and record as much as is convenient. It pays off the moment you need to remember how an old model was produced.
18 · From Big Data to Good Data

💎 Good Data Beats Big Data

Modern AI grew up at consumer-internet giants with a billion users' worth of data. Big data helps enormously when you have it — but most industries simply don't have a billion data points, which makes the focus on good data even more important. Ensuring consistently high-quality data across all phases of the ML project lifecycle is key to a high-performing, reliable deployment.

What makes data "good":

  • Covers the important cases — good coverage of inputs x; if you lack cafe-noise speech, augmentation can fill that gap (modeling phase).
  • Defined consistently — labels y are unambiguous (covered in depth in the data-definition material).
  • Timely feedback from production — monitoring for concept drift and data drift (deployment phase).
  • Reasonable size — enough data for the problem at hand.
Across the lifecycle the goal is the same: feed your learning algorithm good data in every phase — timely feedback from deployment, good coverage from modeling, and consistent labels from data definition.
