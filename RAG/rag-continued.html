RAG Continued — From Retrieval to Generation

Plug an LLM into the search results to build a real Retrieval-Augmented Generation pipeline — modular Python functions, OpenAI alternatives, and swapping minsearch for Elasticsearch.

1 · Recap — Where Generation Fits In

🧭 The Full RAG Loop

In the previous notes we built a search engine that returns relevant FAQ documents for a user query. That covered the R and the A in RAG — retrieval and augmentation. Now we wire in the G: pass the retrieved documents to a Large Language Model so it can generate an answer grounded in our data.

1Query"can I still enroll?"
2Searchminsearch / Elasticsearch
3Build Promptquestion + context
4LLMGPT-4o, Mistral, Bedrock…
5Answergrounded reply

The data setup is identical to the search-engine notes — load the FAQ JSON, fit a minsearch.Index with three text fields and one keyword field:

Loading the data & building the toy index
import json, minsearch
from openai import OpenAI

with open('documents.json', 'rt') as f_in:
    docs_raw = json.load(f_in)

documents = []
for course_dict in docs_raw:
    for doc in course_dict['documents']:
        doc['course'] = course_dict['course']
        documents.append(doc)

index = minsearch.Index(
    text_fields=["question", "text", "section"],
    keyword_fields=["course"],
)
index.fit(documents)

client = OpenAI()                    # reads OPENAI_API_KEY from env
The key API-key tip: put it in an environment variable (OPENAI_API_KEY) so OpenAI() picks it up automatically — never hardcode it in a notebook you'll share.
2 · The Naïve Baseline — LLM Without Context

🤷 What Happens If We Just Ask GPT-4o Directly?

Before adding RAG, it's worth seeing what a plain LLM call returns — that's the baseline RAG has to beat.

Bare LLM call — no retrieved context
q = 'the course has already started, can I still enroll?'

response = client.chat.completions.create(
    model='gpt-4o',
    messages=[{"role": "user", "content": q}],
)
response.choices[0].message.content
Without context (raw GPT-4o)

"It's not uncommon for courses to accept enrollments even after they have started, but policies vary widely… check the course platform, contact the instructor, reach out to administrative offices…"

Generic. The model has no idea what this course's policy is, so it falls back to advice that could apply to any course on Earth.

With FAQ context (RAG)

"Yes, you can still enroll in the course even after it has started. You are eligible to submit homework assignments, but please be mindful of the deadlines for the final projects."

Specific and correct. The model quotes facts straight from the FAQ — exactly what the student needed.

RAG's job is to turn a generic model into a domain expert by feeding it the right facts at inference time — no fine-tuning required.
3 · Building the Prompt

📝 Anatomy of a Good RAG Prompt

The prompt is where retrieval meets generation. A well-formed RAG prompt has three ingredients:

  1. A role — tell the model who it is ("You're a course teaching assistant"). Especially useful for stronger models like GPT-4o; tiny open-source models may need stricter instructions.
  2. A grounding rule — instruct the model to use only the supplied context. This is what reduces hallucinations.
  3. The context block — the retrieved documents pasted in verbatim, with field labels (section, question, answer) so the model knows what each chunk represents.
Prompt template
prompt_template = """
You're a course teaching assistant. Answer the QUESTION based on the CONTEXT from the FAQ database.
Use only the facts from the CONTEXT when answering the QUESTION.

QUESTION: {question}

CONTEXT:
{context}
""".strip()

The context is built by walking the search results and formatting each hit into a small block:

Building the context string
def build_prompt(query, search_results):
    context = ""
    for doc in search_results:
        context += f"section: {doc['section']}\n" \
                   f"question: {doc['question']}\n" \
                   f"answer: {doc['text']}\n\n"

    prompt = prompt_template.format(question=query, context=context).strip()
    return prompt
Prompt engineering is part art, part science. Start with a simple template, iterate by trial and error, and later swap intuition for numbers using the evaluation metrics introduced later in the course.
4 · The Modular RAG Pipeline

🧱 Four Functions, One Flow

Putting everything into named functions makes the pipeline easy to read, test, and modify. Each function corresponds to one box in the RAG diagram, so swapping a search engine or an LLM later is a one-line change.

search() — retrieval step
def search(query):
    boost = {'question': 3.0, 'section': 0.5}

    results = index.search(
        query=query,
        filter_dict={'course': 'data-engineering-zoomcamp'},
        boost_dict=boost,
        num_results=5,
    )
    return results
build_prompt() — augmentation step
def build_prompt(query, search_results):
    prompt_template = """
You're a course teaching assistant. Answer the QUESTION based on the CONTEXT from the FAQ database.
Use only the facts from the CONTEXT when answering the QUESTION.

QUESTION: {question}

CONTEXT:
{context}
""".strip()

    context = ""
    for doc in search_results:
        context += f"section: {doc['section']}\nquestion: {doc['question']}\nanswer: {doc['text']}\n\n"

    return prompt_template.format(question=query, context=context).strip()
llm() — generation step
def llm(prompt):
    response = client.chat.completions.create(
        model='gpt-4o',
        messages=[{"role": "user", "content": prompt}],
    )
    return response.choices[0].message.content
rag() — the whole pipeline in 3 lines
def rag(query):
    search_results = search(query)
    prompt         = build_prompt(query, search_results)
    answer         = llm(prompt)
    return answer

rag('the course has already started, can I still enroll?')
# → "Yes, you can still join the course even if you discovered it after the start date…"
Why Modular Wins
  • Replace search → swap minsearch for Elasticsearch / Qdrant / pgvector
  • Replace llm → swap GPT-4o for Mistral, Claude, Bedrock, local Ollama
  • Each function is independently testable
  • The rag function reads exactly like the diagram
Common Pitfalls
  • Using q in one place and query in another (variable-name typo)
  • Returning too few results — bump num_results from 5 to 10 if answers feel thin
  • Forgetting .strip() and ending up with leading/trailing newlines in the prompt

6 · Why Replace minsearch With Elasticsearch?

🔄 The Toy Was Only Step One

The TF-IDF in-memory index is great for teaching and prototyping but has real limitations:

minsearch Limitations
  • In-memory — index is lost when the kernel dies
  • Rebuilds from scratch every notebook restart
  • No incremental indexing of new documents
  • No fuzzy matching, BM25, analyzers, multilingual support
  • Single-process — won't scale to many concurrent users
Elasticsearch Wins
  • Persistent — data lives on disk, survives restarts
  • BM25 scoring + analyzers + boosting out of the box
  • Add documents incrementally with es.index(...)
  • Production-grade query DSL (filters, aggs, multi_match)
  • Battle-tested at petabyte scale

The course keeps minsearch as a fallback for environments where running a real search server is impractical (e.g. SaturnCloud notebooks). Pick the one that fits your runtime.

7 · Running Elasticsearch in Docker

🐳 One Command, Full Stack

The official Elasticsearch image is the fastest way to get a local cluster. Inside GitHub Codespaces (or any Docker host) one docker run is enough.

Run Elasticsearch 8.x locally
docker run -it \
  --rm \
  --name elasticsearch \
  -m 4GB \
  -p 9200:9200 -p 9300:9300 \
  -e "discovery.type=single-node" \
  -e "xpack.security.enabled=false" \
  docker.elastic.co/elasticsearch/elasticsearch:8.4.3

Two ports matter:

  • 9200 — the HTTP REST API your Python client talks to.
  • 9300 — internal node-to-node transport (only relevant for clusters).
Verify it's up
curl http://localhost:9200
# {
#   "name": "...",
#   "cluster_name": "docker-cluster",
#   "version": { "number": "8.4.3", ... },
#   "tagline": "You Know, for Search"
# }
If you're working in Codespaces, the 9200 port is auto-forwarded — but you only need it within the dev container, since the Python client connects to localhost:9200 from inside the same machine.
8 · Creating the Index

📇 An Index Is Like a Table

An Elasticsearch index is the equivalent of a table in a relational database. Before indexing data we declare its schema via mappings — most importantly, which fields are full-text-searchable (text) and which are exact-match filterable (keyword).

Connect to the cluster
from elasticsearch import Elasticsearch

es_client = Elasticsearch('http://localhost:9200')
es_client.info()                   # sanity check
Index settings & mappings
index_settings = {
    "settings": {
        "number_of_shards": 1,
        "number_of_replicas": 0,
    },
    "mappings": {
        "properties": {
            "text":     {"type": "text"},        # tokenised, full-text
            "section":  {"type": "text"},
            "question": {"type": "text"},
            "course":   {"type": "keyword"},     # exact match → for filters
        }
    },
}

index_name = "course-questions"
es_client.indices.create(index=index_name, body=index_settings)
Field type cheat-sheet
text → tokenised, lowercased, stemmed → use for matching
keyword → stored verbatim, single token → use for filtering / aggregations
The same three text fields and one keyword field as the minsearch setup — same logical schema, different engine.
9 · Indexing the Documents

📥 Pushing Data In

With the schema in place, indexing is a loop calling es.index(...). tqdm gives a progress bar — useful even for the small 948-document FAQ since indexing runs at ~30 docs/s.

Index every FAQ entry
from tqdm.auto import tqdm

for doc in tqdm(documents):
    es_client.index(index=index_name, document=doc)

Two things that make Elasticsearch different from minsearch:

  • Persistence — data is written to disk, so a notebook restart doesn't wipe the index. Re-running this loop would create duplicates; in production you'd guard with es.indices.exists().
  • Volume mounting — if the Docker container itself is recreated, you'd want a -v mount on /usr/share/elasticsearch/data to keep data across container lifecycles. Outside the scope of this notebook, but standard Docker practice.
For larger corpora use the helpers.bulk() API instead of one index call per document — it batches requests and is orders of magnitude faster.
10 · Querying With Boost & Filter

🎯 Translating the minsearch Query

The minsearch call had three concerns: a query string, a per-field boost (question^3), and a filter on course. Elasticsearch expresses all three in one DSL document — verbose, but very explicit.

Search query DSL
search_query = {
    "size": 5,
    "query": {
        "bool": {
            "must": {
                "multi_match": {
                    "query":  query,
                    "fields": ["question^3", "text", "section"],
                    "type":   "best_fields",
                }
            },
            "filter": {
                "term": {"course": "data-engineering-zoomcamp"}
            }
        }
    }
}

response = es_client.search(index=index_name, body=search_query)

Reading the DSL piece by piece:

  • size: 5 — return at most 5 hits, the equivalent of num_results=5 in minsearch.
  • bool.must.multi_match — score documents on full-text similarity across multiple fields.
  • question^3 — the caret syntax boosts the question field by 3×, identical to boost_dict={'question': 3.0}.
  • type: "best_fields" — take the score from the single best-matching field per document (other options: most_fields, cross_fields, phrase).
  • bool.filter.term — exact-match filter on the course keyword field. Filter clauses do not affect the relevance score, only inclusion — exactly what we want for "scope this query to one course".

📦 Unpacking the Response

The Elasticsearch response is deeply nested: response['hits']['hits'] is the list of hits, and each hit's original document lives under '_source'.

elastic_search() — drop-in replacement for search()
def elastic_search(query):
    search_query = {
        "size": 5,
        "query": {
            "bool": {
                "must":   {"multi_match": {
                    "query": query,
                    "fields": ["question^3", "text", "section"],
                    "type": "best_fields",
                }},
                "filter": {"term": {"course": "data-engineering-zoomcamp"}},
            }
        }
    }
    response = es_client.search(index=index_name, body=search_query)

    result_docs = []
    for hit in response['hits']['hits']:
        result_docs.append(hit['_source'])
    return result_docs
11 · Swapping It Into the Pipeline

🔌 Modularity Pays Off

Because search() and elastic_search() share the same signature — query in, list of dicts out — moving the whole RAG flow onto Elasticsearch is a one-line change.

rag() now backed by Elasticsearch
def rag(query):
    search_results = elastic_search(query)        # ← only line that changed
    prompt         = build_prompt(query, search_results)
    answer         = llm(prompt)
    return answer

rag('I just discovered the course. Can I still join it?')
# → "Yes, you can still join the course even if you discovered it after the start date…"

Same answer, but now backed by a persistent, scalable, production-ready search engine. Future swaps — Qdrant for vector search, a different LLM, a different prompt — follow the same pattern: replace one function, leave the rest alone.

12 · Key Takeaways

💡 The Big Picture

  • RAG = retrieve relevant context, build a grounded prompt, generate an answer with an LLM. None of the three steps is optional.
  • A good RAG prompt has a role, a "use only the context" rule, and a structured context block with field labels.
  • Express the pipeline as four small functions — search, build_prompt, llm, rag — and any component becomes swappable.
  • OpenAI is convenient but not mandatory: Mistral, Bedrock, Groq, Together, Anthropic, and Gemini all expose a chat-completions API, often with free credit.
  • Elasticsearch's multi_match + filter is the production-grade equivalent of minsearch's boost_dict + filter_dict. Same logical query, different engine.
  • Persistence, BM25 scoring, scalability, and incremental indexing are why you graduate from a toy in-memory index to a real search server.
