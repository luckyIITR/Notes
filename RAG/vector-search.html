Vector Search with Qdrant

From keyword matching to semantic similarity — building a small but production-shaped vector search solution with Qdrant in Docker, FastEmbed local inference, and a Q&A FAQ dataset.

1 · Why Vector Search

🔎 Beyond Exact Keywords

Traditional keyword search retrieves a document only when the query and the document share the same surface tokens. That works fine when you know the exact words present in the data, but it falls apart in two very common situations:

  • Different modality — searching an image, an audio clip, or a video against a text query. There are no shared "words" to match.
  • Same idea, different words — "flying mouse", "fruit bat", and "the bat hung in the cave" all describe the same animal, yet share almost no overlapping keywords.

Vector search sidesteps this by working on meaning rather than tokens. Each document, image, or audio clip is converted to a dense numerical vector — its embedding — and similarity becomes geometric closeness in that vector space.

Keyword Search

Matches exact tokens. Query "bat in cave" matches a document containing those literal words.

Fails on: synonyms, paraphrases, cross-modal queries (image, audio, video).

Vector Search

Matches meaning. Query "flying mouse" still retrieves bat documents, images of bats, or even bat sounds.

Works because: embeddings of related concepts land close together regardless of phrasing or modality.

Vector search compares vectorised representations, not text. Closeness is measured numerically with metrics like cosine similarity — the smaller the angle between two vectors, the more semantically similar they are.
2 · Qdrant — A Dedicated Vector Search Engine

⚡ What Qdrant Is and Why You'd Use It

Qdrant is an open-source vector search engine written in Rust. It is a dedicated solution for vector search, not a general-purpose database with an embedded ANN library — and the difference matters once you scale past a few thousand vectors.

Three reasons a dedicated vector database earns its keep:

  • Production-scale search — millions or billions of vectors, with predictable latency and throughput.
  • In sync with research — vector search is moving fast (multivectors, late interaction, hybrid search, rerankers). A dedicated engine keeps pace.
  • Beyond similarity — Qdrant supports payload filtering, hybrid (sparse + dense) search, recommendation, discovery, and unstructured data analysis on top of similarity.

Qdrant is open-source and flexible to deploy: self-hosted, Kubernetes, or managed cloud. For these notes we run a local Docker container — the smallest possible setup that still mirrors how you would deploy in production.

🐳 Running Qdrant in Docker

Pull the image and start a container. The whole engine is one binary; there is no separate database to provision.

5 · Step 2 — Study the Dataset

🧐 Look at Your Data Before You Vectorise It

Before touching any embedding model, study the dataset. The "schema" of your vector search solution — what gets embedded vs what gets stored as metadata — and the choice of embedding model both depend on the modality, language, and shape of the data.

Two questions to answer up front:

  • Modality & specifics — Is it text, images, audio, mixed? What language? Are the chunks short or long? Are there special tokens (URLs, code, formulas)?
  • What to vectorise vs what to filter — Some fields are content you want to search semantically; others are categorical labels you want to filter on (course, section, date, region, …).

📚 The FAQ Dataset

We use the same DataTalksClub Zoomcamp FAQ as the rest of the RAG notes — three courses, ~948 documents total, each with four fields:

🎯 What to Embed vs What to Filter

The classic semantic-similarity test: questions and answers about the same thing share almost no keywords.

Same idea, different words
Question   "I have registered for the bootcamp. When can I expect the confirmation email?"
Answer     "You don't need it. You're accepted. Registration is just to gauge interest before the start date."
Both share only the keyword "registered" — but a sentence like "Unregistered participants don't get certification" would also match that keyword while having a completely different meaning. Keyword search can't tell these apart; vector search can.

For a Q&A retrieval system the natural schema is:

  • Embed the text field (the answers) — that's what we'll search semantically with the incoming question.
  • Store course and section as metadata (payload) — used later for hard filtering ("only data-engineering questions").
  • Don't embed the question — it is the query side, not the indexed side.
6 · Offline / Online — How a Vector Search Pipeline Works

🛠️ Two Phases

Comparing two vectors is trivial; comparing one query against billions of stored vectors is not. Dedicated solutions split the work into an offline indexing phase and an online query phase.

Offline · 1Embed DocumentsEach answer → vector via the chosen model
Offline · 2Build IndexVectors stored in Qdrant; ANN index built
Online · 3Embed QuerySame model, same vector space
Online · 4Search IndexTop-k closest vectors returned, ranked by similarity
The single most important rule: query and documents must use the same embedding model. Different models live in different vector spaces and their distances are meaningless.
7 · Step 3 — Choose an Embedding Model with FastEmbed

🧠 Why FastEmbed

The choice of embedding model decides the quality of every result your system will ever return. It depends on the task, modality, language, domain, latency target, and budget. Test and benchmark on your own data when stakes are high.

For this notebook we use FastEmbed, Qdrant's own embedding library:

  • Tightly integrated — pass text directly to the Qdrant client; FastEmbed downloads the model and produces the vectors transparently.
  • Lightweight — uses ONNX Runtime with quantised weights, no PyTorch / TensorFlow dependency. Faster than Sentence Transformers on CPU.
  • Local inference — runs on your machine. Cost is your CPU; no third-party API calls.
  • Multiple vector types — dense (today), sparse (BM25-style, for hybrid search), multivectors (ColBERT, ColPali), image embeddings, rerankers.

📋 Listing Available Text Models

Filter by output dimensionality
from fastembed import TextEmbedding
import json

EMBEDDING_DIMENSIONALITY = 512

for model in TextEmbedding.list_supported_models():
    if model["dim"] == EMBEDDING_DIMENSIONALITY:
        print(json.dumps(model, indent=2))

Smaller embeddings (e.g. 512 dimensions) are a good default for a toy setup — vectors are big data objects, and dimensionality drives storage and memory cost. Higher dimensionality often means richer semantic capture, but at a price.

model_handle = "jinaai/jina-embeddings-v2-small-en"

📐 Cosine Similarity — The Distance Metric

Most dense embedding models, including Jina, are trained to be compared with cosine similarity — you can confirm this on the model's Hugging Face card. Cosine similarity measures the cosine of the angle between two vectors, ignoring their magnitudes.

Cosine similarity
cos(a, b) = ( a · b ) / ( ‖a‖ · ‖b‖ )
= 1 — vectors point the same way → semantically identical.
= 0 — orthogonal vectors → unrelated.
= -1 — opposite directions → semantic opposites.
Two parameters fully describe a dense embedding choice for Qdrant: the output dimension (here 512) and the distance metric (here cosine). Both are required to create a collection.
8 · Qdrant Core Concepts — Points and Collections

🧱 The Two Entities You Need to Know

Everything in Qdrant boils down to two objects:

Point

A single data record. Has an ID (integer or UUID), one or more vectors, and an optional payload (metadata as key-value pairs).

In our case: id = 0..947, vector = jina embedding of the answer, payload = {text, section, course}.

Collection

A named container of points sharing the same vector configuration. Think of it as one solution to one business problem.

In our case: a single collection zoomcamp-rag.

Payload

Optional structured metadata stored alongside each vector. Supports booleans, keywords, numbers, geo-locations, arrays, and nested objects.

Used for: filtering, sorting, and showing human-readable results.

Qdrant supports several vector types per point — dense (today's focus), sparse (BM25-style, for hybrid search), multivectors (ColBERT-style late interaction), and named vectors (multiple vectors per point). For semantic similarity on text, plain dense vectors are the right default.

9 · Step 4 — Create a Collection

🆕 Minimal Required Configuration

Creating a collection needs three things: a name, the vector size, and the distance metric.

Create a Qdrant collection
collection_name = "zoomcamp-rag"

client.create_collection(
    collection_name=collection_name,
    vectors_config=models.VectorParams(
        size=EMBEDDING_DIMENSIONALITY,        # 512, from the model
        distance=models.Distance.COSINE       # cosine similarity
    )
)

After this call, the collection exists but is empty — visible in the dashboard with the right dimensionality and metric, just no points inside yet. All other parameters (HNSW tuning, on-disk storage, quantisation, named vectors, sparse vectors) have sensible defaults you only touch when scale demands it.

10 · Step 5 — Embed and Upsert Points

📥 Building the Point List

Each point pairs an answer's vector with its metadata. The trick that makes FastEmbed integration painless: pass models.Document(text=..., model=...) instead of an explicit numpy array, and the Qdrant client handles the embedding for you.

Construct points with embedded text + payload
points = []
id = 0

for course in documents_raw:
    for doc in course['documents']:
        point = models.PointStruct(
            id=id,
            vector=models.Document(
                text=doc['text'],
                model=model_handle              # jinaai/jina-embeddings-v2-small-en
            ),
            payload={
                "text":    doc['text'],          # human-readable answer
                "section": doc['section'],
                "course":  course['course']
            }
        )
        points.append(point)
        id += 1

⬆️ Upserting Into the Collection

One call does everything: FastEmbed downloads the model on first use (cached under tempfile.gettempdir()/fastembed_cache), runs local inference on each text, and Qdrant inserts the resulting points and updates the vector index.

Upload points
client.upsert(
    collection_name=collection_name,
    points=points
)

Speed is dominated by local inference. To go faster:

  • Use GPU with FastEmbed (or a beefier CPU machine).
  • Use batch upsert via upload_collection or upload_points — both accept column- or record-oriented inputs and provide parallelisation, retries, and lazy batching out of the box.

12 · Step 6 — Similarity Search

🎯 How Search Works in Qdrant

  1. The query text is embedded with the same model used at indexing time, landing it in the same vector space.
  2. Qdrant compares the query vector against the stored vectors using the collection's distance metric (here cosine).
  3. The closest matches are returned, ranked by similarity score.

The vector index is built for approximate nearest neighbour (ANN) search — the cost of making large-scale vector search feasible is giving up perfect recall in exchange for sub-linear query time. Qdrant's index of choice is a custom variant of HNSW called Filterable HNSW, designed to keep accuracy high even when payload filters are applied at query time.

🔍 Defining a Search Function

Top-k semantic search
def search(query, limit=1):
    results = client.query_points(
        collection_name=collection_name,
        query=models.Document(
            text=query,
            model=model_handle                 # same model as upsert
        ),
        limit=limit,                           # top closest matches
        with_payload=True                     # return human-readable metadata
    )
    return results

🧪 Sanity Check — Question From the Dataset

Pick a random question from the dataset (note we never indexed the questions, only the answers) and run search against it. The expected behaviour: the matching answer is returned at the top, with a high cosine score.

Search with a known question
import random

course        = random.choice(documents_raw)
course_piece  = random.choice(course['documents'])

result = search(course_piece['question'])

print("Top retrieved answer:", result.points[0].payload['text'])
print("Score:",                  result.points[0].score)

The result contains a ScoredPoint with the matching point ID, the cosine score (e.g. 0.88), and the payload — the original answer text, section, and course.

🆕 Question Not in the Dataset

The real test is a query that doesn't exist in the indexed corpus at all.

Free-form natural-language query
print(search("What if I submit homeworks late?").points[0].payload['text'])
# → "No, late submissions are not allowed. But if the form is still not closed
#    and it's after the due date, you can still submit the homework..."
Vector search returned a sensible, semantically-on-target answer to a question phrased in completely different words from any indexed text. That is the entire promise of semantic search.
13 · Step 7 — Search With Filters

🎛️ Why Filters Matter

Pure semantic search ignores hard constraints. If a user asks "are late homeworks allowed?" while studying the MLOps course, the data-engineering answer is almost certainly wrong for them — even if it scores higher semantically. Payload filters let you pin search to specific subsets of the data.

Qdrant supports must, should, must_not, range, and other filter clauses. must is the AND-style requirement: every condition must be satisfied for a point to be returned.

📑 Indexing Payload Fields for Filters

For efficient filtering, build a payload index on the field you'll filter against. "keyword" means exact-match on a string.

Index the "course" payload field
client.create_payload_index(
    collection_name=collection_name,
    field_name="course",
    field_schema="keyword"            # exact string match
)

🔍 Filtered Search Function

Semantic search restricted to one course
def search_in_course(query, course="mlops-zoomcamp", limit=1):
    results = client.query_points(
        collection_name=collection_name,
        query=models.Document(text=query, model=model_handle),
        query_filter=models.Filter(
            must=[
                models.FieldCondition(
                    key="course",
                    match=models.MatchValue(value=course)
                )
            ]
        ),
        limit=limit,
        with_payload=True
    )
    return results

Now the same question — "What if I submit homeworks late?" — gives different, course-appropriate answers depending on the filter:

  • data-engineering-zoomcamp — strict: late submissions not allowed.
  • machine-learning-zoomcamp — softer: depends on whether the form is still open.
  • mlops-zoomcamp — no semantically close answer exists, so the result drops in score and relevance.
Qdrant's Filterable HNSW index is specifically engineered so that adding filters does not wreck recall — a known weakness of naive ANN-with-filters approaches that filter after the graph search.
14 · Plugging Vector Search Into a RAG Pipeline

🧩 RAG Is Three Steps — Only One of Them Changes

The RAG pattern from module 1 is a three-step function: search → build prompt → call LLM. The middle and the last step are independent of how you retrieve. Swapping the search backend — from minsearch to Qdrant vector search — is a one-line change.

ReplaceableSearchminsearch · Elasticsearch · Qdrant
StableBuild PromptInject retrieved context
StableLLM CallOpenAI · GPT-4o-mini

📜 The Starting Point — Keyword RAG

The starter notebook uses the same minsearch.Index retriever from module 1, with field boosts and a course filter:

Original RAG with minsearch
import minsearch
from openai import OpenAI

openai_client = OpenAI()

index = minsearch.Index(
    text_fields=["question", "text", "section"],
    keyword_fields=["course"]
)
index.fit(documents)

def search(query):
    return index.search(
        query=query,
        filter_dict={"course": "data-engineering-zoomcamp"},
        boost_dict={"question": 3.0, "section": 0.5},
        num_results=5
    )

def build_prompt(query, search_results):
    template = """You're a course teaching assistant. Answer the QUESTION based on the CONTEXT from the FAQ database.
Use only the facts from the CONTEXT when answering the QUESTION.

QUESTION: {question}

CONTEXT:
{context}""".strip()
    context = ""
    for doc in search_results:
        context += f"section: {doc['section']}\nquestion: {doc['question']}\nanswer: {doc['text']}\n\n"
    return template.format(question=query, context=context).strip()

def llm(prompt):
    response = openai_client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": prompt}]
    )
    return response.choices[0].message.content

def rag(query):
    search_results = search(query)
    prompt         = build_prompt(query, search_results)
    return llm(prompt)

Note the rename — the OpenAI client is now openai_client rather than just client, because we're about to introduce a second client (the Qdrant one) and we don't want them to collide.

🔁 Swapping In Qdrant Vector Search

The Qdrant pieces are exactly what we built in earlier sections — connect, create a collection, upsert points, query. Two small but important changes for RAG:

  1. Use a separate variable name qd_client for the Qdrant client, side-by-side with openai_client.
  2. Embed question + text together, not just the answer. RAG queries can match either side — keywords from the question or context from the answer — so giving the embedding both gives retrieval more surface to match against.
Setup — separate clients, new collection

# qd_client.delete_collection(collection_name=collection_name)

qd_client.create_collection(
    collection_name=collection_name,
    vectors_config=models.VectorParams(
        size=EMBEDDING_DIMENSIONALITY,
        distance=models.Distance.COSINE
    )
)

qd_client.create_payload_index(
    collection_name=collection_name,
    field_name="course",
    field_schema="keyword"
)
Upsert points — embed question + text together, full doc as payload
points = []
for i, doc in enumerate(documents):
    text   = doc["question"] + " " + doc["text"]   # both sides → one vector
    vector = models.Document(text=text, model=model_handle)
    points.append(models.PointStruct(
        id=i,
        vector=vector,
        payload=doc                                # whole doc kept for downstream prompts
    ))

qd_client.upsert(collection_name=collection_name, points=points)

🔍 The New vector_search Function

This replaces the original search entirely. Course filtering happens server-side via query_filter; the function returns a list of payload dicts so build_prompt doesn't need any change.

Drop-in replacement for the search step
def vector_search(question):
    print("vector_search is used")                # sanity-check trace
    course = "data-engineering-zoomcamp"

    query_points = qd_client.query_points(
        collection_name=collection_name,
        query=models.Document(text=question, model=model_handle),
        query_filter=models.Filter(
            must=[models.FieldCondition(
                key="course",
                match=models.MatchValue(value=course)
            )]
        ),
        limit=5,
        with_payload=True
    )

    return [point.payload for point in query_points.points]
RAG function — only the search call changes
def rag(query):
    search_results = vector_search(query)            # ← the only line that's different
    prompt         = build_prompt(query, search_results)
    return llm(prompt)

rag("how do I run kafka?")
# → "vector_search is used"
# → "To run Kafka, you need to follow these steps..."
RAG is modular. Build prompt and LLM stay identical; the retriever is a plug. That's why minsearch → Qdrant → Elasticsearch → hybrid is a one-function swap, and why evaluation in module 3 is meaningful — you can compare retrievers fairly with everything else held constant.
15 · Hybrid Search — Sparse, Dense, Fusion & Reranking

🪢 Why Combine Methods at All

Dense embeddings capture meaning but stumble on exact matches — product codes, identifiers, library names, version numbers, proper nouns the model never saw. Keyword methods (BM25, TF-IDF) excel at exact matches but ignore semantics. Real users send both styles of query. Hybrid search serves both audiences by running multiple retrieval methods and combining their results.

Dense Search

Captures meaning. "flying mouse" finds bat docs.

Always returns some results — even when nothing truly matches, vectors stay somewhat similar.

Sparse Search (BM25)

Captures exact words. "qdrant" finds Qdrant-mentioning docs.

Returns nothing when no token matches — useful negative signal.

🪶 Sparse Vectors

Keyword search can be expressed as vector search — but the vectors are sparse: dimensions = vocabulary size, but almost every entry is zero. Storing all those zeros would be wasteful, so vector databases keep only the non-zero entries. The big advantages over dense embeddings:

  • Flexible dictionary — sparse vectors are not limited to tokens seen during training. New product codes, fresh slang, future technical terms all get a column.
  • Fast to build — BM25 is purely statistical, no neural network involved.
  • Strong baseline — BM25 still tops several IR benchmarks; do not skip it.

📏 BM25 In One Paragraph

BM25 ("Best Matching 25" — the 25th attempt at the formula) is a ranking function combining three ideas:

BM25 — three signals
score(q, d)  ∝  Σt∈q   IDF(t) · TF(t, d) · length-norm(d)
TF rewards documents that contain query terms — but with diminishing returns, so 10 occurrences are not 10× better than 1.
IDF boosts rare terms and downweights common words that appear everywhere.
length-norm stops long documents from winning just by being long.

🆕 Sparse-Only Collection in Qdrant

Sparse vectors need a different config slot — sparse_vectors_config rather than vectors_config. Crucially, set Modifier.IDF so Qdrant computes IDF globally across the collection. Without it, BM25 won't reward rare words.

Sparse-only collection with global IDF
client.create_collection(
    collection_name="zoomcamp-sparse",
    sparse_vectors_config={
        "bm25": models.SparseVectorParams(
            modifier=models.Modifier.IDF        # Qdrant computes IDF for us
        )
    }
)
Upsert with the BM25 model from FastEmbed
import uuid

points = []
for course in documents_raw:
    for doc in course["documents"]:
        points.append(models.PointStruct(
            id=uuid.uuid4().hex,
            vector={
                "bm25": models.Document(
                    text=doc["text"],
                    model="Qdrant/bm25"             # statistical, no neural net
                )
            },
            payload=doc | {"course": course["course"]}
        ))

client.upsert(collection_name="zoomcamp-sparse", points=points)

This upsert finishes in seconds, not minutes — BM25 is statistical, not neural. The price of zero inference cost is paid in capability: BM25 cannot match synonyms or paraphrases.

🔍 Sparse Search

Querying a named vector requires using= to tell Qdrant which vector type to compare against.

BM25 search helper
def search(query, limit=1):
    return client.query_points(
        collection_name="zoomcamp-sparse",
        query=models.Document(text=query, model="Qdrant/bm25"),
        using="bm25",                          # pick the named sparse vector
        limit=limit,
        with_payload=True
    )

Two behaviours surprise people coming from dense search:

  • BM25 returns zero results when no query token appears in any document. Dense always returns something (some weak similarity). Empty results are useful signal.
  • BM25 scores are unbounded — not in [-1, 1] like cosine. They are only meaningful within a single query for ranking; you can't compare scores across queries, and you definitely can't compare BM25 vs cosine scores directly.
When the query is short and keyword-heavy ("qdrant", "pandas"), BM25 wins. When the query is a long natural-language sentence ("Uploading to S3 fails — what should I do?"), BM25 drowns in matched-but-irrelevant words. Different queries, different methods.

🛤️ Universal Query API & Prefetch

Qdrant's query_points is actually a universal query API. It can take one or more prefetch stages — a fast initial retriever pulls candidates, and a slower / more precise method reranks them. The classic search architecture (retrieve then rerank) collapses into one Qdrant call.

To use multiple methods on the same data, store both representations in the same collection using named vectors:

Collection with both sparse and dense named vectors
client.create_collection(
    collection_name="zoomcamp-sparse-and-dense",
    vectors_config={
        "jina-small": models.VectorParams(
            size=512,
            distance=models.Distance.COSINE
        )
    },
    sparse_vectors_config={
        "bm25": models.SparseVectorParams(modifier=models.Modifier.IDF)
    }
)
Each point carries two vectors
points.append(models.PointStruct(
    id=uuid.uuid4().hex,
    vector={
        "jina-small": models.Document(text=doc["text"], model="jinaai/jina-embeddings-v2-small-en"),
        "bm25":       models.Document(text=doc["text"], model="Qdrant/bm25"),
    },
    payload=doc
))

🪜 Multi-Stage Search — Dense Retrieve, Sparse Rerank

One useful pipeline: use the fast, semantic dense model to pull a wide set of candidates, then let BM25 re-order them based on exact-keyword overlap. The "main" query here applies to the candidates returned by prefetch, not the entire collection.

Dense retrieve → BM25 rerank
def multi_stage_search(query, limit=1):
    return client.query_points(
        collection_name="zoomcamp-sparse-and-dense",
        prefetch=[
            models.Prefetch(
                query=models.Document(text=query, model="jinaai/jina-embeddings-v2-small-en"),
                using="jina-small",
                limit=(10 * limit)             # pull 10× more candidates
            ),
        ],
        query=models.Document(text=query, model="Qdrant/bm25"),
        using="bm25",                          # rerank prefetched candidates
        limit=limit,
        with_payload=True
    )

🎲 Fusion — Reciprocal Rank Fusion (RRF)

The other strategy: run dense and sparse in parallel and combine their result lists. The challenge — dense and BM25 scores live on incomparable scales — is sidestepped by ignoring the raw scores entirely. Reciprocal Rank Fusion only looks at the position of each document in each ranked list.

Reciprocal Rank Fusion
rrf(d)  =  Σm   1 / ( k + rankm(d) )
m — each retrieval method (dense, sparse, …)
rankm(d) — position of document d in method m's list (1 = best)
k — small smoothing constant (typically 60)
A document ranked highly by multiple methods wins, even if no single method placed it first.

Concrete intuition: suppose dense ranks [D1, D2, D3, D4, D5] and sparse ranks [D5, D3, D1, D4, D2]. RRF doesn't pick D1 (the dense winner) or D5 (the sparse winner) — it picks D3, the document both methods agreed was good even though neither put it first. Consistent across methods often beats top of one method.

⚙️ RRF in Qdrant

Qdrant has RRF built in via FusionQuery. Both prefetches run, and the final query is "fuse the prefetched lists":

Hybrid search with RRF fusion
def rrf_search(query, limit=1):
    return client.query_points(
        collection_name="zoomcamp-sparse-and-dense",
        prefetch=[
            models.Prefetch(
                query=models.Document(text=query, model="jinaai/jina-embeddings-v2-small-en"),
                using="jina-small",
                limit=(5 * limit)
            ),
            models.Prefetch(
                query=models.Document(text=query, model="Qdrant/bm25"),
                using="bm25",
                limit=(5 * limit)
            ),
        ],
        query=models.FusionQuery(fusion=models.Fusion.RRF),
        with_payload=True
    )

🔄 Fusion vs Reranking

Reranking is the broader term — anything that re-orders an initial candidate set. Fusion is one specific kind of reranking, but reranking can be much more:

  • Cross-encoders — a model that consumes (query, document) jointly and scores them. Far more accurate than dense embeddings, far too slow to run on the whole corpus, perfect as a reranker over the top-k.
  • Multivectors / late-interaction models (ColBERT, ColPali) — represent each document as ~hundreds of small vectors and use a custom similarity (MaxSim). Resource-heavy but quality-superior; supported in Qdrant alongside named vectors.
  • Business rules — promote vendors you have contracts with, demote out-of-stock items, hard-pin editorial picks. These are rerankers too.
When Hybrid Search Helps
  • Mixed-style queries (keywords + sentences)
  • Domain with proper nouns, IDs, codes
  • Chatbot UX with broad audience
  • You don't know your query distribution yet
Watch Out
  • Two retrievers = two latency budgets
  • Two upserts = more storage
  • Quality gains need real evaluation, not cherry-picked queries
  • RRF requires sane k and prefetch limits
Hybrid search is a family, not a single algorithm — multi-stage rerank, RRF fusion, cross-encoder rerank, business-rule rerank, late-interaction multivector. Pick based on query distribution, latency budget, and what your evaluation set actually tells you.
16 · Key Takeaways

💡 What to Remember

  • Vector search compares meaning, not tokens. Same idea phrased differently still matches; cross-modal queries become possible.
  • Cosine similarity is the default distance metric for dense text embeddings — confirm it on the model card before configuring the collection.
  • The two phases of a vector search system are offline indexing (embed + insert) and online query (embed query + nearest-neighbour search). The same model must be used in both.
  • Qdrant boils down to two entities — points (id + vector + optional payload) and collections (named groups of points with one vector configuration).
  • Decide what to embed (the searchable content) and what to keep as payload (categorical or filterable metadata) before picking a model.
  • FastEmbed handles model download, local inference, and serialisation — pass models.Document(text=..., model=...) and the client does the rest.
  • Vector search at scale is approximate — Qdrant's Filterable HNSW gives sub-linear queries while preserving high recall under payload filters.
  • The dashboard's Visualize tab is invaluable for understanding your data: 2D projections expose clusters, gaps, and surprises before you write any query.
  • Filters (must, should, must_not, ranges, …) make semantic search contextual — the same query can return different answers depending on user, region, or product.
  • RAG is three modular steps — search, build prompt, LLM. Swapping retrievers is a one-function change; everything else stays the same.
  • For RAG, embed question + answer together: the query may match either side, and combining them gives the embedding more surface to score against.
  • Use distinct names — qd_client vs openai_client — to keep Qdrant and OpenAI clients from colliding once both live in the same notebook.
  • Sparse vectors (BM25) excel at exact matches, identifiers, and proper nouns; dense vectors excel at meaning. Real query distributions need both.
  • BM25 is statistical, not neural — seconds to index thousands of documents and a strong baseline you should always have on hand.
  • Dense and BM25 scores are not directly comparable. Hybrid search either avoids the issue (RRF, which uses ranks) or trains a model to do the combining.
  • Qdrant's universal query API (query_points with prefetch) collapses retrieve-then-rerank pipelines into a single call.
  • Reciprocal Rank Fusion rewards documents ranked highly by multiple methods — consistency beats winning a single list.
  • Reranking is broader than fusion: cross-encoders, multivector / late-interaction models, and even business rules are all rerankers.
  • Vector search is the foundation, not the ceiling — Qdrant also powers hybrid search, recommendation, discovery, and unstructured data analysis.
