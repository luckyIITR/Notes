RAG Evaluation — Offline & Online

Beyond retrieval — how do you evaluate the whole RAG system, including the prompt and the LLM? An overview of offline vs online evaluation, plus the two foundational offline techniques: answer-cosine similarity and LLM-as-a-judge.

1 · Recap — Where We Are in the RAG Series

🧭 The Three Steps, Revisited

Across the previous modules we built a complete RAG pipeline as a small, modular function:

The recurring rag(query) shape

    search_results = search(query)              # module 1, 2 — text / vector / hybrid
    prompt         = build_prompt(query, search_results)
    answer         = llm(prompt)                    # module 2 — OpenAI or alternative
    return answer

In module 3 we evaluated the search step in isolation — Hit Rate and MRR against a ground-truth dataset of synthetic queries. That answers "is the retriever finding the right documents?" — but says nothing about the rest of the pipeline.

Retrieval is necessary but not sufficient for a good RAG system. A perfect retriever paired with a bad prompt or a hallucinating LLM still gives bad answers. We need to evaluate the whole system end-to-end.
2 · The Open Questions

❓ What's Left to Measure

Module 3's metrics covered the retrieval box. Everything to the right of it is still untested:

  • Is our prompt good? — Different prompt templates, system messages, and context-formatting strategies all change the answer quality. None are obviously best a priori.
  • Is the LLM the right one? — GPT-4o vs GPT-4o-mini vs Claude vs an open-source 70B model. Quality, latency, and cost all trade against each other.
  • How good is the answer the user actually sees? — Even with perfect retrieval and a strong LLM, the final answer may be incomplete, hallucinated, or off-topic.
  • Is the system getting better or worse over time? — Models drift, prompts get edited, source data changes. You cannot manage what you don't watch.

These questions need different metrics and a different evaluation methodology than HR/MRR. That methodology splits cleanly along one axis: do we measure before or after deployment?

3 · Offline vs Online Evaluation

🔀 Two Modes, Different Roles

Every production ML system — RAG included — gets evaluated twice: once before deployment on a fixed dataset, and once after deployment on real user traffic. Neither replaces the other; they answer different questions.

Offline Evaluation

Before deployment. Run the system on a fixed dataset, compute metrics, compare against a baseline.

Examples: Hit Rate, MRR (retrieval) · Cosine similarity between answers · LLM-as-a-judge.

Strength: reproducible, cheap to iterate, runs in CI. Weakness: dataset bias, may not reflect real user behaviour.

Online Evaluation

After deployment. Watch real users, collect feedback, run controlled experiments, monitor health.

Examples: A/B tests on business metrics · Thumbs-up / thumbs-down · Click-through rates · System health dashboards.

Strength: ground truth from real users. Weakness: needs traffic, slow to iterate, expensive to get wrong.

Offline evaluation tells you "the new version is probably better". Online evaluation tells you "the new version is actually better for real users". Ship the offline winner, then confirm online — never skip either.

🩺 Monitoring Lives Alongside Online Eval

Monitoring is the continuous form of online evaluation. Where A/B tests answer "is variant B better?", monitoring answers "is the system healthy right now?" — and it covers both system-level signals (CPU, latency, error rate) and quality signals (user feedback, answer length distribution, refusal rate).

4 · Offline RAG Evaluation — Two Foundational Methods

🧪 Reusing the Ground Truth

The synthetic ground truth from module 3 already gives us thousands of (question, expected_document) pairs. Each pair carries an original answer — the FAQ entry the question was rephrased from. That answer is the reference point for evaluating generation quality.

The evaluation triangle
question  →  rag(question)  =  generated_answer
question  ↦  expected_doc  .text  =  original_answer
For every ground-truth row we now have two answers — the original and the LLM-generated one. The whole game is to measure how close they are.

🛠️ Two Techniques, Different Trade-offs

Cosine similarity

Embed both answers, take cosine similarity of the vectors. Higher = more semantically aligned.

cheapdeterministicneeds reference
LLM-as-a-judge

Ask an LLM to rate how well the generated answer matches the original (or just answers the question).

flexiblecostlyreference optional

Both produce per-query scores you average across the dataset. Cosine similarity is what you reach for first — fast, free, and reproducible. LLM-as-a-judge is the heavier hammer for nuanced quality dimensions cosine cannot capture (factuality, helpfulness, tone).

5 · Method 1 — Cosine Similarity Between Answers

📐 The Idea

Run the full rag(question) on every ground-truth row to produce a generated answer. Embed both the original answer and the generated answer with the same embedding model. Take the cosine similarity. Average across queries.

  1. For each ground-truth row, look up the original answer (the FAQ entry the row was generated from).
  2. Run generated_answer = rag(row["question"]).
  3. Embed both answers with the same model used in retrieval (e.g. multi-qa-MiniLM-L6-cos-v1).
  4. Compute cosine(embed(original), embed(generated)) ∈ [-1, 1].
  5. Average across all rows → one number per (retriever, prompt, LLM) configuration.

📏 What the Number Means

  • Higher = more semantically aligned. The LLM is producing answers that say similar things to the source FAQ entry.
  • Embedding-space, not surface-level. Paraphrases, reordering, and added/removed wording all stay close — exact-string match would be unfairly strict.
  • Comparable across configurations. Swap the LLM, swap the prompt, swap the retriever — same metric, same dataset, apples-to-apples.
When Cosine Works
  • You have a reference / original answer per query
  • You want a fast, cheap, deterministic score
  • You want to compare many configurations (prompt sweeps, model sweeps)
  • Quality differences are big enough to register in semantic similarity
Where It Fails
  • Two answers can be semantically similar but one is factually wrong
  • Tone, helpfulness, and specificity are invisible to cosine
  • Embedding model bias becomes evaluation bias
  • Useless when no reference answer exists
6 · Method 2 — LLM-as-a-Judge

⚖️ The Idea

Cosine similarity captures "are these saying similar things?". It cannot capture "does this actually answer the question?" or "is this factually correct?". For those, ask another LLM.

LLM-as-a-judge is a structured prompt that hands the judge model the inputs and asks for a graded score (or a verdict like RELEVANT / PARTLY_RELEVANT / NON_RELEVANT). It comes in two flavours depending on whether a reference answer exists:

With reference

"Here is the original answer. Here is the LLM's answer. How similar / how good?"

used when ground-truth exists
Without reference

"Here is the question. Here is the answer. Does it actually answer the question?"

used in production / open-domain

📝 The Pattern

  1. Build a clear judging prompt — system role + criteria + a structured output format (typically JSON with verdict and explanation).
  2. For each ground-truth row, pass (question, original_answer, generated_answer) — or just (question, generated_answer) when no reference exists — into the judge LLM.
  3. Parse the judge's response. Aggregate verdicts across the dataset (% RELEVANT, average score, etc.).
  4. Spot-check a sample by hand — judge models hallucinate too. Calibration matters.
LLM-as-a-judge is flexible but biased. Stronger judges (GPT-4-class) correlate well with human ratings on most dimensions, but they share the prompt-style biases of every LLM. Always cross-check a sample manually — and never use the same model as judge and as the system being judged.

💸 Cost & Reproducibility

  • Cost — one judge call per evaluated row. For a 4,600-row dataset on GPT-4o-class, expect single-digit to low-double-digit dollars per full eval pass.
  • Cache verdicts by (question, generated_answer) hash — repeated re-runs of identical inputs should not pay twice.
  • Pin the judge model and prompt in your eval logs. Switching either changes the metric — note it the same way you'd note an evaluation-set version.
  • Sample, don't always full-run — judge a stratified sample of 200–500 rows for fast iteration; reserve the full sweep for major releases.

🧠 Re-using the Best Retriever

From the vector-search evaluation, question_text_vector won decisively (HR ≈ 0.94 / MRR ≈ 0.85). Drop in the same embedding model, the same Elasticsearch KNN function, and the same per-query encoder.

The vector retriever from earlier modules
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("multi-qa-MiniLM-L6-cos-v1")

def elastic_search_knn(field, vector, course):
    knn = {"field": field, "query_vector": vector,
           "k": 5, "num_candidates": 10000,
           "filter": {"term": {"course": course}}}
    body = {"knn": knn,
            "_source": ["text", "section", "question", "course", "id"]}
    response = es_client.search(index=index_name, body=body)
    return [hit["_source"] for hit in response["hits"]["hits"]]

def question_text_vector(q):
    v_q = model.encode(q["question"])
    return elastic_search_knn("question_text_vector", v_q, q["course"])
Module 3 evaluated the retriever; modules 1–2 built the prompt and the LLM call. Offline RAG eval just glues them back together and compares the final answer to the original answer the question was generated from.
8 · Step 2 — A RAG Function That Takes a Dict

🔌 Two Small Generalisations

The original rag(query: str) from module 1 needs two tweaks before it can be evaluated at scale:

  1. Take a dict, not a string. Each ground-truth row carries question and course. Passing the whole dict avoids re-deriving the course from elsewhere and matches the shape question_text_vector already expects.
  2. Make the LLM model a parameter. Hard-coding "gpt-4o" means we cannot compare configurations. Default to a strong model; allow callers to swap in cheaper ones for cost / latency sweeps.
build_prompt + llm + rag — generalised
from openai import OpenAI
client = OpenAI()

def build_prompt(query, search_results):
    template = """You're a course teaching assistant. Answer the QUESTION based on the
CONTEXT from the FAQ database. Use only the facts from the CONTEXT.

QUESTION: {question}

CONTEXT:
{context}""".strip()
    context = ""
    for doc in search_results:
        context += f"section: {doc['section']}\nquestion: {doc['question']}\nanswer: {doc['text']}\n\n"
    return template.format(question=query, context=context).strip()

def llm(prompt, model="gpt-4o"):                  # model is now a parameter
    response = client.chat.completions.create(
        model=model,
        messages=[{"role": "user", "content": prompt}]
    )
    return response.choices[0].message.content

def rag(query: dict, model="gpt-4o") -> str:        # dict, not string
    search_results = question_text_vector(query)
    prompt         = build_prompt(query["question"], search_results)
    return llm(prompt, model=model)

That single model= parameter is what enables every comparison in the rest of these notes — GPT-4o vs GPT-3.5-turbo vs GPT-4o-mini, all driven from the same rag(...) call site.

9 · Step 3 — Toy Cosine Example on One Row

🧪 Sanity-Check Before Spending Money

Before iterating over 4,627 rows at GPT-4o pricing, verify the cosine pipeline works on a single ground-truth row. The recipe is exactly what §5 described, just instantiated.

Single-row cosine demonstration
record         = ground_truth[10]                              # any row
original_doc   = doc_index[record["document"]]
answer_orig    = original_doc["text"]                            # reference answer

answer_llm     = rag(record)                                    # run the full RAG pipeline

v_llm          = model.encode(answer_llm)
v_orig         = model.encode(answer_orig)

similarity     = v_llm.dot(v_orig)                              # cosine since model normalises
print(similarity)
# → 0.84 (typical for a sensible answer; 1.0 = identical, 0.0 = unrelated)

Two implementation details worth knowing:

  • multi-qa-MiniLM-L6-cos-v1 outputs L2-normalised embeddings, so the dot product equals the cosine similarity. No need for an explicit cosine function.
  • Cosine values land in [-1, 1] in theory, but for sensible Q&A answers compared with a Q&A-trained embedder, expect [0.4, 0.95] in practice. Anything below ~0.3 deserves a manual look.
10 · Step 4 — Generating Answers Across the Dataset

🔁 The Cache-as-You-Go Pattern (Again)

Same lesson as §9 of retrieval-evaluation.html: when a loop is going to call OpenAI 4,627 times, make it resumable. Store every result in a dict keyed by row index, and skip indices that already have a value. Define the storage dict in a separate cell from the loop, so a re-run of the loop doesn't wipe the cache.

Cell 1 — define the cache (run once)
from tqdm.auto import tqdm
answers = {}                                  # index → {answer_llm, answer_orig, document}
Cell 2 — the resumable generation loop
for i, record in tqdm(enumerate(ground_truth), total=len(ground_truth)):
    if i in answers:
        continue

    answer_llm    = rag(record)                              # model defaults to gpt-4o
    doc_id        = record["document"]
    answer_orig   = doc_index[doc_id]["text"]

    answers[i] = {
        "answer_llm":  answer_llm,
        "answer_orig": answer_orig,
        "document":    doc_id
    }

⏱️ What the Sequential Run Costs

  • Time — almost 3 hours end-to-end on a single thread. The OpenAI API itself isn't slow; one-request-at-a-time discipline is.
  • Money — about $10 on GPT-4o for the full ground-truth dataset.
  • Reliability — sequential runs survive transient errors gracefully thanks to the cache. Parallel runs that crash mid-way without a cache lose everything.
Sequential generation with a per-row cache is the safe default for first runs of a new evaluation pipeline. Optimise for survival first, throughput second.

📦 Persist the Results

Build a flat list aligned with ground_truth, merge in the question / course / document fields, and dump to CSV. The output rows now contain everything needed to run cosine similarity, LLM-as-a-judge, or any other answer-level metric — without ever calling the LLM again.

Build aligned results and save
results_gpt4o = [None] * len(ground_truth)

for i, value in answers.items():
    results_gpt4o[i] = value.copy()
    results_gpt4o[i].update(ground_truth[i])         # adds question, course

df_gpt4o = pd.DataFrame(results_gpt4o)
df_gpt4o.to_csv("data/results-gpt4o.csv", index=False)

The resulting frame columns: answer_llm, answer_orig, document, question, course. That's the canonical shape every offline-eval metric in the rest of this notebook (and the next video) will consume.

11 · Step 5 — Speeding It Up: Threads & Cheaper Models

🧵 Parallel Calls With ThreadPoolExecutor

OpenAI's API is I/O-bound — most of the wall time is spent waiting for the server, not running Python. The straightforward speed-up is a thread pool that issues several requests in flight at once. Six concurrent workers is a sensible default; higher numbers risk hitting rate limits, which kills the run completely.

map_progress — parallel iterator with a tqdm bar
from concurrent.futures import ThreadPoolExecutor

pool = ThreadPoolExecutor(max_workers=6)

def map_progress(pool, seq, f):
    results = []
    with tqdm(total=len(seq)) as progress:
        futures = [pool.submit(f, x) for x in seq]
        for fut in futures:
            results.append(fut.result())
            progress.update(1)
    return results

Refactor the per-row work into a single process_record that the pool can submit:

process_record — one row of the eval
def process_record(rec):
    model       = "gpt-3.5-turbo"                       # cheaper, faster
    answer_llm  = rag(rec, model=model)
    doc_id      = rec["document"]
    answer_orig = doc_index[doc_id]["text"]

    return {
        "answer_llm":  answer_llm,
        "answer_orig": answer_orig,
        "document":    doc_id,
        "question":    rec["question"],
        "course":      rec["course"],
    }

results_gpt35 = map_progress(pool, ground_truth, process_record)
df_gpt35      = pd.DataFrame(results_gpt35)
df_gpt35.to_csv("data/results-gpt35.csv", index=False)

📉 What Two Switches Buy

Combining parallelism (six workers) with a smaller model (gpt-3.5-turbo) on the same 4,627-row eval:

ConfigurationWall timeCost
GPT-4o · sequential~3 hours~$10
GPT-3.5-turbo · 6 threads~6m 32s~$0.79

That's roughly 30× faster and 12× cheaper. Whether the quality holds up is a question that only the cosine and judge metrics can answer — which is exactly the point of having an offline evaluator. Without it, "is GPT-3.5-turbo good enough?" is a vibe; with it, it's a number.

When to Parallelise
  • You have a stable pipeline you've already run sequentially once
  • Rate limits are generous (corporate / paid tier)
  • The cost of re-running on failure is acceptable
  • You're sweeping many configurations and time matters
When to Stay Sequential
  • First run of a new pipeline — surface bugs before scaling them
  • Tight rate limits — parallel calls just trigger throttling
  • Per-call cost is significant — failures lose more money
  • You need deterministic, debuggable execution order
The same rag(query, model=...) signature now powers three configurations (GPT-4o, GPT-3.5-turbo, and later GPT-4o-mini) on the same dataset. Three CSV files in, three rows in a comparison table out — the next video computes cosine and the LLM-as-judge metrics on top.
12 · Step 6 — Computing Cosine Across the Dataset

📐 A→Q→A' Cosine Similarity

Each ground-truth row carries an original answer A; from A we synthetically generated a question Q back in module 3; through the RAG pipeline that Q produced a regenerated answer A'. The metric computes cosine(A, A') per row and aggregates across the dataset. Loosely, "the closer A' lands to A in embedding space, the better the system answered its own paraphrased question."

AOriginal answerdoc_index[doc_id]["text"]
QSynthetic questionFrom the ground-truth row
A'RAG-generated answerFrom rag(record, model=...)

🔁 Loading Saved Results & Computing Per Row

If you're picking up from a fresh kernel, load the CSVs you wrote in §10 / §11 and convert them back to record dicts. Each metric is one function applied to each row; tqdm shows progress.

Reload results & compute cosine row-by-row
# continuing in same notebook → df_gpt4o is already in memory
# fresh notebook → df_gpt4o = pd.read_csv("data/results-gpt4o.csv")

results_gpt4o = df_gpt4o.to_dict(orient="records")

def compute_similarity(record):
    answer_orig = record["answer_orig"]
    answer_llm  = record["answer_llm"]
    v_llm       = model.encode(answer_llm)
    v_orig      = model.encode(answer_orig)
    return v_llm.dot(v_orig)               # L2-normalised → cosine

similarity = []
for record in tqdm(results_gpt4o):
    similarity.append(compute_similarity(record))

df_gpt4o["cosine"] = similarity        # attach metric column
df_gpt4o["cosine"].describe()

The full sweep takes roughly 5 minutes — pure CPU work on the embedder, no API calls. Repeat the same three lines for the other two CSVs, naming the columns similarity_35 and similarity_4o_mini.

⚡ A Speed-Up You Can Do Later

Each original answer is repeated several times in the ground truth — module 3 generated five questions per FAQ entry, so encode(answer_orig) recomputes the same vector five times. Pre-computing one embedding per unique doc_id cuts work by ~5×:

Optional optimisation — cache encodings of original answers
orig_vecs = {                                        # compute once per unique doc
    doc_id: model.encode(d["text"])
    for doc_id, d in doc_index.items()
}

def compute_similarity_fast(record):
    v_llm  = model.encode(record["answer_llm"])
    v_orig = orig_vecs[record["document"]]
    return v_llm.dot(v_orig)

Skipped in the live demo for clarity, but worth knowing exists when the dataset gets larger.

13 · Step 7 — Visualising the Distribution

📈 Why Plot, Not Just Average

A single mean hides the tails. Two systems can have identical means but very different distributions — one with a heavy left tail of catastrophic failures, another with steady mid-quality answers. Overlaying KDE plots makes that obvious in seconds.

Three-way KDE overlay with seaborn
import seaborn as sns
import matplotlib.pyplot as plt

sns.distplot(df_gpt4o["cosine"],      label="4o")
sns.distplot(df_gpt35["cosine"],      label="3.5")
sns.distplot(df_gpt4o_mini["cosine"], label="4o-mini")

plt.title("RAG LLM performance — A→Q→A' cosine")
plt.legend()
plt.show()

What to look at on the plot:

  • Mode location — where most of the answers cluster. A higher mode means most rows are good.
  • Right shoulder — the share of near-perfect answers (cosine > 0.85). Heavier is better.
  • Left tail — the share of bad answers (cosine < 0.4). Even one model with a thicker left tail is risky in production, even if its mean looks fine.
  • Overlap — when two distributions overlap heavily, the practical difference between them is small. Use cost / latency as the tie-breaker.

sns.distplot is deprecated in modern seaborn; for new code prefer sns.kdeplot(..., fill=True) or sns.histplot(..., kde=True, stat="density"). The interpretation is identical.

🧪 Beyond Picking a Model

The same harness answers other configuration questions with no extra plumbing:

  • Prompt changes — tweak build_prompt, re-run on the ground truth, compare cosine distributions before and after.
  • Retriever swaps — replace question_text_vector with hybrid search or a different embedder; the rest of the pipeline is unchanged.
  • Temperature / decoding params — pass through llm() the same way as the model parameter and sweep.
  • Context-window experiments — change top-k, change context formatting, see whether answer-level quality moves.

Cosine alone won't catch every failure mode — semantically different but factually wrong answers can still score high. That's why the next video adds LLM-as-a-judge on top of this same generated dataset. Two complementary metrics on a fixed corpus is the standard offline-eval pattern.

15 · Step 9 — LLM-as-a-Judge: Two Prompt Variants

⚖️ Why a Second Metric

Cosine similarity grades by semantic distance. Two answers can be cosine-close because they share vocabulary, even when one is factually wrong. LLM-as-a-judge grades by relevance — does the generated answer actually solve the question? — and the answer is a small string label, not a continuous score, which makes failures easy to inspect by hand.

The same (answer_orig, question, answer_llm) rows from §10 / §11 feed straight into this new metric. No new dataset, no new retrieval — just a new prompt around the same data.

🔀 Two Prompts, Two Use-Cases

Two situations call for two judging prompts:

Prompt 1 — A→Q→A' (with reference)
  • Inputs: original answer, generated question, generated answer
  • Use-case: offline evaluation only — needs the gold answer
  • Stronger signal — the judge can directly compare two answers
  • The "did we recover the right thing?" question
Prompt 2 — Q→A' (no reference)
  • Inputs: only the question and generated answer
  • Use-case: works online too — there is no original answer for real-user traffic
  • Weaker (no reference) but always available
  • The "did we answer this user's question?" question
Prompt 1 is for the offline harness; Prompt 2 is the one that ports to production for real-time monitoring of generated answers — the very signal Grafana / Postgres dashboards in the next module will surface.

📝 Both Prompts Verbatim

The course author asked GPT to write these prompts (meta-prompting), then iterated until the output was strictly parseable JSON. Three properties of a good judging prompt are visible here: a clear role, a fixed label vocabulary, and a strict output schema.

prompt1_template — with reference answer
prompt1_template = """
You are an expert evaluator for a Retrieval-Augmented Generation (RAG) system.
Your task is to analyze the relevance of the generated answer compared to the original answer provided.
Based on the relevance and similarity of the generated answer to the original answer, you will classify
it as "NON_RELEVANT", "PARTLY_RELEVANT", or "RELEVANT".

Here is the data for evaluation:

Original Answer: {answer_orig}
Generated Question: {question}
Generated Answer: {answer_llm}

Please analyze the content and context of the generated answer in relation to the original
answer and provide your evaluation in parsable JSON without using code blocks:

{{

prompt2_template — no reference, online-friendly
prompt2_template = """
You are an expert evaluator for a Retrieval-Augmented Generation (RAG) system.
Your task is to analyze the relevance of the generated answer to the given question.
Based on the relevance of the generated answer, you will classify it
as "NON_RELEVANT", "PARTLY_RELEVANT", or "RELEVANT".

Here is the data for evaluation:

Generated Answer: {answer_llm}

  "Explanation": "[Provide a brief explanation for your evaluation]"
}}
""".strip()

The double curly braces ({{ and }}) escape the Python .format(...) placeholders so the literal JSON braces survive prompt assembly. The single-brace fields ({answer_orig}, {question}, {answer_llm}) match the column names in the results CSV — every record dict can **record straight into .format(**record).

16 · Step 10 — Sampling, Building Prompts & JSON Discipline

🎯 Sample, Don't Sweep

Cosine on every row is cheap (CPU only). LLM-as-judge on every row is another 1,830 LLM calls — for a metric that only needs a representative sample to be informative. n = 150 with a fixed random seed gives reproducible numbers fast enough to iterate on the prompt itself.

Reproducible sample → list of dicts
df_sample = df_gpt4o_mini.sample(n=150, random_state=1)   # seed → reproducible
samples   = df_sample.to_dict(orient="records")         # dicts that .format(**record) into the prompt

🔁 The Judging Loop

Same shape as every other LLM loop in this notebook — build prompt, call llm(), append the raw string. JSON parsing happens in a second pass so a single bad row doesn't lose the whole batch.

Run prompt 1 across the sample
evaluations = []
for record in tqdm(samples):
    prompt     = prompt1_template.format(**record)
    evaluation = llm(prompt, model="gpt-4o-mini")
    evaluations.append(evaluation)                       # raw string for now

For 150 rows this finishes in roughly 3 minutes on GPT-4o-mini.

🧱 The JSON Block-Tag Bug

First-pass parsing fails on a handful of rows because the model wraps its JSON in a Markdown code fence:

```json
{
  "Relevance": "RELEVANT",
  "Explanation": "..."
}
```

Same failure mode as ground-truth generation in module 3. Two fixes, applied together:

  1. Patch the prompt — add "in parsable JSON without using code blocks" directly to the instruction line. This is what the verbatim prompts in §15 already include.
  2. Track failures by index — when a row still slips through, fix that single string in place rather than re-running the whole batch. The notebook used a manual answers[i] = "..." patch and re-parsed.
Two-pass parsing — survive single-row failures
import json

json_evaluations = []
for i, raw in enumerate(evaluations):
    try:
        json_evaluations.append(json.loads(raw))
    except json.JSONDecodeError as e:
        print(f"row {i}: {e}")            # inspect, fix manually, retry
Whenever an LLM is supposed to emit structured output, run prompts on a sample first, parse, and only then scale up. Catching format bugs at row 5 instead of row 1,830 saves both money and the will to live.

📊 Aggregate the Labels

df_evaluations = pd.DataFrame(json_evaluations)

For GPT-4o-mini on a 150-row sample of the ground truth: roughly 83% relevant, 11% partly, 7% non-relevant. That ~7% non-relevant slice is where the value of this metric lives.

🔍 Inspecting What Went Wrong

Filter the frame to the failures and read the Explanation field. Each one is a free-text diagnostic from the judge — much easier to act on than a low cosine score.

Pull the failure rows for inspection
df_evaluations[df_evaluations.Relevance == "NON_RELEVANT"]

Sample explanations from the run (paraphrased):

  • "The generated answer discusses a pip version error related to scikit-learn, which is not directly connected to the original answer focused on Docker build error."
  • "The generated answer does not address the specific question..."
  • "The generated answer incorrectly states that..."
  • "The generated answer addresses a different issue..."

Three failure-mode patterns recur across most NON_RELEVANT rows:

  1. Retrieval miss — the right document wasn't in the top-k, so the LLM answered from an unrelated context. Fix: revisit retriever, top-k, hybrid weights.
  2. Prompt drift — the LLM took a tangentially related context and ran with it. Fix: tighten the prompt, add "answer ONLY using the context" guards.
  3. Question ambiguity — the synthetic question itself was bad. Fix: improve the ground-truth-generation prompt; consider filtering low-quality rows.
Cosine returns a number; LLM-as-a-judge returns a diagnostic. The two metrics complement each other — cosine for tracking trend lines across runs, judge for actually understanding what's broken.

Prompt 2 is identical mechanically — the only change is the prompt template (no answer_orig) and the output frame name. Crucially, this is the version that can run in production: real users supply only the question; there is no gold answer to compare against.

🪜 What This Sets Up Next

The non-relevant rows from prompt 2 are exactly the events a production system wants to catch online:

  • Wire prompt 2 into the live RAG service — rate every generated answer at request time, asynchronously.
  • Log (question, answer_llm, Relevance, Explanation) tuples to Postgres.
  • Surface the NON_RELEVANT rate (and recent failure explanations) on a Grafana dashboard.
  • Trigger alerts when the failure rate crosses a threshold — your retriever or your prompt has just regressed.
The same prompt that grades the offline sample becomes the online quality monitor — one judge, two contexts. That's why the no-reference variant exists at all.
19 · Online Evaluation — A Quick Tour

📡 What Real Users Tell You

Once the system is deployed, users become the evaluation set. The patterns:

A/B TestCompare variantsSplit traffic, measure business metrics
FeedbackThumbs up/downDirect quality signal from users
MonitoringSystem healthLatency, errors, refusal rate

🔬 A/B Tests

Split traffic between the current system (control) and a new variant (treatment). Measure business metrics downstream — task completion rate, session length, conversion, retention — not the offline metrics. Statistically rigorous, slow to read, the gold standard for "did this change actually help?".

👍 User Feedback (Thumbs / Stars)

The lightest-weight online signal. Add a thumbs-up / thumbs-down next to every generated answer. Aggregate over time, slice by query type, and you get a per-cohort quality trend. Easy to wire up; biased (only some users click), but directionally informative.

Implementing user feedback is a recurring topic in LLM-Zoomcamp — it's the simplest thing that gives a continuous quality signal in production.

📊 Monitoring

Continuous observation of system health. Two layers:

  • Infrastructure layer — CPU, memory, request latency, error rate, queue depth. Standard SRE concerns; existing tooling (Prometheus, Datadog, Grafana) just works.
  • Quality layer — LLM-specific. Average answer length, refusal rate, token usage per query, retrieval hit-rate on production traffic, user-feedback rate. These need custom dashboards.
Monitoring and online evaluation blur into each other in production — both watch live traffic, both compute metrics over time. The line is fuzzy on purpose: "is the system healthy?" and "is the system answering well?" should be visible on the same dashboard.

21 · Key Takeaways

💡 What to Remember

  • Retrieval evaluation (HR, MRR) measures only the search step. The full RAG system needs end-to-end evaluation covering prompt quality, LLM choice, and final answer quality.
  • Every production system splits evaluation into offline (before deployment, on a fixed dataset) and online (after deployment, on real users). Neither replaces the other.
  • Offline evaluation is reproducible and cheap; online evaluation is realistic but slow. Ship the offline winner, then confirm online.
  • Reuse the synthetic ground truth from module 3 — it provides (question, original_answer) pairs for free.
  • Cosine similarity between original and generated answers is the deterministic, cheap, first-pass offline metric.
  • Cosine fails on factuality, tone, and helpfulness — it sees only semantic alignment.
  • LLM-as-a-judge covers what cosine can't, in two variants — with reference (closer to ground truth) and without (closer to production usage).
  • Pin the judge model + prompt + sample size in eval logs. Treat the judge as part of the metric.
  • Online tools — A/B testing, user feedback (thumbs), monitoring dashboards — answer different questions: "is variant B better?", "did users like this?", "is it healthy?".
  • Monitoring spans infrastructure (latency, errors) and quality (refusal rate, feedback rate, answer length). Both belong on the same dashboard.
  • Implementation in upcoming videos: cosine similarity, then LLM-as-a-judge, then user-feedback wiring, then monitoring.
  • Offline RAG eval reuses everything from module 3 — same corpus, same ground truth, same retriever. Only the answer-comparison step is new.
  • Generalise the RAG function to take a dict (question + course) and a model parameter — both are needed to run multiple configurations through the same harness.
  • Always sanity-check the cosine pipeline on a single row before unleashing it on thousands at GPT-4o pricing.
  • Cache LLM outputs in a dict keyed by row index, defined in a separate cell. Loops crash; you should not lose 2 hours of API calls when they do.
  • For Q&A-trained, L2-normalised embedders (e.g. multi-qa-MiniLM-L6-cos-v1), the dot product equals cosine similarity — no need for a wrapper function.
  • First run sequentially, then parallelise. ThreadPoolExecutor with 6 workers turns ~3 hours into ~6 minutes for I/O-bound LLM calls — but only after the pipeline is debugged.
  • The model parameter on rag(...) turns "is GPT-3.5-turbo good enough?" from a vibe into a measurable comparison: GPT-3.5-turbo is ~30× faster and ~12× cheaper than GPT-4o on this benchmark; quality is what you measure next.
  • Persist results as flat CSVs with columns answer_llm, answer_orig, document, question, course. Every downstream metric (cosine, judge, custom) reads the same shape.
  • A→Q→A' cosine — embed both answers, dot-product, attach as a cosine column, run describe(). Five minutes of CPU, no API calls.
  • Negative cosine values are not bugs — they flag genuinely bad ground-truth pairs (ambiguous synthetic question or off-topic reference). Sample and inspect, don't clip.
  • Each original answer repeats five times in the ground truth (5 questions per FAQ entry). Pre-computing one embedding per doc_id gives ~5× speed-up when datasets get larger.
  • Always overlay the distributions, not just the means — two systems with the same mean can have very different left tails.
  • On this dataset, GPT-4o-mini matches GPT-4o quality (mean 0.680 vs 0.679, median essentially equal) at ~36× lower cost. GPT-3.5-turbo is ~2 percentage points behind on both stats.
  • Cosine alone won't catch fluent-but-wrong answers — that's what LLM-as-a-judge in the next video covers.
  • Two judging prompts: prompt 1 (A→Q→A') uses the original answer as a reference and is offline-only; prompt 2 (Q→A') has no reference and can run online on real-user traffic.
  • Sample n = 150 with a fixed seed for the judge — full sweeps are unnecessary for a metric whose value is a label, not a score.
  • LLMs love wrapping JSON in Markdown code fences. Add "in parsable JSON without using code blocks" to the prompt and parse in a second pass so single-row failures don't take the batch down.
  • On GPT-4o-mini answers: ~83% RELEVANT, ~11% PARTLY_RELEVANT, ~7% NON_RELEVANT. The 7% slice is where the metric pays for itself — each row comes with a human-readable diagnostic.
  • Three recurring failure modes the explanations expose: retrieval misses, prompt drift, and question ambiguity. Each maps to a different fix.
  • Cosine returns a number; the judge returns a diagnostic. Use cosine for trend lines across runs, judge for understanding what's broken.
  • The no-reference judge (prompt 2) is what feeds the future Grafana / Postgres production dashboard — same prompt, two contexts.
