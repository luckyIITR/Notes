Retrieval Evaluation

How do you actually know which retriever is best? An overview of why evaluation matters in RAG, what a ground-truth dataset looks like, where to get one, and which ranking metrics make sense for the FAQ-style problem at hand.

1 · Why Evaluate Retrieval

🎯 "It Depends" Is Not an Answer

By now you've seen several ways to do the R in RAG — minsearch as a toy in-memory index, Elasticsearch with text search, and Qdrant with vector search. A natural question follows: which one is best?

The honest reply is "it depends" — on the dataset, the query distribution, the latency budget, and what your users actually care about. But "it depends" is only useful if you can turn it into a number. That's what evaluation does.

1Pick a retrieverminsearch · ES · Qdrant · hybrid
2Run all queriesFrom a labelled dataset
3Score the resultsCompare to known relevant docs
4Compare retrieversHighest score wins for your data
Without evaluation, choosing a retriever is folklore. With evaluation, it's a measurement. Same retriever wins on different data sets for different reasons — that's why "best in general" doesn't exist.
2 · The Tunable Surface

🎚️ So Many Knobs, So Little Time

Even within a single retriever, the search behaviour is shaped by a long list of choices. Without a ground-truth dataset, every one of these is a guess.

  • Search type — pure text, pure vector, hybrid (BM25 + dense), multi-stage rerank.
  • Field selection — index question, text, section separately or together?
  • Field boosts — does question: 3 beat question: 5? Should section get a small 0.5 boost or be excluded entirely?
  • Filters — hard filters by course, soft filters by section.
  • Top-k — how many candidates does the LLM actually need?
  • Embedding model — Jina, MiniLM, BGE, Cohere, OpenAI. Different vector spaces, different qualities.
  • Vector database — Qdrant, Elasticsearch, FAISS, Milvus, Weaviate, Pinecone.
  • Indexing strategy — embed only the answer, only the question, both concatenated, or chunked differently.

The combinatorial explosion is not theoretical — every one of these makes a real difference on real data. The only way to navigate it is to fix everything else and measure one change at a time against a stable, labelled benchmark.

3 · Ground Truth — The Centerpiece

📋 What a Gold-Standard Dataset Looks Like

Retrieval evaluation needs a ground-truth dataset (also called gold-standard data or relevance judgments). For each query, it lists the documents that should appear in the search results.

"I just discovered the course, can I still join?"
Relevant doc_1 doc_10 doc_11
"How do I run Kafka?"
Relevant doc_42 doc_117

A useful evaluation set is hundreds to thousands of (query → relevant docs) entries. With it, every retriever's output can be compared against the truth: did the relevant documents show up at all? At the top? In the top-k?

Schema of a ground-truth row
( query  ,  relevant_doc_ids )
query — the text a user might submit.
relevant_doc_ids — the set of documents the system should retrieve. Sometimes one ID, sometimes many. Sometimes graded (highly relevant / partially relevant / not relevant).
The ground-truth dataset is the compass for every retrieval decision you make. Spend more time on getting it right than on tuning hyperparameters — a clean compass beats a fast guess.
4 · Where Ground Truth Comes From

🏭 Three Sources, Different Trade-Offs

User Feedback

Implicit (clicks, dwell time, "thumbs up") or explicit (ratings) signals from a deployed system.

Pros: real, free, scales with traffic.
Cons: needs production traffic, biased toward what was already shown.

Human Annotators

People (in-house experts or external annotators) read query + document pairs and judge relevance.

Pros: high quality, gold standard for benchmarks.
Cons: slow, expensive, hard to scale to thousands of queries.

LLM-Generated

An LLM generates plausible queries from each document and names which documents the query points to.

Pros: cheap, fast, scales well.
Cons: noisy — biased by the LLM's own preferences.

In a mature production system, you typically sample data from real traffic and have annotators (or an LLM) label the relevance. Without a deployment, you can't sample anything — so for these notes, we generate ground truth synthetically with an LLM, then use it to compare retrievers.

5 · Why Synthetic Ground Truth Works Here

🤖 The LLM-as-User Trick

The FAQ dataset has a special structure: each entry is already a (question, answer) pair. We know that this question maps to this answer. But real users won't phrase questions exactly like the FAQ author did.

The trick: ask an LLM to rephrase each FAQ question into 5 different ways a real user might ask it. Each rephrasing is a synthetic query, and we already know its relevant document — the original FAQ entry. That gives us thousands of (synthetic query → known document) pairs, virtually for free.

  1. Take an FAQ document (question, answer, course, section).
  2. Prompt an LLM: "Generate 5 alternative ways a student might phrase this question."
  3. Each generated query inherits the original document ID as its relevant doc.
  4. Repeat across all FAQ entries → ground-truth dataset of size 5 × |corpus|.
  5. Use this dataset to evaluate every retriever you want to compare.
Synthetic ground truth is noisy but consistent. The absolute scores may be optimistic, but relative comparisons between retrievers stay meaningful — which is exactly what we need to pick the best one.
6 · Step 1 — Stable Document IDs

🆔 Why IDs Come First

Ground-truth rows are (query, relevant_doc_id). To name the relevant document, every record in the knowledge base needs a stable identifier — one that is the same today, tomorrow, and after the next FAQ refresh.

The naïve approach is just to assign a sequential number:

Sequential IDs — convenient but fragile
for i in range(len(documents)):
    documents[i]["id"] = i

This breaks the moment the FAQ is reordered, an entry is added, or one is deleted — every downstream relevant_doc_id in the ground-truth set silently points to the wrong document. Useless for any system that will be re-indexed.

🥇 The Ideal — IDs from the Source

The best identifier is one the source itself already maintains. The DataTalksClub FAQs are Google Docs; each heading has an anchor ID inside the URL that survives renaming the heading. Had the corpus been parsed via the Google Docs API instead of python-docx on a downloaded .docx file, those anchor IDs would have been kept and the problem would already be solved.

Since the IDs are gone, second-best is to derive a stable hash from the content itself.

🔐 Content-Hash IDs

Combine fields that uniquely identify a record (course + question + a slice of the answer text), MD5-hash the result, and take the first 8 hex characters. The ID does not depend on row order — only on content.

Stable, content-derived 8-char IDs
import hashlib

def generate_document_id(doc):
    combined = f"{doc['course']}-{doc['question']}-{doc['text'][:10]}"
    hash_object = hashlib.md5(combined.encode())
    return hash_object.hexdigest()[:8]

for doc in documents:
    doc["id"] = generate_document_id(doc)

Worth noting:

  • Why include the first 10 chars of the answer? Some FAQ entries share the exact same question heading. Adding a slice of the answer text breaks ties without making the ID depend on tiny edits across the whole answer.
  • Why MD5 and not a longer hash? We're not signing anything — we just need a deterministic short identifier. 8 hex chars = 168 ≈ 4 billion buckets, plenty for a few thousand FAQ rows.
  • Trade-off — content hashes change when the question or first 10 chars of the answer are edited. Acceptable here; not acceptable for systems where stability across edits is required (those need source IDs).
Pick the most stable ID source you have access to. Source IDs > content hashes > sequential numbers. An unstable ID is worse than no ID — it lets bugs hide in plain sight.
7 · Step 2 — Detecting Duplicate IDs

🔁 Hash Collisions vs Real Duplicates

Hash IDs can collide for two completely different reasons:

  • True content duplicates — two FAQ entries with the same course + question + answer prefix. The hash is doing its job; the data is just dirty.
  • Same heading, different answers — same question text and course but different answer bodies. Including the answer prefix in the hash usually breaks these apart.

Either way, you want to see the collisions before generating a ground-truth dataset built on top of them.

Group by ID with defaultdict — surface collisions
from collections import defaultdict

hashes = defaultdict(list)
for doc in documents:
    hashes[doc["id"]].append(doc)

print(len(hashes), len(documents))      # (947, 948) → one collision

for doc_id, items in hashes.items():
    if len(items) > 1:
        print(doc_id, len(items))
        # 593f7569 2 — same question, identical first 10 chars of answer

defaultdict(list) is the canonical Python pattern for "group by" — every new key is initialised to an empty list, so append just works without an if check. Comparing len(hashes) to len(documents) tells you instantly how many collisions you have.

For the FAQ corpus, the lone surviving collision is a real data duplicate — two near-identical entries that differ only by an annotation suffix. The pragmatic fix would be to dedupe the source data; the practical fix is to shrug and move on. Real-world data is dirty.

💾 Persist the Tagged Documents

Save the ID-tagged corpus once so the rest of the pipeline (and you, six months from now) can re-use it without re-running the hash.

documents-with-ids.json
import json

with open("documents-with-ids.json", "wt") as f_out:
    json.dump(documents, f_out, indent=2)
8 · Step 3 — The Question-Generation Prompt

📝 Prompt Engineering for Synthetic Queries

The whole approach turns on one prompt: ask an LLM to impersonate a student and write the kinds of questions that student might ask, given an FAQ entry as context. Three details matter more than they look.

prompt_template
prompt_template = """
You emulate a student who's taking our course.
Formulate 5 questions this student might ask based on a FAQ record. The record
should contain the answer to the questions, and the questions should be complete and not too short.
If possible, use as fewer words as possible from the record.

The record:

section: {section}
question: {question}
answer: {text}

Provide the output in parsable JSON without using code blocks:

["question1", "question2", ..., "question5"]
""".strip()
  1. "Complete and not too short" — without it, the LLM produces telegraphic 3-word fragments that don't resemble real student questions. Forces realistic phrasing.
  2. "Use as fewer words as possible from the record" — the entire point of synthetic ground truth is to test retrieval against queries phrased differently from the source. If the LLM copies the FAQ wording, text-search wins by accident and you learn nothing.
  3. "Parsable JSON without using code blocks" — without it, GPT-4 returns the JSON wrapped in ```json … ``` markdown fences, which json.loads cannot parse. Removing the fences in the prompt is much cleaner than stripping them in post-processing.

🔌 Calling the LLM

generate_questions(doc)
from openai import OpenAI
client = OpenAI()

def generate_questions(doc):
    prompt = prompt_template.format(**doc)         # section, question, text → slots
    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[{"role": "user", "content": prompt}]
    )
    return response.choices[0].message.content     # raw JSON string

Note: prompt_template.format(**doc) works because the FAQ dicts already contain the keys section, question, and text — the template's slot names match the schema exactly.

A good synthetic-data prompt tells the LLM who it is (a student), what to produce (5 questions), what shape the output must take (raw JSON, no code blocks), and what to avoid (copying words). Skip any of these and you spend the next hour cleaning data instead of evaluating retrievers.
9 · Step 4 — The Generation Loop

🔁 Cache-as-You-Go

Generating questions for ~950 FAQ entries means ~950 OpenAI calls. Things will go wrong: rate limits, transient network errors, your laptop sleeping. The pattern that survives all of these is the same: store every result keyed by document ID, and skip IDs that are already done.

Resumable generation loop
from tqdm.auto import tqdm

results = {}                                          # doc_id → raw JSON string

for doc in tqdm(documents):
    doc_id = doc["id"]
    if doc_id in results:                          # already generated → skip
        continue
    results[doc_id] = generate_questions(doc)
  • Resume after rate limits — when the loop dies 30% through, restart it: every previously-completed ID is found in results and skipped instantly.
  • Free deduplication — when two FAQ entries collide on the same hash (the duplicate from §7), the LLM is only called once for them, saving an API request.
  • Cost in practice — with GPT-4o on the ~950-document FAQ, the full generation comes out to about $4 end-to-end. Use gpt-4o-mini if budget matters more than quality.

🥒 Persist the Raw Output

Pickle the results so you never have to pay the LLM cost twice. Plain JSON works too; pickle is just convenient when the keys are hex-string doc IDs and the values are unparsed strings.

Save / restore generated questions
import pickle

with open("results.bin", "wb") as f_out:
    pickle.dump(results, f_out)

# Later — load without re-calling the LLM:
with open("results.bin", "rb") as f_in:
    results = pickle.load(f_in)
10 · Step 5 — From Raw Output to Ground-Truth CSV

📤 Parsing JSON (And Patching the One That Breaks)

Even with "no code blocks" in the prompt, the LLM occasionally returns invalid JSON — a stray apostrophe inside a string, a missing comma, a literal smart quote. The realistic flow:

  1. Try to json.loads every result.
  2. For the 1–2 that fail, print the offending string, fix it by hand, and reinject the cleaned JSON into results[doc_id].
  3. Re-run the parse loop. Now everything passes.
Parse all generated JSON strings
parsed_results = {}
for doc_id, json_questions in results.items():
    parsed_results[doc_id] = json.loads(json_questions)

# If it fails on one — manually fix that single string:
# results["abcd1234"] = json.dumps(["fixed q1", "fixed q2", ...])

Spending five minutes patching two malformed responses is far cheaper than re-running the entire $4 generation just to dodge them.

🧱 Building the Final Triples

Each generated question becomes one ground-truth row: (question, course, document_id). To attach the course, build a quick dictionary from doc ID to full document — an O(1) lookup instead of scanning the list each time.

doc_index for fast lookup
doc_index = {d["id"]: d for d in documents}

final_results = []
for doc_id, questions in parsed_results.items():
    course = doc_index[doc_id]["course"]
    for q in questions:
        final_results.append((q, course, doc_id))

For ~948 FAQ entries × 5 questions each, this yields roughly 4,700 ground-truth rows.

11 · Ranking Evaluation Metrics — The Landscape

📊 Why "Ranking" and Not Just "Accuracy"

Search and recommender systems return a ranked list: for a given query, every document gets a relevance score, and the top-k are returned in order. Evaluation needs to reward not just "the relevant document was found" but "the relevant document was found at the top".

The standard ranking metrics — there are many — measure different facets of this. None is universally best; pick based on what your downstream system actually needs.

Hit Rate @ k

Did any relevant document appear in the top-k? Binary, simplest. Used in these notes.

MRR — Mean Reciprocal Rank

Average of 1 / rank-of-first-relevant. Rewards finding the first relevant doc as early as possible. Used in these notes.

Precision @ k

Fraction of top-k that are relevant. Cares about how many good results are in the top-k.

Recall @ k

Fraction of all relevant docs that show up in the top-k. Cares about coverage.

MAP — Mean Average Precision

Averages precision at every rank where a relevant doc appears. Penalises burying good results.

nDCG @ k

Normalised Discounted Cumulative Gain. Rewards higher placement; supports graded relevance. The de-facto standard.

🎯 Why Hit Rate & MRR for the FAQ Problem

For the FAQ-style RAG setup the choice is simple:

  • Each query in the synthetic dataset has exactly one relevant document (the original FAQ entry it was rephrased from). Recall and precision become coarse with single-target queries — Hit Rate captures it cleanly.
  • The downstream LLM only sees the top few results — pushing the relevant doc to position 1 matters far more than including it at position 4. MRR rewards exactly this.
  • Both are easy to compute, easy to explain, and easy to plot retriever-by-retriever.

Other metrics are not wrong — they're tuned for different problems. nDCG shines when relevance is graded (highly / partially / not relevant). MAP shines when each query has many relevant docs. Pick based on the shape of your data.

12 · Hit Rate — Did We Find It?

🎯 The Simplest Possible Metric

Hit Rate at k asks one yes/no question per query: did at least one relevant document appear in the top-k results? Average across all queries to get the dataset-wide hit rate.

Hit Rate @ k
HR@k  =  (1 / |Q|) · Σq∈Q   1[ hit(q, k) ]
Q — set of queries in the evaluation set.
hit(q, k) — true if at least one relevant doc appears among the top-k results for query q.
1[·] — indicator function: 1 if the condition is true, 0 otherwise.
Range: [0, 1]. 1.0 means every query found its relevant doc.
Hit Rate in three lines
def hit_rate(relevance_per_query):
    # relevance_per_query: list of lists of bools, one per query
    # Each inner list = is each retrieved doc relevant?
    return sum(any(r) for r in relevance_per_query) / len(relevance_per_query)
  • Strength — easy to interpret. "Our retriever finds the answer 87% of the time."
  • Limit — ignores rank within the top-k. A relevant doc at position 1 looks identical to position 5.
  • Best for — first-pass benchmarking, RAG where the LLM can read all top-k anyway.
13 · MRR — Did We Find It Quickly?

🥇 Reward Putting the Relevant Doc First

Mean Reciprocal Rank looks at the position of the first relevant document. The earlier it appears, the higher the score: position 1 → 1.0, position 2 → 0.5, position 5 → 0.2, never appears → 0.0.

Mean Reciprocal Rank
MRR  =  (1 / |Q|) · Σq∈Q   1 / rankq
rankq — position of the first relevant doc for query q (1-indexed). Use ∞ if no relevant doc appears in the top-k.
Range: [0, 1]. Heavily skewed toward top placements — the gap between rank 1 and rank 2 is the same as between rank 2 and infinity.
MRR in five lines
def mrr(relevance_per_query):
    total = 0.0
    for r in relevance_per_query:                         # r = [True, False, ...]
        for rank, hit in enumerate(r, start=1):
            if hit:
                total += 1 / rank
                break
    return total / len(relevance_per_query)
  • Strength — directly captures "is the answer the first thing the LLM sees?" The whole reason rankings exist.
  • Limit — only considers the first relevant doc. Doesn't reward retrieving multiple relevant docs.
  • Best for — single-target evaluation (like our synthetic FAQ set), and any system where users only look at the top result.
Hit Rate and MRR together cover most of what matters in a RAG retriever: did we find it? (Hit Rate) and did we find it fast? (MRR).
14 · Step 6 — Building the Relevance Set

🧪 One Pass Over Ground Truth

With ground-truth-data.csv (~4,627 synthetic queries) and a retriever in hand, evaluation reduces to one loop. For each ground-truth row, run the retriever, ask "is the expected document anywhere in the top-k?", and record a list of booleans — one bool per returned result.

Importantly, this requires the corpus to be indexed with the same stable IDs we generated in §6 — the search results must carry the id field so we can match them against the ground-truth row's expected document.

🔧 Indexing With IDs in Elasticsearch

The ID is added as a keyword field — exact-match, not tokenised. We could rely on Elasticsearch's built-in _id, but storing the ID in the document body is simpler and means the rest of the code stays unchanged.

Index settings — id as keyword
from elasticsearch import Elasticsearch
from tqdm.auto import tqdm

es_client = Elasticsearch("http://localhost:9200")

index_name = "course-questions"
index_settings = {
    "settings": {"number_of_shards": 1, "number_of_replicas": 0},
    "mappings": {
        "properties": {
            "text":     {"type": "text"},
            "section":  {"type": "text"},
            "question": {"type": "text"},
            "course":   {"type": "keyword"},
            "id":       {"type": "keyword"}      # exact match
        }
    }
}

es_client.indices.delete(index=index_name, ignore_unavailable=True)
es_client.indices.create(index=index_name, body=index_settings)

for doc in tqdm(documents):
    es_client.index(index=index_name, document=doc)
elastic_search — multi_match with course filter
def elastic_search(query, course):
    search_query = {
        "size": 5,
        "query": {
            "bool": {
                "must": {
                    "multi_match": {
                        "query":  query,
                        "fields": ["question^3", "text", "section"],
                        "type":   "best_fields"
                    }
                },
                "filter": {"term": {"course": course}}
            }
        }
    }
    response = es_client.search(index=index_name, body=search_query)
    return [hit["_source"] for hit in response["hits"]["hits"]]

⚙️ The Relevance Loop

Build relevance_total — one list of bools per query
relevance_total = []

for q in tqdm(ground_truth):
    doc_id    = q["document"]
    results   = elastic_search(query=q["question"], course=q["course"])
    relevance = [d["id"] == doc_id for d in results]
    relevance_total.append(relevance)

The output structure is exactly what the metric functions in §12–§13 expect: a list[list[bool]] where each inner list has length-5 (top-5 results). Elasticsearch is fast — about ~100 queries per second on a single shard — so all 4,627 rows finish in roughly a minute.

In single-target retrieval (one relevant doc per query), Hit Rate equals Recall. The terms get used interchangeably here; with multi-target queries they diverge.

⚖️ Versus minsearch

Same evaluation, different retriever. minsearch is the toy in-memory TF-IDF index from module 1 — it accepts an extra id field as a keyword field so results carry the same identifiers Elasticsearch produces.

minsearch with id as keyword
import minsearch

index = minsearch.Index(
    text_fields=["question", "text", "section"],
    keyword_fields=["course", "id"]
)
index.fit(documents)

def minsearch_search(query, course):
    return index.search(
        query=query,
        filter_dict={"course": course},
        boost_dict={"question": 3.0, "section": 0.5},
        num_results=5
    )

16 · Step 8 — A Reusable Evaluator

♻️ One Function, Any Retriever

The two evaluation runs above were nearly identical — only the inner search call differed. The natural refactor is a generic evaluate(ground_truth, search_function) that takes the retriever as a parameter. Pass any callable that maps a ground-truth row to a list of result dicts.

evaluate — drop-in any search function
def evaluate(ground_truth, search_function):
    relevance_total = []
    for q in tqdm(ground_truth):
        doc_id    = q["document"]
        results   = search_function(q)
        relevance = [d["id"] == doc_id for d in results]
        relevance_total.append(relevance)

    return {
        "hit_rate": hit_rate(relevance_total),
        "mrr":      mrr(relevance_total),
    }

The search_function takes the whole row dict instead of fixed positional arguments — that way each retriever can pull whatever fields it needs (question, course, possibly more for filtered or hybrid searches).

Calling evaluate for each retriever
evaluate(ground_truth, lambda q: elastic_search(q["question"], q["course"]))
# → {'hit_rate': 0.7396, 'mrr': 0.6032}

evaluate(ground_truth, lambda q: minsearch_search(q["question"], q["course"]))
# → {'hit_rate': 0.7722, 'mrr': 0.6615}

evaluate(ground_truth, lambda q: minsearch_search_tweaked(q["question"], q["course"]))
# → both metrics drop. Revert.

The "obvious" change makes things worse. Without the evaluator that wouldn't be visible — gut feeling and cherry-picked queries cannot tell you that the section field, with a tiny boost of 0.5, was actually pulling its weight.

What This Setup Buys You
  • One number per retriever — apples-to-apples comparison
  • Easy to swap retrievers (text, vector, hybrid) into the same harness
  • Hyperparameter changes become measurements, not opinions
  • Reproducible: same CSV, same metrics, anyone can re-run
Watch Out
  • Synthetic ground truth is biased — the LLM's idea of "a student question" isn't a student
  • Single-target queries underestimate retrievers that bring back multiple correct docs
  • Re-running tweaks one at a time gets messy fast — graduate to MLflow or similar for serious tuning
  • The 25% of queries the system misses deserve manual review — they often expose data issues, not retrieval ones
A retrieval evaluator is an experimentation flywheel: try a tweak, measure, decide, commit. It turns "I think hybrid is better" into "hybrid lifts MRR from 0.60 to 0.71 on our 4,627-query benchmark" — the second sentence is the one that survives a code review.
17 · Evaluating Vector Search — The Setup

🧭 What Changes vs Text Evaluation

Vector search swaps lexical matching for semantic similarity, but the evaluation harness from §14–§16 stays untouched. Same ground-truth-data.csv, same evaluate(ground_truth, search_function), same Hit Rate and MRR. Only the inner search_function changes — and that's the whole point of the modular design.

Three new design choices appear in the vector path that don't exist in the text path:

  1. Which embedding model? — small fast (MiniLM, 384-dim), strong general-purpose (BGE, E5, mxbai-large, 768–1024-dim), domain-specific (BioBERT, SciBERT, code-search models), or proprietary (OpenAI text-embedding-3-*, Cohere). Different vector spaces, different qualities.
  2. Which field to embed? — the question, the answer text, or their concatenation. This single choice typically moves Hit Rate by 5–15 points on Q&A data.
  3. Exact vs approximate KNN? — exact (brute-force) is the ceiling on retrieval quality. ANN (HNSW, IVF) trades a sliver of recall for orders-of-magnitude speed. Always evaluate both when you care about quality.

🎯 Industry-Standard Evaluation Protocol

The IR community has converged on a small set of practices for benchmarking dense retrievers. Stick to them and your numbers will be comparable to MTEB / BEIR results that practitioners trust.

  • Freeze the ground truth — version it (ground-truth-data-v1.csv), do not regenerate it between experiments. Otherwise you are measuring the LLM's mood, not your retriever.
  • Cache embeddings — encode the corpus once, save vectors to disk (.npy, parquet, or directly into the index). A 950-doc test set is fast; a million-doc corpus is not.
  • Evaluate exact KNN first — this is your retrieval quality ceiling. ANN tuning happens after, comparing against this number.
  • Report top-k for k in {1, 5, 10} — Hit Rate@1 and MRR@10 are common defaults. RAG systems usually consume top-3 to top-10.
  • Hold one variable constant per run — model, field, k, hybrid weights. Otherwise diff attribution becomes guesswork.
  • Track everything — model name, embedding-field, index params, k, num_candidates, ANN type, and the resulting metrics. MLflow, Weights & Biases, or even a CSV all work.
  • Pair quality with latency — P50 / P95 query latency matters as much as MRR for production. A 0.02 MRR gain that doubles latency is rarely worth it.
Vector search evaluation is three separate sweeps — embedding model, embedding field, retrieval method (vector / hybrid / rerank) — held against a frozen ground-truth and pinned to the same metrics. Anything else is wishful thinking.
18 · Step 9 — Embedding the Corpus

🧠 Picking and Loading an Embedding Model

For an FAQ Q&A corpus, a strong CPU-friendly default is multi-qa-MiniLM-L6-cos-v1 from sentence-transformers — explicitly trained for Q&A retrieval, 384 dimensions, normalised for cosine similarity. It is the scikit-learn LogisticRegression of dense retrieval: simple, fast, surprisingly hard to beat.

Load the embedding model
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("multi-qa-MiniLM-L6-cos-v1")   # 384 dims, cosine
EMBEDDING_DIM = 384

🎚️ Three Embedding Strategies

For each FAQ document, compute three different vectors. This lets us evaluate which field carries the most retrieval signal — a fundamental design question that almost no one asks because the answer is "it depends".

question_vector

Embeds the FAQ question only. Tends to win when synthetic queries are question-like.

text_vector

Embeds the answer only. Tends to win when queries are statement-like ("the course has started, can I join?").

question_text_vector

Embeds question + " " + text. Usually a safe winner across query styles — more surface area to match against.

Compute and attach all three vectors
from tqdm.auto import tqdm

for doc in tqdm(documents):
    question = doc["question"]
    text     = doc["text"]

    doc["question_vector"]      = model.encode(question).tolist()
    doc["text_vector"]          = model.encode(text).tolist()
    doc["question_text_vector"] = model.encode(question + " " + text).tolist()

For ~948 documents, three encodings each, this finishes in a couple of minutes on CPU. For larger corpora, batch the inputs (model.encode(batch, batch_size=64)) and use a GPU — encoding cost dominates everything downstream.

🗄️ Indexing Dense Vectors in Elasticsearch

Elasticsearch's dense_vector field stores vectors and (with index: true) builds an HNSW graph for ANN search. Three vector fields = three independent HNSW indexes you can switch between at query time.

Mapping with three dense_vector fields
index_settings = {
    "settings": {"number_of_shards": 1, "number_of_replicas": 0},
    "mappings": {
        "properties": {
            "text":     {"type": "text"},
            "section":  {"type": "text"},
            "question": {"type": "text"},
            "course":   {"type": "keyword"},
            "id":       {"type": "keyword"},

            "question_vector": {
                "type": "dense_vector",
                "dims": EMBEDDING_DIM,
                "index": True,
                "similarity": "cosine"
            },
            "text_vector":          {"type": "dense_vector", "dims": EMBEDDING_DIM, "index": True, "similarity": "cosine"},
            "question_text_vector": {"type": "dense_vector", "dims": EMBEDDING_DIM, "index": True, "similarity": "cosine"}
        }
    }
}

index_name = "course-questions-vectors"
es_client.indices.delete(index=index_name, ignore_unavailable=True)
es_client.indices.create(index=index_name, body=index_settings)

for doc in tqdm(documents):
    es_client.index(index=index_name, document=doc)

Three knobs in the dense_vector spec are worth understanding because they show up in every vector database, just under different names:

  • dims — must match the model's output (here 384). A mismatch is a silent rejection.
  • index: true — build an HNSW graph for ANN. Without it, every search is brute-force linear scan.
  • similarity — cosine, dot_product, or l2_norm. Must match how the model was trained; pick wrong and your scores are nonsense.
Vector indexing is mostly about book-keeping discipline — match dims, match similarity, version the model, freeze the ground truth. The math is identical across libraries; the bugs come from mismatches.
19 · Step 10 — KNN Search and the First Numbers

🎯 The Vector Search Function

Elasticsearch's KNN query takes the field, the query vector, top-k, a num_candidates parameter that controls HNSW exploration, and an optional metadata filter. It returns the same hit shape as text search — the rest of the harness does not care.

elastic_search_knn — generic over which vector field to search
def elastic_search_knn(field, vector, course):
    knn = {
        "field":          field,                # question_vector, text_vector, ...
        "query_vector":   vector,
        "k":              5,
        "num_candidates": 10000,                # HNSW recall knob — higher = better recall, slower
        "filter": {"term": {"course": course}}
    }
    search_query = {
        "knn": knn,
        "_source": ["text", "section", "question", "course", "id"]
    }
    response = es_client.search(index=index_name, body=search_query)
    return [hit["_source"] for hit in response["hits"]["hits"]]

num_candidates is the HNSW analogue of "how many neighbours to explore before returning the top-k". Industry-standard default is 10 × k for fast tier (latency-bound) and 100 × k or higher for the quality tier — same model, two indexes, route by SLA.

🔌 Plug Into the Existing Evaluator

The evaluator from §16 already takes the row dict; we just write a new lambda that encodes the question and forwards to KNN. Nothing else changes.

Evaluate question_vector retrieval
def question_vector_knn(q):
    vec = model.encode(q["question"]).tolist()
    return elastic_search_knn("question_vector", vec, q["course"])

evaluate(ground_truth, question_vector_knn)
# → {'hit_rate': 0.7728, 'mrr': 0.6592}

Already on par with minsearch — and we've used the simplest possible embedding strategy. The full benefit shows up when we switch fields.

🧊 Exact vs Approximate — Sanity Check

To know whether ANN is silently costing you recall, run the same query with script_score (brute-force exact KNN) on a sample of the ground truth. The gap between exact and ANN — recall@k loss — is the price you pay for HNSW's speed.

Exact KNN via script_score (slow but ground truth)
exact = {
    "size": 5,
    "query": {
        "script_score": {
            "query": {"term": {"course": course}},
            "script": {
                "source": "cosineSimilarity(params.query_vector, 'question_vector') + 1.0",
                "params": {"query_vector": vec}
            }
        }
    }
}

If exact KNN beats your ANN configuration by >1–2 points, increase num_candidates, raise HNSW m / ef_construction, or accept the trade-off explicitly. Just make sure you know what the trade-off is.

20 · Step 11 — Comparing Embedding Strategies & Hybrid Search

🪜 Three Strategies, Same Harness

Run evaluate three times — one per vector field — using the cached embeddings already stored in the index. The lambdas are identical except for the field name and which text is encoded into the query vector.

Sweep over embedding strategies
def vector_knn(field, encode_text):
    def _search(q):
        vec = model.encode(encode_text(q)).tolist()
        return elastic_search_knn(field, vec, q["course"])
    return _search

evaluate(ground_truth, vector_knn("question_vector",      lambda q: q["question"]))
evaluate(ground_truth, vector_knn("text_vector",          lambda q: q["question"]))
evaluate(ground_truth, vector_knn("question_text_vector", lambda q: q["question"]))

The numbers above are typical of what this benchmark reports with multi-qa-MiniLM-L6-cos-v1. Three lessons that generalise far beyond this dataset:

  • Embedding choice swamps retriever choice. The jump from question_vector to question_text_vector (+17 HR points) dwarfs the gap between Elasticsearch BM25 and minsearch TF-IDF (+3 HR points).
  • Concatenation usually wins for Q&A. Both sides of the (question, answer) pair contribute matchable signal. This is why module 2's RAG section embedded question + " " + text.
  • "Field engineering" is real. What you put into the embedder matters more than which embedder you use, up to a point.

🪢 Hybrid Search — BM25 + Dense Together

Vector search misses on exact tokens — model names, course codes, library identifiers. BM25 misses on paraphrases. Hybrid search runs both and combines them. Elasticsearch supports this directly via the knn + query compound search; rank fusion (RRF) is built into ES 8.9+.

Hybrid search — BM25 + KNN combined
def elastic_search_hybrid(field, query, vector, course):
    knn = {
        "field": field, "query_vector": vector,
        "k": 5, "num_candidates": 10000,
        "filter": {"term": {"course": course}},
        "boost": 0.5                                    # weight of vector signal
    }
    keyword = {
        "bool": {
            "must": {"multi_match": {
                "query":  query,
                "fields": ["question^3", "text", "section"],
                "type":   "best_fields",
                "boost":  0.5                              # weight of keyword signal
            }},
            "filter": {"term": {"course": course}}
        }
    }
    body = {"size": 5, "query": keyword, "knn": knn,
            "_source": ["text", "section", "question", "course", "id"]}
    response = es_client.search(index=index_name, body=body)
    return [hit["_source"] for hit in response["hits"]["hits"]]

The two boost values combine linearly into the final score. They become hyperparameters — sweep them with the same evaluator. For corpora with a healthy mix of natural-language and keyword-style queries, hybrid usually adds another 1–3 points on top of the best dense strategy.

For a less ad-hoc combination, use Reciprocal Rank Fusion (RRF) — score-free, robust, and what production systems converge to. Elasticsearch 8.9+ exposes it directly:

RRF in Elasticsearch (8.9+)
body = {
    "retriever": {
        "rrf": {
            "retrievers": [
                {"standard": {"query": keyword["bool"]}},
                {"knn":      knn}
            ],
            "rank_window_size": 50,
            "rank_constant":    60
        }
    },
    "size": 5
}

🔁 Reranking — The Final Layer

Once a dense retriever returns 50–100 strong candidates, a cross-encoder reranker (e.g. BAAI/bge-reranker-base, cross-encoder/ms-marco-MiniLM-L-6-v2) reads (query, doc) pairs jointly and re-scores them. Cross-encoders are far more accurate than bi-encoders but linear in candidates — perfect as a final stage on the retriever's top-50.

Plug a cross-encoder reranker into the harness
from sentence_transformers import CrossEncoder

reranker = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

def vector_then_rerank(q, top_n=50):
    vec        = model.encode(q["question"]).tolist()
    candidates = elastic_search_knn("question_text_vector", vec, q["course"])[:top_n]
    pairs      = [(q["question"], c["text"]) for c in candidates]
    scores     = reranker.predict(pairs)
    ranked     = [c for _, c in sorted(zip(scores, candidates), reverse=True)]
    return ranked[:5]

evaluate(ground_truth, vector_then_rerank)

For RAG production deployments at scale, the retrieve → fuse → rerank → top-k pattern is the de-facto standard. Each layer is independently swappable, independently measurable, and (with the harness from §16) independently tunable.

📈 What "Done" Looks Like

Vector Eval Checklist
  • Frozen, versioned ground truth
  • Same metrics across all runs (HR + MRR or HR@k + nDCG@k)
  • Embedding model and field swept independently
  • Exact KNN computed on a sample as quality ceiling
  • Hybrid (BM25 + dense, or RRF) measured
  • P50 / P95 latency tracked alongside quality
  • Results logged in MLflow / W&B / a CSV
Smells of an Unrigorous Eval
  • "This embedding model felt better" — no numbers
  • Different ground-truth set each comparison
  • Cherry-picked queries demoed in a slide
  • Quality reported, latency hidden
  • ANN tuned without an exact-KNN baseline
  • Hybrid weights eyeballed, not swept
Vector search evaluation is boring on purpose. The discipline of "freeze the data, fix the metrics, sweep one variable" is what separates a credible benchmark from a marketing graph. Boring beats persuasive — it survives audit.

22 · Key Takeaways

💡 What to Remember

  • The "best" retriever depends on data, query distribution, and downstream usage. Evaluation turns "it depends" into a number.
  • A retrieval system has a huge tunable surface — search type, fields, boosts, filters, top-k, embedding model, vector database, indexing strategy. Without ground truth, every choice is a guess.
  • A ground-truth dataset is rows of (query, relevant_doc_ids). Hundreds to thousands of rows make benchmarking meaningful.
  • Three sources of ground truth — user feedback, human annotators, LLM generation — with different cost / quality / scalability trade-offs.
  • Without production traffic, LLM-generated ground truth is a practical default: rephrase known FAQ questions into synthetic user queries.
  • Synthetic ground truth is noisy in absolute terms but consistent for relative comparisons — exactly what's needed to pick a retriever.
  • Search is a ranking problem. Rewarding "found" is not enough; reward "found early".
  • Hit Rate answers "did we find it?". MRR answers "did we find it first?". Together they cover the FAQ-style retrieval problem cleanly.
  • Other metrics — Precision@k, Recall@k, MAP, nDCG — are not wrong, they're tuned for different shapes of data (multi-target queries, graded relevance, …).
  • Evaluation makes retrieval decisions defensible: "MRR went from 0.61 to 0.74" survives scrutiny; "I think this is better" doesn't.
  • Always tag documents with stable IDs before generating ground truth. Source IDs > content hashes > sequential numbers — sequential IDs silently break every time the corpus is reordered.
  • Content-hash IDs (MD5 of course + question + first 10 chars of answer, truncated to 8 hex chars) are stable across re-indexing and free to compute.
  • Use defaultdict(list) + a length comparison to surface duplicate IDs immediately — collisions reveal either dirty source data or a too-narrow hash input.
  • The question-generation prompt needs three hidden levers: "complete and not too short" (no telegraphic fragments), "use as fewer words as possible from the record" (force paraphrasing), and "no code blocks" (clean JSON).
  • Make the generation loop resumable from cache — keyed by doc ID, with if doc_id in results: continue. Rate limits and crashes become non-events.
  • Persist raw LLM output (pickle or JSON) before parsing. Fix the 1–2 malformed responses by hand instead of paying to regenerate the whole batch.
  • The deliverable is a flat CSV: (question, course, document). Every retriever scores against the same column, every comparison is reproducible.
  • Index the corpus with an explicit id field (mapped as keyword in Elasticsearch, listed in keyword_fields for minsearch) so result documents carry the same identifiers the ground truth references.
  • The relevance loop produces a list[list[bool]] — one inner list per query, one bool per top-k result. Hit Rate ≡ Recall when each query has exactly one relevant document.
  • Sanity-check metric implementations with a tiny hand-built example first. Off-by-one errors hide trivially in a 4,000-row run.
  • On the FAQ benchmark, minsearch (TF-IDF) outscores Elasticsearch (BM25 multi_match) by ~3–6 points — "production-grade tool" is not the same as "best for this data". Always measure.
  • Refactor the evaluation loop into a generic evaluate(ground_truth, search_function). New retrievers become a one-line lambda; comparisons stay apples-to-apples.
  • Tuning blind hurts: dropping the section field and bumping question boost from 3 to 5 lowered both metrics. The evaluator catches regressions opinions can't.
  • For more than a handful of tuning experiments, graduate to experiment tracking (MLflow, etc.) — Jupyter is fine until it isn't.
  • The 20–25% of queries the retriever misses deserve a manual look. Often the LLM-generated query is nonsense, not the retriever's fault.
  • Vector evaluation re-uses the same harness — only the inner search_function changes. Modular design pays off here.
  • Field engineering beats model engineering on Q&A data. question_text_vector beat question_vector by ~17 HR points; the gap between any two reasonable models on the same field is usually smaller.
  • Match the model's training similarity in the index (cosine / dot_product / l2_norm). A mismatch silently corrupts every score.
  • Always evaluate exact KNN as a ceiling on a sample of queries — ANN parameters (num_candidates, HNSW m / ef_construction) trade recall for speed, and you should know the cost.
  • Hybrid (BM25 + dense, weighted or RRF-fused) typically adds 1–3 points on top of the best dense strategy, especially on corpora with mixed query styles.
  • For production, the retrieve → fuse → rerank stack with a cross-encoder reranker on the top-50 is the de-facto standard. Each layer is independently tunable.
  • Track P50 / P95 latency alongside quality — a 2-point MRR gain that doubles tail latency is rarely a win.
  • Freeze and version your ground truth (ground-truth-data-v1.csv). Regenerating it between experiments invalidates every comparison you've made.
  • Cache embeddings on disk; encode the corpus once. At any non-trivial scale, encoding is the cost bottleneck.
  • Industry-standard rigor is just discipline: frozen data, fixed metrics, one variable per run, latency tracked, results logged.
