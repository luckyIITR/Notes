Build Your Own Search Engine

An end-to-end walkthrough of the retrieval step in RAG — building a toy in-memory search engine from bag-of-words to BERT embeddings, using only scikit-learn and pandas.

1 · Why Search Matters in RAG

🧭 Retrieval Is the "R" in RAG

Retrieval-Augmented Generation lets a language model answer questions using your data. The pattern is simple but every step matters:

1User Query"Is it too late to join?"
2SearchTop-k relevant docs
3PromptQuery + context
4LLMe.g. GPT-3.5
5AnswerGrounded reply

Search is the foundation of the whole system — if retrieval gives the LLM the wrong documents, no amount of prompt engineering can save the answer. Understanding how relevance scoring works is therefore essential before reaching for Elasticsearch, FAISS, or any vector database.

The retrieval step controls what the model can know. Bad retrieval = confidently wrong answers. Good retrieval = grounded, useful responses.

📚 The Working Dataset

Throughout these notes we use FAQ documents from the DataTalksClub Zoomcamp courses (Data Engineering, ML, MLOps). These are real ~300-page Google Docs students search through before asking on Slack. Each entry has four fields:

  • course — which course the FAQ belongs to (used for filtering).
  • section — top-level grouping (e.g. "Module 1 — Docker").
  • question — the FAQ question.
  • text — the answer body.
Loading the data
import requests, pandas as pd

docs_url = 'https://github.com/alexeygrigorev/llm-rag-workshop/raw/main/notebooks/documents.json'
documents_raw = requests.get(docs_url).json()

documents = []
for course in documents_raw:
    for doc in course['documents']:
        doc['course'] = course['course']
        documents.append(doc)

df = pd.DataFrame(documents, columns=['course', 'section', 'question', 'text'])
2 · Two Flavours of Search

🔀 Text Search vs Semantic / Vector Search

There are two fundamentally different ways to match a query to documents. Both have the same goal — return relevant results — but they look at the text very differently.

Text Search

Matches exact words. If the query and document share the same tokens (after lowercasing and stop-word removal), they match.

Query: "I just discovered the course, can I still join?"

Matches: docs containing discovered, course, join.

Semantic / Vector Search

Matches meaning. Encodes query and documents into dense vectors so synonyms and paraphrases land near each other.

Query: "I just found out about the program, can I still enroll?"

Still matches: docs about discovering and joining the course.

Text search fails on synonyms. Vector search fails on rare exact terms (product codes, acronyms). Real systems combine both — that is what production "hybrid search" means.
3 · Vector Spaces & Bag of Words

📐 Turning Documents Into Vectors

The trick that powers every text-search algorithm — from 1970s SMART to modern Lucene — is the same: represent each document as a point in a high-dimensional vector space. Then "relevance" becomes geometric closeness.

The simplest such representation is the term-document matrix: rows are documents, columns are vocabulary words, and each cell counts how many times a word appears in a document.

Consider five toy documents:

  1. "Course starts on 15th Jan 2024"
  2. "Prerequisites listed on GitHub"
  3. "Submit homeworks after start date"
  4. "Registration not required for participation"
  5. "Setup Google Cloud and Python before course"

🎒 The "Bag-of-Words" Assumption

This representation is called bag of words because it throws away word order — only the presence and frequency of each token matter. Two documents with the same words in different order produce identical vectors.

  • Lossy — "dog bites man" and "man bites dog" are the same vector.
  • Sparse — most documents contain only a tiny fraction of the vocabulary, so most cells are zero. Storing it as a sparse matrix (only non-zero entries) is essential.
  • Surprisingly effective — for a huge class of retrieval tasks, word-order-free matching is enough.
CountVectorizer — the simplest bag of words
from sklearn.feature_extraction.text import CountVectorizer

cv = CountVectorizer(stop_words='english')
X = cv.fit_transform(docs_example)

names = cv.get_feature_names_out()        # vocabulary
df_docs = pd.DataFrame(X.toarray(), columns=names)

Two arguments worth knowing:

  • stop_words='english' — drops "the", "is", "and", … which carry no retrieval signal.
  • min_df=5 — only keep terms that appear in at least 5 documents. Without it, a single typo or rare token gets its own column. The full FAQ has ~6,700 unique tokens; setting min_df=5 cuts this to a much more meaningful vocabulary.
4 · TF-IDF — Weighting What Matters

⚖️ Not All Words Are Equal

Raw counts treat every word the same: matching course contributes as much as matching yaml. But intuitively, yaml is a much stronger signal — it appears in only a few documents, while course appears almost everywhere.

TF-IDF (Term Frequency × Inverse Document Frequency) formalises this: rare terms get higher weights, common terms get lower weights.

TF-IDF Weight
tfidf(t, d) = tf(t, d) · log( N / df(t) )
tf(t, d) — number of times term t appears in document d
df(t) — number of documents that contain term t
N — total number of documents in the corpus
A term in many documents (large df) → small log term → small weight.

In scikit-learn, swapping CountVectorizer for TfidfVectorizer is the only change needed — the rest of the pipeline stays identical.

Bag of words → TF-IDF (one-line swap)
from sklearn.feature_extraction.text import TfidfVectorizer

cv = TfidfVectorizer(stop_words='english', min_df=3)
X = cv.fit_transform(df['text'])

After fitting, cells contain floats like 0.21 or 0.05 instead of integer counts — these are the term weights. TfidfVectorizer also L2-normalises each row by default, which is what makes the dot-product equal to cosine similarity (next section).

TF-IDF gives you importance-aware matching for free. It is the strongest no-ML baseline in information retrieval and the foundation that BM25 and Lucene refine.
5 · Scoring — Dot Product & Cosine Similarity

📏 Measuring How Close Query and Document Are

Once the query is in the same vector space as the documents (using the same fitted vectorizer), comparing them is straight linear algebra: multiply the corresponding weights and sum. Words that appear in both contribute; words missing from either contribute zero.

Dot product as relevance score
score(q, d) = Σt   qt · dt
qt — TF-IDF weight of term t in the query
dt — TF-IDF weight of term t in the document
The sum runs over every term in the vocabulary; non-matching terms add zero.

Doing this for all documents in one shot is just a matrix–vector multiplication — and sparse matrices make it blazingly fast.

Score every document against the query
query = "I just signed up. Is it too late to join the course?"

q = cv.transform([query])           # same vectorizer as for X
scores = X.dot(q.T).toarray().flatten()

📐 Cosine Similarity

Cosine similarity is the dot product after both vectors are normalised to unit length — it measures the angle between vectors, ignoring magnitude. Long documents do not get an unfair boost just for being long.

Cosine similarity
cos(q, d) = ( q · d ) / ( ‖q‖ · ‖d‖ )
With TfidfVectorizer's default L2 normalisation, ‖q‖ = ‖d‖ = 1, so cosine similarity equals the plain dot product.
scikit-learn one-liner
from sklearn.metrics.pairwise import cosine_similarity

scores = cosine_similarity(X, q).flatten()

🏆 Ranking the Top-k

Once you have a score per document, sort descending and take the top k. np.argsort returns indices, not values — perfect for indexing back into the dataframe.

Top-5 results
import numpy as np

idx = np.argsort(-scores)[:5]      # negate to sort descending
top_docs = df.iloc[idx]

For very large corpora, np.argpartition is asymptotically faster — it finds the top-k without fully sorting the rest.

6 · Multi-Field Search, Boosting & Filtering

🎚️ Searching Across Multiple Fields

Real documents have structure — a question field, a text answer body, a section heading. A match in the question field is much stronger evidence of relevance than the same word buried in a long answer.

The recipe:

  1. Fit a separate TfidfVectorizer on each field — each gets its own vocabulary and its own term-document matrix.
  2. For each query, transform it through every field's vectorizer and compute cosine similarity per field.
  3. Sum the per-field scores, multiplying by a boost that gives some fields more weight than others.
  4. Apply hard filters (e.g. course == "data-engineering-zoomcamp") by zeroing out scores of documents that fail the filter.
  5. Sort and return the top-k.
Per-field TF-IDF index
fields       = ['section', 'question', 'text']
transformers = {}
matrices     = {}

for f in fields:
    cv = TfidfVectorizer(stop_words='english', min_df=3)
    matrices[f]     = cv.fit_transform(df[f])
    transformers[f] = cv
Combined score with boosting + filtering
boost   = {'question': 3.0}                # question matches count 3×
filters = {'course': 'data-engineering-zoomcamp'}

score = np.zeros(len(df))

for f in fields:
    b = boost.get(f, 1.0)
    q = transformers[f].transform([query])
    s = cosine_similarity(matrices[f], q).flatten()
    score = score + b * s

for field, value in filters.items():
    mask  = (df[field] == value).values             # bool array
    score = score * mask                            # zero out non-matches

idx     = np.argsort(-score)[:5]
results = df.iloc[idx].to_dict(orient='records')
This is exactly how Elasticsearch's multi_match query with boosts works under the hood. You now have the same retrieval primitive in ~25 lines of Python.
7 · Wrapping It Up — The TextSearch Class

🧱 A Reusable Search Index

Bundling the steps into a class gives an Elasticsearch-style API: fit once on a list of records, then call search repeatedly with different queries, boosts, and filters.

minsearch — toy in-memory index
class TextSearch:
    def __init__(self, text_fields):
        self.text_fields = text_fields
        self.matrices    = {}
        self.vectorizers = {}

    def fit(self, records, vectorizer_params={}):
        self.df = pd.DataFrame(records)
        for f in self.text_fields:
            cv = TfidfVectorizer(**vectorizer_params)
            self.matrices[f]    = cv.fit_transform(self.df[f])
            self.vectorizers[f] = cv

    def search(self, query, n_results=10, boost={}, filters={}):
        score = np.zeros(len(self.df))
        for f in self.text_fields:
            b = boost.get(f, 1.0)
            q = self.vectorizers[f].transform([query])
            s = cosine_similarity(self.matrices[f], q).flatten()
            score = score + b * s
        for field, value in filters.items():
            score = score * (self.df[field] == value).values
        idx = np.argsort(-score)[:n_results]
        return self.df.iloc[idx].to_dict(orient='records')
Usage — Elasticsearch-style
index = TextSearch(text_fields=['section', 'question', 'text'])
index.fit(documents)

results = index.search(
    query='I just signed up. Is it too late to join the course?',
    n_results=5,
    boost={'question': 3.0},
    filters={'course': 'data-engineering-zoomcamp'},
)

This exact class lives in the minsearch library and is the retriever used in subsequent RAG modules.

What This Buys You
  • Zero infrastructure — pure Python, no server
  • Field-aware scoring with boosting and filtering
  • Fast enough for thousands of documents
  • Great for prototyping a RAG pipeline
What It's Not
  • Not for production — in-memory only
  • No incremental indexing of new documents
  • No fuzzy matching, stemming, or BM25
  • Crashes? You lose the index
8 · Vector Search — The Synonym Problem

🧠 When Exact Matching Fails

Text search depends on shared tokens. But users rarely use the exact words from the document:

Same meaning, zero matching tokens
Document   "I just discovered the course, can I still join?"
Query      "I just found out about the program, can I enroll?"
Every important content word is different. TF-IDF score ≈ 0. We need to compare meaning, not surface words.

The answer is embeddings: dense numerical vectors where similar items end up close, regardless of the exact words used.

  • Conversion to numbers — words, sentences, documents become arrays of floats.
  • Similarity preserved — synonyms ("enroll", "join", "sign up") land in nearby regions of the vector space.
  • Dimensionality reduced — instead of a 6,700-column sparse matrix, embeddings are typically 16 to 768 dense floats per document.
  • Reusable — the same embeddings feed downstream ML models, clustering, recommendation, classification.
9 · LSA via SVD

🗜️ Compressing the Term-Document Matrix

Latent Semantic Analysis (Deerwester et al., 1990) is the simplest way to turn a bag-of-words matrix into embeddings. It applies Singular Value Decomposition to "compress" the matrix while keeping as much information as possible.

Singular Value Decomposition
X  ≈  U · Σ · VT
X — original sparse term-document matrix (e.g. 948 × 6700)
U · Σ — dense, low-rank embedding for each document (e.g. 948 × 16)
Truncating to k components is a lossy compression — exact reconstruction is impossible, but the most informative directions are kept.

The intuition is the same as SVD on images: keep the top k singular values and you preserve the broad structure (synonyms grouped together as "topics") while throwing away noise.

Truncated SVD on the TF-IDF matrix
from sklearn.decomposition import TruncatedSVD

X  = matrices['text']
cv = transformers['text']

svd   = TruncatedSVD(n_components=16)
X_emb = svd.fit_transform(X)        # (n_docs, 16) dense

Q     = cv.transform([query])
Q_emb = svd.transform(Q)            # same projection

scores = cosine_similarity(X_emb, Q_emb).flatten()
idx    = np.argsort(-scores)[:10]

The same query that scored zero with text search now retrieves relevant documents — synonyms like "enroll" and "join" get folded into the same latent dimensions because they co-occur with the same neighbouring words.

SVD/LSA is almost as old as the internet (1990) yet still produces remarkably competitive embeddings for small corpora. The technique is timeless; only the names ("LSA", "Latent Semantic Indexing", "Truncated SVD") change.
10 · NMF — Interpretable Topics

🎨 Non-Negative Matrix Factorization

SVD is powerful but spits out negative components. What does it mean for a document to belong to a topic with score -0.08? Hard to say.

NMF solves this: if the input matrix is non-negative (and TF-IDF always is), the output factors are non-negative too. Each column of the embedding can be read as "how much this document is about topic k", with zero meaning "not at all".

NMF in place of SVD
from sklearn.decomposition import NMF

nmf   = NMF(n_components=16)
X_emb = nmf.fit_transform(X)
Q_emb = nmf.transform(cv.transform([query]))

scores = cosine_similarity(X_emb, Q_emb).flatten()

NMF embeddings are typically sparse — each document scores high on only a handful of topics. This makes them an entry-level form of topic modelling: you can interpret what each dimension represents by looking at the documents that score highest in it.

SVD vs NMF — When to Use Which
  • SVD: maximum information retention, fastest, classic LSA
  • NMF: interpretable, sparse, topic-modelling-friendly
Shared Limitations
  • Both built on bag-of-words → still no word order
  • Negations ("not deprecated") still ignored
  • Won't handle words never seen at fit time
11 · BERT Embeddings

🤖 Contextual Embeddings That Know Word Order

BERT (Bidirectional Encoder Representations from Transformers) is a neural network pre-trained on huge amounts of text. Unlike bag-of-words methods, BERT does see word order, handles negation, and produces strong embeddings even for words it has never seen in the FAQ corpus.

The HuggingFace transformers library makes this nearly trivial:

Install
pip install transformers tqdm torch
Loading a pre-trained BERT model
import torch
from transformers import BertModel, BertTokenizer

tokenizer = BertTokenizer.from_pretrained("bert-base-uncased")
model     = BertModel.from_pretrained("bert-base-uncased")
model.eval()    # inference mode (no dropout / gradients)

🔢 From Text to Vector — Two Steps

  1. Tokenize — split text into sub-word IDs the model understands. padding=True aligns sequences in a batch; truncation=True caps overly long inputs.
  2. Forward pass — feed token IDs through the model, take the last hidden states (one vector per token), then average across tokens to get a single sentence embedding.
Compute a sentence embedding
texts = [
    "Yes, we keep all materials after the course finishes.",
    "You can follow the course at your own pace after it finishes.",
]
encoded = tokenizer(texts, padding=True, truncation=True, return_tensors='pt')

with torch.no_grad():                       # inference, no autograd
    out      = model(**encoded)
    hidden   = out.last_hidden_state          # (batch, tokens, 768)
    sent_emb = hidden.mean(dim=1)            # mean-pool → (batch, 768)

X_emb = sent_emb.cpu().numpy()              # move off GPU if needed

📦 Batching for the Whole Corpus

Encoding ~1,000 documents one-by-one is slow; batching 8 at a time and using a progress bar is the standard recipe.

Batched corpus embedding
from tqdm.auto import tqdm

def make_batches(seq, n):
    return [seq[i:i+n] for i in range(0, len(seq), n)]

def compute_embeddings(texts, batch_size=8):
    all_emb = []
    for batch in tqdm(make_batches(texts, batch_size)):
        enc = tokenizer(batch, padding=True, truncation=True, return_tensors='pt')
        with torch.no_grad():
            h = model(**enc).last_hidden_state
            all_emb.append(h.mean(dim=1).cpu().numpy())
    return np.vstack(all_emb)

X_text = compute_embeddings(df['text'].tolist())
For a small FAQ, BERT is overkill — TF-IDF + SVD usually wins on speed and is competitive on quality. BERT shines on longer, more complex text where word order, negation, and out-of-vocabulary terms matter.
12 · Going to Production

🏭 Why You Can't Use This At Google Scale

Computing cosine similarity against every document works for 1,000 FAQ entries. It does not work for billions of web pages. Two engineering ideas make web-scale search possible:

📇 Inverted Index & Posting Lists (for text search)

Instead of "for each document, list its terms" (the term-document matrix), you store "for each term, list the documents that contain it" — the inverted index. Each list is a posting list:

Inverted index entry
"yaml"  →  [doc_42, doc_117, doc_503, …]
A query like "yaml docker" intersects (or unions) the posting lists for yaml and docker — only those documents are scored, not the entire corpus. This is the foundation of Lucene, Elasticsearch, OpenSearch, Solr.

🎲 LSH & Random Projections (for vector search)

Locality-Sensitive Hashing turns dense embeddings into short bit-vectors such that similar embeddings produce similar bit-strings. The simplest variant is random projection:

  1. Pick b random hyperplanes in embedding space.
  2. For each document, output 1 if it falls on the positive side of a hyperplane, 0 otherwise — that's a b-bit hash.
  3. Group documents into "bins" by their hash. At query time, only compare against documents in the same (or nearby) bin — usually a tiny fraction of the corpus.
  4. Cosine similarity between dense vectors is well-approximated by Hamming similarity between their hashes.

Real vector databases (FAISS, Milvus, Pinecone, Qdrant, Weaviate) use refinements of this idea — IVF, HNSW graphs, product quantization — but the core insight is the same: do not scan everything.

For RAG prototypes, the toy in-memory index is fine. For anything beyond ~100k documents, reach for Elasticsearch (text) or FAISS / a managed vector DB (embeddings).
13 · Evaluating Search Quality

📊 Picking Hyperparameters Without Guesswork

Many knobs need tuning: number of SVD components, vectorizer's min_df, boost weights, choice of embedder. The only honest way to choose is to measure retrieval quality on a labelled set of (query, relevant doc) pairs.

MetricWhat it measures
Precision@kFraction of the top-k results that are relevant. Easy to interpret, ignores rank order within the top-k.
Recall@kFraction of all relevant documents that show up in the top-k. Cares about coverage.
Mean Average Precision (MAP)Averages precision at every rank where a relevant document appears. Penalises burying good results.
nDCG@kNormalised Discounted Cumulative Gain. Rewards relevant results appearing earlier; the de-facto standard for ranking quality.
If a heavy BERT setup gives only marginal gains over TF-IDF + SVD on your nDCG, the simpler model wins. Measure first, optimise second.
14 · Key Takeaways

💡 The Big Picture

  • Every relevance-search algorithm — old or new — represents queries and documents in the same vector space and ranks by similarity.
  • TF-IDF + cosine similarity is a remarkably strong baseline. Don't skip it for a fancier method until you have measured.
  • Multi-field search needs per-field vectorizers, additive boosting, and post-hoc filtering — together they reproduce 90% of Elasticsearch's relevance behaviour.
  • Bag-of-words methods (CountVectorizer, TF-IDF, LSA, NMF) throw away word order. For most FAQ-style tasks that's fine; for nuanced text, BERT-style embeddings help.
  • SVD/LSA, NMF, and BERT differ mainly in how they produce embeddings — once you have vectors, the cosine-similarity ranking step is identical.
  • In production, the cost is dominated by not scanning the whole corpus: inverted indexes for text, LSH/HNSW for vectors.
  • Hybrid search (text + vector) is what most real-world RAG systems use — exact keyword matches plus semantic similarity, the best of both worlds.
