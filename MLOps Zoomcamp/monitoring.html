MLOps Zoomcamp · Module 5

Model Monitoring

Watching machine learning models in production — what metrics to track, batch vs online serving, and a hands-on stack with Evidently, Prefect, PostgreSQL, and Grafana.

1 · Why Monitor Models?

📉 Models Degrade Over Time

Building a model is hard; running a production service around that model is harder. When you plot model quality over time, you often see the same shape — performance is fine at launch, then slowly degrades as the world changes around it.

Production software has had a solution for ages: monitoring. We reuse that idea for ML, but a model needs extra layers beyond ordinary service health, because we also have to account for data and the model itself.

Monitoring is the answer to model decay. If you can implement only a few metrics, start with service health — first make sure the service even works.
2 · The Four Core Metric Groups

📊 A Solid Starting Point

For an ML-based service, four groups of metrics cover most of what you need. They build outward from plain software up to model- and data-specific concerns.

Group 1 · Must-have

Service health

Is the service even up? Track uptime, memory usage, and latency — the same metrics you'd use for any production service.

Group 2

Model performance

How good are the predictions? The exact metrics depend on the problem statement.

Group 3

Data quality & integrity

When the model misbehaves it's usually the input data. Catch issues early with data checks.

Group 4

Data & concept drift

Has the world changed? Compare current distributions against a reference where the model worked well.

Service health, model performance, data quality & integrity, and drift are a very good starting point — but never limit yourself to just these four. Scope depends on case sensitivity, risk, and resources.
3 · Model Performance Metrics

🎯 Pick Metrics by Problem Type

The most straightforward way to ask "how good is the model" is its standard performance metrics — well-known and easy to interpret. The right set depends entirely on the problem statement:

Problem typeExample metricsNotes
Ranking (search, recommenders)Ranking metrics (NDCG, MAP, etc.)Content recommendation, SEO-style ordering
RegressionMAE, MAPEMean absolute / mean absolute percentage error
ClassificationLog loss, precision, recallLog loss especially for probabilistic classifiers
The taxi project is a regression problem (predict trip duration in minutes), so MAE and MAPE are the natural performance metrics.
4 · The Ground-Truth Delay

⏳ When You Can't Compute Error Yet

Very often the ground truth isn't available right when the model makes a prediction — there's a delay between generating output and learning the true label. During that gap you simply cannot compute error-based quality metrics.

Use a proxy instead. Since problems usually trace back to the input data, data quality and integrity metrics let you catch and fix most issues even without labels:

  • Share or count of missing values
  • Type checks — does a column match the type it's supposed to have?
  • Value range checks per column

A lot of subtle bugs get caught this way before they ever reach the model.

5 · Data Drift & Concept Drift

🌊 The World Keeps Changing

Even with clean data you can still have problems, because a model never works in a vacuum — it interacts with a real-world environment, and environments change. Two related early-warning signals:

  • Data drift — the distribution of input data shifts away from the reference.
  • Concept drift — the relationship between inputs and the target changes.

Both work by comparing current distributions (inputs, model output, or target) against a reference dataset — the data from when you were happy with the model's performance.

Drift doesn't guarantee something is wrong — but it's a very nice early signal of potential problems worth investigating.
6 · Going Beyond the Four Groups

🧩 Extra Metrics for Higher-Risk Cases

Depending on your domain and risk tolerance, you may want custom metrics that cover specific business risks:

  • Quality by segment — for wide/diverse audiences, split objects by category and track quality per category.
  • Bias & fairness — for sensitive domains like healthcare or finance.
  • Outlier detection — when each individual error is costly, flag outliers for manual review to reduce risk.
  • Explanations — for recommenders that retrain frequently, monitor the explanations you share with users to build trust.

You can invent many custom metrics; the goal is to make sure you at least understand each risky aspect of your data pipeline and model.

7 · Reusing Existing Monitoring

♻️ Start From What You Already Have

You're lucky if production services already exist in your company — they probably come with monitoring you can reuse for ML:

  • Prometheus + Grafana — Prometheus pulls metrics from your service; Grafana uses it as a data source, visualizes dashboards, and fires alerts. Reuse this scheme for models.
  • BI tools (Tableau, Looker) — if you only have batch pipelines, analyze each batch of predictions in a few dashboards. It's a low-effort way to reduce risk before automating.
Always begin with something the company already has, then improve monitoring quality through iterations.
8 · Batch vs Online Serving

⚖️ Deployment Shape Affects Monitoring

Many metrics are inherently batch calculations — drift detection needs two distributions to compare, and quality metrics like precision/recall are computed over a set of rows, not a single object.

MetricBatch modelOnline / REST API model
Missing values, range violationsPer batchCan compute in real time, per request
Data drift, model performanceReference vs latest batchBetter to accumulate a small batch first

For online models, use window functions: choose a window size and step size, wait until you collect a small batch, compute the batch metrics, and store them.

Even with an online service, you can — and usually should — do data and ML monitoring in batch. Reference data is typically the validation set or a previous batch you were happy with.

10 · Setting Up the Environment

🐳 Conda, Requirements & Docker Compose

Create a working directory (e.g. taxi_monitoring), a Python 3.11 conda env, and a requirements.txt covering the whole stack:

# requirements.txt
prefect
tqdm
requests
joblib
psycopg
psycopg_binary
evidently
pandas
numpy
scikit-learn
  1. Create & activate the env: conda create -n py11 python=3.11 then conda activate py11.
  2. Install everything: pip install -r requirements.txt.
  3. Confirm Docker is installed and running, then describe all services in a docker-compose.yml.

Docker Compose builds a multi-container app. The stack uses three services — a Postgres database, an Adminer DB UI, and Grafana — plus a Grafana volume (so dashboards/data sources survive a down/up) and two networks (back_tier for service-to-service, front_tier for browser access).

# docker-compose.yml
version: '3.7'

volumes:
  grafana_data: {}

networks:
  front_tier:
  back_tier:

services:
  db:
    image: postgres
    restart: always
    environment:
      POSTGRES_PASSWORD: example
    ports:
      - "5432:5432"
    networks:
      - back_tier

  adminer:
    image: adminer
    restart: always
    ports:
      - "8080:8080"
    networks:
      - back_tier
      - front_tier

  grafana:
    image: grafana/grafana
    user: "472"
    ports:
      - "3000:3000"
    volumes:
      - grafana_data:/var/lib/grafana
      - ./config/grafana_datasources.yaml:/etc/grafana/provisioning/datasources/datasource.yaml:ro
    networks:
      - back_tier
      - front_tier
    restart: always

Grafana needs to know where its data lives. Create a config/ folder with grafana_datasources.yaml pointing at Postgres:

# config/grafana_datasources.yaml
apiVersion: 1

datasources:
  - name: PostgreSQL
    type: postgres
    access: proxy
    url: db.:5432
    database: test
    user: postgres
    secureJsonData:
      password: 'example'
    jsonData:
      sslmode: 'disable'
Bring everything up with docker-compose up --build. Grafana lives at localhost:3000 (default login admin / admin), Adminer at localhost:8080. The data source url/port must match the Postgres service in compose.

🧪 Baseline Model + Reference Data

We work with NYC green taxi data and predict trip duration in minutes. Two folders hold the outputs: models/ and data/. In a notebook, download January + February 2022 data, train a baseline, and save a reference set for drift.

Key choices in the walkthrough:

  • Target = (drop_off − pickup) / 60 minutes; filter to trips of 1–60 min and 1–6 passengers to drop outliers.
  • Features — a handful of numeric (passenger count, trip distance, fare amount, total amount) plus categorical location IDs.
  • Split by time — first chunk = train, rest = validation (the data is ordered by datetime, so this respects the time series).
  • Model — a plain LinearRegression; no heavy experimentation, since the goal is just a baseline. MAE ≈ 4 minutes is good enough.
  • Dump the model with joblib and save the validation set as reference.parquet — that becomes the drift reference.
The validation set doubles as the reference dataset: it was held out from training and is the data where we judged the model "good enough." February data is saved for simulating production later.
12 · The Evidently Report

📋 Evidently as the Evaluation Layer

Evidently computes the metrics so you write no metric code yourself. Three pieces to import:

  • ColumnMapping — tells Evidently how to parse the frame (which column is the target, the prediction, which are numeric vs categorical). Important when categoricals are encoded as digits.
  • Report — a container of metric results that can render as HTML, JSON, or a Python dict.
  • Metrics — e.g. ColumnDriftMetric, DatasetDriftMetric, DatasetMissingValuesMetric.
from evidently import ColumnMapping
from evidently.report import Report
from evidently.metrics import (
    ColumnDriftMetric,
    DatasetDriftMetric,
    DatasetMissingValuesMetric,
)

column_mapping = ColumnMapping(
    target=None,                 # not analysing the target here
    prediction='prediction',
    numerical_features=num_features,
    categorical_features=cat_features,
)

report = Report(metrics=[
    ColumnDriftMetric(column_name='prediction'),
    DatasetDriftMetric(),
    DatasetMissingValuesMetric(),
])

report.run(reference_data=ref, current_data=cur, column_mapping=column_mapping)
report.show(mode='inline')       # HTML view inside the notebook

HTML is great for human analysis. For automation, render to a Python dict and pull out exactly the values you'll log to the database:

result = report.as_dict()

prediction_drift = result['metrics'][0]['result']['drift_score']
num_drifted_cols = result['metrics'][1]['result']['number_of_drifted_columns']
share_missing    = result['metrics'][2]['result']['current']['share_of_missing_values']
Metrics live in a list, not a dict, so you index by position ([0], [1], …). That's because Evidently lets you add the same metric with different parameters, which would make name-based lookup ambiguous.
13 · Evidently Monitoring Dashboards

🖥️ A Quick Built-In Dashboard

Before wiring up Grafana, Evidently offers its own dashboards — ideal for batch models when you have no monitoring infra yet. The recipe: compute reports on a schedule, store them in a workspace (a local folder), and the UI builds panels for you.

  1. Create a Workspace and a Project (give it a name + description, then project.save()).
  2. Build a report (e.g. with DataQualityPreset), setting an explicit timestamp per report so each maps to a specific date.
  3. Add it: workspace.add_report(project.id, report).
  4. Add dashboard panels — DashboardPanelCounter and DashboardPanelPlot (line, bar, scatter) — then project.save().
  5. Launch the UI from the project directory: evidently ui.

Each report is one data point. Add reports for several dates (e.g. Jan 28 and Jan 29) and the trend lines come alive. With a workflow that produces reports on a schedule (daily, weekly, every 10 minutes), you get a nice monitoring dashboard almost for free.

14 · Dummy Metrics → Postgres → Grafana

🧱 Prove the Pipe End-to-End First

Before real metrics, write a script that creates the DB + table and inserts random rows — this verifies Postgres, Adminer, and Grafana are wired correctly.

  • prep_db() — connect with psycopg; SELECT 1 FROM pg_database WHERE datname='test'; create the DB if absent, then create the table.
  • Table — dummy_metrics with a timestamp plus value1/2/3 (drop if exists so each run starts clean).
  • calculate_dummy_metrics_postgresql(curr) — generate random values and INSERT them with a timezone-aware timestamp.
  • main() — call prep_db(), then loop ~100 iterations, sending a row every ~10 s (a SEND_TIMEOUT gap so plots look nice).
Check the rows in Adminer (server db, user postgres, password example, database test), then build a Grafana panel: pick the PostgreSQL source, select a value column, and narrow the time range (e.g. last 5 minutes) to see the data flow.
15 · Real Metrics & the Prefect Pipeline

⚙️ Swap Dummy Values for Evidently Metrics

Now replace the dummy columns with real ones — prediction_drift, num_drifted_columns, share_missing_values. The prep_db() structure stays; only the metric calculation changes.

  • Load reference.parquet and the joblib model once as globals.
  • Read the whole February dataset as raw_data; in the loop, slice it day by day to simulate daily batches.
  • Each iteration i (day): select rows in [begin + i, begin + i + 1), fill NaNs, apply the model to get prediction, then run the report and derive the three values.
  • INSERT the metrics keyed by the simulated date.

The functions convert cleanly into a Prefect pipeline with decorators:

from prefect import task, flow

@task
def prep_db(): ...

@task
def calculate_metrics_postgresql(curr, i): ...

@flow
def batch_monitoring_backfill():
    prep_db()
    ...   # loop over days, insert metrics

if __name__ == '__main__':
    batch_monitoring_backfill()
Because rows use the simulated (historical) timestamps, set the Grafana panel range wide (e.g. last 2 years) to see them. Add panels for prediction drift, number of drifted columns, and missing values.
16 · Saving the Grafana Dashboard

💾 Make Dashboards Survive a Restart

To reuse a dashboard after a container restart (without rebuilding), provision it from config files instead of clicking it together each time:

  1. Add config/grafana_dashboards.yaml — a standard dashboard provider config pointing at a dashboards path.
  2. Create dashboards/data_drift.json — Grafana stores dashboards as JSON.
  3. Mount both into Grafana via the volumes section of docker-compose.yml (alongside the existing data-source mount).
  4. In Grafana, open Dashboard settings → JSON Model, copy it, and paste into data_drift.json.

Now docker-compose down then up (without --build) brings the same dashboard back automatically — no manual panel rebuilding.

17 · Debugging with Evidently

🔍 Test Suites & Reports for Root Cause

The point of monitoring is to detect when something breaks and then fix it. Suppose a Grafana threshold on prediction drift is crossed on Feb 2nd — open a notebook, load that day's slice, and dig in with two Evidently tools.

Test Suites
  • Check whether each metric meets a condition / threshold
  • Use presets like DataDriftTestPreset
  • Quickly see which tests passed vs failed
Reports
  • Maximize information rather than pass/fail
  • Use presets like DataDriftPreset
  • Inspect per-column distributions & trends
from evidently.test_suite import TestSuite
from evidently.test_preset import DataDriftTestPreset
from evidently.metric_preset import DataDriftPreset

# remember to apply the model + fill NaNs on the current slice first,
# or Evidently errors: "prediction column is partially present"

suite = TestSuite(tests=[DataDriftTestPreset()])
suite.run(reference_data=ref, current_data=cur, column_mapping=column_mapping)
suite.show(mode='inline')

report = Report(metrics=[DataDriftPreset()])
report.run(reference_data=ref, current_data=cur, column_mapping=column_mapping)
report.show(mode='inline')

The test suite summary shows e.g. 8 tests, 1 passed, 7 failed — drill into "share of drifted columns" to see which columns shifted (the prediction column shows clear distribution shift; passenger count looks stable, as expected). The report's Data Drift tab adds value-over-time plots, helping you spot trends (e.g. a rising fare amount) and form a hypothesis.

All formats (HTML, JSON, dict) are available. Use save_html() to share a standalone report with non-Python colleagues; use the dict/JSON form to drive downstream pipeline tasks based on which tests failed.
