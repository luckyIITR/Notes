Experiment Tracking with MLflow

What experiment tracking is and why it matters, plus a deep tour of MLflow — logging runs, hyperparameter tuning, model management, the model registry, and how to run MLflow in real teams.

1 · Key Vocabulary

📚 The Terms We Keep Using

Before anything else, a few words that come up constantly. Note that experiment here means offline ML experiments — building a model — not A/B testing.

  • ML experiment — the whole process of building a model (e.g. playing with different models and hyperparameters for one problem).
  • Experiment run — a single trial inside an experiment. This is exactly how MLflow models the problem: one experiment is a bunch of runs.
  • Run artifact — any file associated with a run (a plot, a pickled model, a dataset…).
  • Experiment metadata — all the information related to a run: source code, version, environment, and so on.
2 · What Is Experiment Tracking?

🔍 Keeping Track of What Matters

Experiment tracking is the process of keeping track of all the relevant information from an ML experiment — and "relevant" depends entirely on what you're doing. If you're varying the data, you track the data source, version, and preprocessing. If you're tuning hyperparameters, you track those values.

Because it's hard to pin down the exact set of things to record, the common approach is to capture a lot of standard items — source code, version, environment — and then customize per experiment.

Experiment tracking matters for three big reasons: reproducibility (science should be repeatable), organization (so anyone — including future-you — can find things), and optimization (visualizing results to improve the model).

The most basic "tracking system" is a spreadsheet of metrics — and it has real problems:

  • Error prone — usually filled in manually by copy-pasting results.
  • No standard format — a new data scientist can't easily tell what was tested, which hyperparameters changed, or what data was used.
  • Poor visibility & collaboration — a manager would have to collect everyone's spreadsheets; a teammate has to reverse-engineer what you did.
3 · Meet MLflow

🌊 An Open-Source ML Lifecycle Platform

MLflow is "an open source platform for the machine learning lifecycle." Unlike tools that need extra infrastructure (e.g. Kubeflow), MLflow is just a simple Python package — pip install mlflow gives you a local setup immediately. You can later run a server for collaboration, but that's not required at first.

The package has four main modules:

ModulePurpose
TrackingExperiment tracking — the focus of this module.
ModelsA standard format for packaging models.
Model RegistryManaging model versions and lifecycle stages.
ProjectsPackaging reproducible runs (out of scope for the course).
4 · What MLflow Tracks Per Run

🏷️ Parameters, Metrics, Tags, Artifacts, Models

MLflow organizes experiments into runs, and for each run you can track several kinds of entities:

  • Parameters — hyperparameters and anything else that affects the metric, like the path to the training data or a preprocessing choice.
  • Metrics — any evaluation number (accuracy, F1, RMSE…), on train, validation, or test.
  • Tags — metadata for later search/filter, e.g. the developer's name or the algorithm type.
  • Artifacts — any file (a visualization, even the dataset — though logging data as an artifact doesn't scale well due to duplication).
  • Models — the trained model itself (sometimes you skip this — during a hyperparameter sweep you may only care about the metrics).
On top of what you log manually, MLflow automatically records extra info: the source file name, the code version (git commit), start/end time, and the author.
5 · Setup & Basic Logging

⚙️ From Install to First Run

Work inside an isolated environment so you don't pollute your system. Then install the requirements and launch the UI — crucially, with a backend store so the model registry works:

# create + activate an isolated env, then:
pip install -r requirements.txt

# launch the UI backed by a SQLite database
mlflow ui --backend-store-uri sqlite:///mlflow.db

In the notebook, point MLflow at that backend and name the experiment (MLflow creates it if it doesn't exist, otherwise appends runs to it):

import mlflow

mlflow.set_tracking_uri("sqlite:///mlflow.db")
mlflow.set_experiment("nyc-taxi-experiment")

Wrap a trial in start_run() — everything inside the with block is attached to that run:

with mlflow.start_run():
    mlflow.set_tag("developer", "cristian")
    mlflow.log_param("train-data-path", "data/green_tripdata_2021-01.parquet")
    mlflow.log_param("valid-data-path", "data/green_tripdata_2021-02.parquet")

    alpha = 0.01
    mlflow.log_param("alpha", alpha)
    lr = Lasso(alpha)
    lr.fit(X_train, y_train)

    rmse = mean_squared_error(y_val, lr.predict(X_val), squared=False)
    mlflow.log_metric("rmse", rmse)
Logging the data paths as parameters is a poor-man's data versioning — if metrics suddenly shift later, you can see whether a different dataset was the cause. Logging a metric over epochs gives you a learning-curve plot in the UI.
6 · Hyperparameter Tuning & Visualization

🎛️ Searching With Hyperopt + XGBoost

Hyperopt uses Bayesian methods to find good hyperparameters. Your model becomes an objective function that takes parameters and returns a value to minimize; fmin drives the search over a defined search space using the tpe algorithm, and a Trials object records each run.

import xgboost as xgb
from hyperopt import fmin, tpe, hp, STATUS_OK, Trials
from hyperopt.pyll import scope

def objective(params):
    with mlflow.start_run():
        mlflow.set_tag("model", "xgboost")
        mlflow.log_params(params)
        booster = xgb.train(
            params=params, dtrain=train, num_boost_round=1000,
            evals=[(valid, "validation")], early_stopping_rounds=50,
        )
        y_pred = booster.predict(valid)
        rmse = mean_squared_error(y_val, y_pred, squared=False)
        mlflow.log_metric("rmse", rmse)
    return {"loss": rmse, "status": STATUS_OK}

search_space = {
    "max_depth": scope.int(hp.quniform("max_depth", 4, 100, 1)),
    "learning_rate": hp.loguniform("learning_rate", -3, 0),
    "reg_alpha": hp.loguniform("reg_alpha", -5, -1),
    "reg_lambda": hp.loguniform("reg_lambda", -6, -1),
    "min_child_weight": hp.loguniform("min_child_weight", -1, 3),
    "objective": "reg:linear",
    "seed": 42,
}

best = fmin(fn=objective, space=search_space, algo=tpe.suggest,
            max_evals=50, trials=Trials())

XGBoost needs its own data type — wrap the feature matrices in DMatrix:

train = xgb.DMatrix(X_train, label=y_train)
valid = xgb.DMatrix(X_val, label=y_val)

With every trial logged and tagged model=xgboost, you can filter to those runs in the UI, select them all, and compare. Three visualizations are especially useful:

  • Parallel coordinates plot — see all hyperparameters at once and which combinations drive low RMSE (e.g. lower min_child_weight correlates with lower error).
  • Scatter plot — one hyperparameter vs the metric, to spot patterns and regions worth exploring.
  • Contour plot — two hyperparameters vs the metric at once, to find promising regions.
The "best" model isn't only about the lowest metric. Also weigh training time and model size/complexity — a slightly worse but far smaller, faster model is often the better production choice.
7 · Autolog

⚡ Logging in One Line

For supported frameworks (scikit-learn, TensorFlow/Keras, Gluon, XGBoost, LightGBM, statsmodels, Spark, fastai, PyTorch), autolog records parameters, metrics, model, and even feature-importance plots automatically:

mlflow.xgboost.autolog()
# ...then just train as usual — MLflow logs everything

Autolog captures far more parameters than you'd bother to log by hand, saves the model in MLflow's format, and records the environment — at the cost of a slightly longer run while it does the logging.

8 · Model Management

📦 Saving Models the MLflow Way

Model management is the broader area around experiment tracking that also covers model versioning, deployment, and scaling. There are two ways to save a model with MLflow.

Option A — as a plain artifact (the basic way). You still have to know how to load and run the pickle yourself:

mlflow.log_artifact("models/lin_reg.bin", artifact_path="models_pickle")

Option B — as an MLflow model (the good way). MLflow stores extra metadata that makes the model easy to reload later:

mlflow.xgboost.log_model(booster, artifact_path="models_mlflow")

This produces an MLmodel file describing the model, the available flavors (every model gets a generic python_function flavor, plus a framework one like xgboost), and a conda.yaml / requirements.txt recording the exact dependencies (it infers them from your environment).

It's a good idea to also save the preprocessor (e.g. the DictVectorizer) as an artifact, since you'll need to transform raw data the same way before predicting:

with open("models/preprocessor.b", "wb") as f_out:
    pickle.dump(dv, f_out)
mlflow.log_artifact("models/preprocessor.b", artifact_path="preprocessor")

Loading a logged model later — in two flavors — using its model_uri (runs:/<run_id>/<artifact_path>):

# generic python function
model = mlflow.pyfunc.load_model(model_uri)

# native xgboost object (exposes booster methods)
booster = mlflow.xgboost.load_model(model_uri)
y_pred = booster.predict(valid)
Because the saved model carries its flavors and environment, you can later deploy it as a Python function, a Docker container, a Spark batch job, on Kubernetes, or to SageMaker / Azure — without rewriting loading code.
9 · The Model Registry

🗂️ Promoting Models to Production

Imagine a data scientist emails you a new model to deploy. You immediately have questions: what changed, what preprocessing is needed, which environment and library versions? And if production breaks, how do you roll back? The model registry solves this by being the central place where production-ready models live.

The data scientist registers promising models; the deployment engineer inspects them (hyperparameters, size, performance) and moves them between stages:

  • Staging

    Candidate

    A model considered ready and under evaluation for production.

  • Production

    Live

    The model currently serving (the registry only labels it — it does not deploy).

  • Archived

    Retired

    Older models kept around; can be pulled back to production for a rollback.

The registry does not deploy anything — stages are just labels. You complement it with CI/CD code that does the actual deployment.

You can drive the registry from Python with MlflowClient — searching runs, registering models, and transitioning stages:

from mlflow.tracking import MlflowClient
from mlflow.entities import ViewType

client = MlflowClient(tracking_uri="sqlite:///mlflow.db")

runs = client.search_runs(
    experiment_ids="1",
    filter_string="metrics.rmse < 6.8",
    run_view_type=ViewType.ACTIVE_ONLY,
    max_results=5,
    order_by=["metrics.rmse ASC"],
)

# register a run's model
mlflow.register_model(model_uri="runs:/<run_id>/model",
                      name="nyc-taxi-regressor")

# move version 4 to Production
client.transition_model_version_stage(
    name="nyc-taxi-regressor", version=4,
    stage="Production", archive_existing_versions=True,
)
Model lineage: each registered version links back to the run that produced it, so you can always recover its parameters, metrics, data paths, and environment. Versioning is automatic, and you can annotate versions with descriptions and tags.

To choose between candidates, load each stage's model and compare on fresh test data (e.g. March), weighing both error and latency before promoting.

10 · MLflow in Practice

🏢 Three Scenarios, Three Setups

How you run MLflow depends on context. Three configuration knobs matter: the backend store (metadata — local files or a SQL database), the artifact store (models/files — local or S3), and whether you run a tracking server (none, local, or remote).

ScenarioTracking serverBackend storeArtifact storeRegistry?
Solo data scientist (competition)NoneLocal filesLocal filesNot needed
Cross-functional team, one DSLocalSQLiteLocal (or cloud)Useful
Multiple DS collaboratingRemote (e.g. EC2)Postgres (RDS)S3Required

With no configuration, MLflow defaults to a local mlruns/ folder and the model registry is unavailable — switch the backend to SQLite to enable it. A common gotcha: launch mlflow ui from the same directory where the runs were created, or it won't find them.

For a full team setup, run a remote server backed by Postgres + S3:

mlflow server \
  -h 0.0.0.0 -p 5000 \
  --backend-store-uri postgresql://USER:PASS@ENDPOINT:5432/DB_NAME \
  --default-artifact-root s3://mlflow-artifacts-remote

The AWS recipe: launch an EC2 instance, create an S3 bucket, create a Postgres RDS database, open the right security-group ports (5000 for HTTP, 5432 for Postgres), then SSH in and start the server. Clients point at it with mlflow.set_tracking_uri("http://<public-dns>:5000") — and must have AWS credentials configured locally so MLflow can write artifacts to S3.

11 · Limits, Issues & Alternatives

⚖️ Where MLflow Stops

Running a shared remote server raises real operational issues:

  • Security — restrict access (e.g. behind a VPN) so outsiders can't see experiments and models.
  • Scalability — for thousands of runs, consider deploying on AWS Fargate or scaling patterns used by large teams.
  • Isolation — shared servers get messy; standardize experiment/model naming (prefix with team name) and enforce default tags to keep teams' runs separable. Use separate S3 buckets/accounts to control artifact access.

Known limitations of open-source MLflow:

  • No authentication / users / teams (the managed Databricks version adds this).
  • No built-in data versioning — work around it with data paths, hashes, or (for small data) artifacts.
  • No model/data monitoring & alerting — outside MLflow's scope; better handled by a dedicated tool.
Alternatives like Neptune, Weights & Biases, and Comet are free for individuals but paid for teams. They offer similar tracking, sometimes with extras MLflow lacks — users/teams, dirty-commit detection, hardware-consumption logging, and richer integrations.
