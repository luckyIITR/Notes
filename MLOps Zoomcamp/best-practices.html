MLOps Zoomcamp · Module 6

Best Engineering Practices

Turning a working model service into solid software — unit & integration tests, testing cloud services locally, linting and formatting, pre-commit hooks, Makefiles, Infrastructure as Code with Terraform, and CI/CD with GitHub Actions.

1 · The Starting Point

🧱 The Streaming Example from Module 4

This module reuses the optional streaming service built in Module 4 and improves it from an engineering point of view. The architecture is intentionally simple:

input stream
(events)
→
Lambda
(model service)
→
S3 model
→
predictions stream

A Kinesis stream carries ride events. An AWS Lambda reads each event, fetches the model from an S3 bucket, computes the trip duration, and writes the prediction to a second Kinesis stream. Two streams, one function, one bucket.

The goal of Module 6 isn't new ML — it's taking existing code and making it reliable, readable, and automated. Start by copying the Module 4 streaming folder so you never touch the original.
2 · Unit Tests with pytest

🧪 Setting Up the Test Environment

The first improvement is a test suite. pytest is installed as a dev dependency — needed when developing, not in production:

# install only for the dev environment
pipenv install --dev pytest
  • Create a tests/ folder with an __init__.py so Python treats it as a package and tests can import the application code.
  • In VS Code, select the right interpreter (use pipenv --venv to find the virtualenv path) and configure the testing panel to use pytest pointed at tests/.
  • Tests can also run from the shell with pipenv run pytest tests/.
Unit tests check small, isolated pieces of code — individual functions and methods. They must be fast and independent: no S3 calls, no model downloads, no network.

♻️ Refactoring for Testability

The original Lambda code resisted testing: simply importing lambda_function ran module-level code that tried to download the model and crashed. Long functions plus global side-effects make code hard to test.

The fix is to wrap everything in a ModelService class so nothing executes on import. The Lambda handler becomes a thin delegate:

# lambda_function.py — just wiring
import model

PREDICTIONS_STREAM_NAME = os.getenv('PREDICTIONS_STREAM_NAME', 'ride_predictions')
RUN_ID = os.getenv('RUN_ID')
TEST_RUN = os.getenv('TEST_RUN', 'False') == 'True'

model_service = model.init(
    prediction_stream_name=PREDICTIONS_STREAM_NAME,
    run_id=RUN_ID,
    test_run=TEST_RUN,
)

def lambda_handler(event, context):
    return model_service.lambda_handler(event)
# model.py — logic lives in a class
class ModelService:
    def __init__(self, model, model_version=None, callbacks=None):
        self.model = model
        self.model_version = model_version
        self.callbacks = callbacks or []

    def prepare_features(self, ride):
        features = {}
        features['PU_DO'] = '%s_%s' % (ride['PULocationID'], ride['DOLocationID'])
        features['trip_distance'] = ride['trip_distance']
        return features

    def predict(self, features):
        pred = self.model.predict(features)
        return float(pred[0])
The class is slightly more verbose, but it never runs on import, is easier to extend, and — crucially — is testable.

✅ Writing the First Tests

With the refactor in place, test the pure pieces. Pass None as the model when the method under test doesn't use it:

import model

def test_prepare_features():
    model_service = model.ModelService(None)
    ride = {'PULocationID': 130, 'DOLocationID': 205, 'trip_distance': 3.66}

    actual_features = model_service.prepare_features(ride)
    expected_features = {'PU_DO': '130_205', 'trip_distance': 3.66}

    assert actual_features == expected_features

The base64 decode logic — easy to get wrong — is extracted into its own method and tested in isolation:

def test_base64_decode():
    base64_input = read_text('data.b64')          # the encoded Kinesis record

    actual_result = model_service.base64_decode(base64_input)
    expected_result = {
        'ride': {'PULocationID': 130, 'DOLocationID': 205, 'trip_distance': 3.66},
        'ride_id': 256,
    }
    assert actual_result == expected_result

🎭 Mocking the Model

To test predict without touching S3 or MLflow, swap the real model for a tiny mock — an object that mimics the real thing but returns a fixed value:

class ModelMock:
    def __init__(self, value):
        self.value = value

    def predict(self, X):
        return [self.value] * len(X)      # same value for every row

def test_predict():
    model_mock = ModelMock(10.0)
    model_service = model.ModelService(model_mock)

    features = {'PU_DO': '130_205', 'trip_distance': 3.66}
    actual_prediction = model_service.predict(features)

    assert actual_prediction == 10.0
A mock keeps the test fast and offline. You're verifying your code's flow, not the model's accuracy.
3 · Decoupling Side-Effects with Callbacks

🔌 Keeping Kinesis out of the Core

Putting records onto Kinesis is a side-effect that would force ModelService to know about clients and stream names. Instead, the class accepts a list of callbacks invoked after each prediction; the Kinesis writer becomes one such callback in its own class:

class KinesisCallback:
    def __init__(self, kinesis_client, prediction_stream_name):
        self.kinesis_client = kinesis_client
        self.prediction_stream_name = prediction_stream_name

    def put_record(self, prediction_event):
        ride_id = prediction_event['prediction']['ride_id']
        self.kinesis_client.put_record(
            StreamName=self.prediction_stream_name,
            Data=json.dumps(prediction_event),
            PartitionKey=str(ride_id),
        )

def init(prediction_stream_name, run_id, test_run):
    model = load_model(run_id)
    callbacks = []
    if not test_run:                               # only wire Kinesis for real runs
        kinesis_client = boto3.client('kinesis')
        kinesis_callback = KinesisCallback(kinesis_client, prediction_stream_name)
        callbacks.append(kinesis_callback.put_record)
    return ModelService(model, run_id, callbacks)

In the handler, the service just loops over callbacks: for callback in self.callbacks: callback(prediction_event). Default is an empty list, so existing tests keep passing untouched.

Use the Python idiom callbacks or [] — if callbacks is None, the expression evaluates to an empty list.

🧬 Testing the Whole Handler

A final unit test runs the full lambda_handler end-to-end with the mock model, asserting on the complete response dictionary:

def test_lambda_handler():
    model_mock = ModelMock(10.0)
    model_version = 'Test123'
    model_service = model.ModelService(model_mock, model_version)

    event = read_json('event.json')               # a trimmed Kinesis event
    actual_predictions = model_service.lambda_handler(event)

    expected_predictions = {
        'predictions': [{
            'model': 'ride_duration_prediction_model',
            'version': model_version,
            'prediction': {'ride_duration': 10.0, 'ride_id': 256},
        }]
    }
    assert actual_predictions == expected_predictions
4 · Integration Tests

🔗 From Manual Script to Real Test

Unit tests check pieces in isolation; an integration test checks the whole service works together. The starting point is the manual test_docker.py that sends a request to the containerized service and prints the response. Turning it into a real test is mostly adding asserts:

actual_response = requests.post(url, json=event).json()
print('actual response:')
print(json.dumps(actual_response, indent=2))

assert actual_response == expected_response
assert is built into Python — pytest just makes running many tests convenient. A failing assert exits with a non-zero code, which is exactly what CI needs.

🔬 DeepDiff for Friendly Comparisons

A bare assert tells you that two dicts differ, not where. The DeepDiff library pinpoints the difference and supports float tolerance — vital since we care about minutes, not floating-point noise:

from deepdiff import DeepDiff

diff = DeepDiff(actual_response, expected_response, significant_digits=1)
print(f'diff = {diff}')

assert 'type_changes' not in diff
assert 'values_changed' not in diff

significant_digits=1 tells DeepDiff to ignore everything past the first decimal, so an approximate duration still passes. An empty diff means the responses match.

This comparison surfaced a real bug — the model version came back null because ModelService wasn't initialized with model_version and callbacks. Good tests find your mistakes.

📦 Removing the S3 Dependency

An integration test shouldn't depend on S3 — the network might be down, the bucket gone, or permissions missing. The model loader is made flexible enough to read from a local path instead, configured via environment variables:

def get_model_location(run_id):
    model_location = os.getenv('MODEL_LOCATION')
    if model_location is not None:                 # local path wins if set
        return model_location

    model_bucket = os.getenv('MODEL_BUCKET', 'mlflow-models-alexey')
    experiment_id = os.getenv('MLFLOW_EXPERIMENT_ID', '1')
    return f's3://{model_bucket}/{experiment_id}/{run_id}/artifacts/model'

Download the model once into an integration-test/model/ folder and mount it into the container so the test runs fully offline:

aws s3 cp --recursive s3://<bucket>/1/<run_id>/ ./model
docker run -it --rm \
  -p 8080:8080 \
  -e MODEL_LOCATION=/app/model \
  -v $(pwd)/model:/app/model \
  stream-model-duration:v2
5 · Automating the Integration Test

🤖 A run.sh Orchestrator

Building, running, and testing by hand across terminals is tedious. A single run.sh automates everything. Key tricks:

  • cd to the script's own directory so it works no matter where it's called from.
  • Tag the image with a timestamp (not just latest) so each build is identifiable — a git hash isn't enough when code is uncommitted.
  • Use docker-compose to keep configuration tidy, and start it in detached mode (up -d) so the script keeps control of the terminal.
cd "$(dirname "$0")"

LOCAL_TAG=$(date +"%Y-%m-%d-%H-%M")
export LOCAL_IMAGE_NAME="stream-model-duration:${LOCAL_TAG}"

docker build -t ${LOCAL_IMAGE_NAME} ..
docker-compose up -d
sleep 1                                 # give services time to start

pipenv run python test_docker.py
ERROR_CODE=$?

if [ ${ERROR_CODE} != 0 ]; then
    docker-compose logs                 # show why it failed
    docker-compose down
    exit ${ERROR_CODE}
fi

docker-compose down
Capture the exit code in a variable rather than relying on set -e — otherwise an early exit skips docker-compose down and leaves containers running. A non-zero exit makes CI mark the job failed.
6 · Testing Cloud Services with LocalStack

☁️ A Local AWS for the Kinesis Callback

The one piece still untested is the KinesisCallback — it only fires on real runs. LocalStack provides a fully local AWS stack so you can test Kinesis (or S3) with no AWS account and no internet. Add it to docker-compose:

# docker-compose.yaml
kinesis:
  image: localstack/localstack
  ports:
    - "4566:4566"
  environment:
    - SERVICES=kinesis

The AWS CLI talks to LocalStack by overriding the endpoint URL — nothing reaches the real cloud:

aws --endpoint-url=http://localhost:4566 \
    kinesis list-streams

aws --endpoint-url=http://localhost:4566 \
    kinesis create-stream --stream-name ride_predictions --shard-count 1

🔧 Pointing the Code at LocalStack

The service picks up an endpoint from an environment variable. Inside the compose network, services reference each other by name (kinesis); from outside, use localhost:

def create_kinesis_client():
    endpoint_url = os.getenv('KINESIS_ENDPOINT_URL')
    if endpoint_url is None:
        return boto3.client('kinesis')
    return boto3.client('kinesis', endpoint_url=endpoint_url)

A test_kinesis.py script (using boto3) fetches a shard iterator, reads the records, decodes them, and compares against the expected record with DeepDiff. The integration run.sh creates the stream, runs test_docker.py, then test_kinesis.py, tearing everything down at the end.

In Lambda you never write this shard-iterator plumbing — AWS handles consuming records for you. LocalStack is what lets you verify the producer side locally.
7 · Code Quality: Linting

🔎 pylint & PEP 8

Beyond reliability, code should be clean. PEP 8 is Python's style guide; a linter statically analyzes code to check style and catch common mistakes (unused arguments, bad names, missing docstrings, global variables). The module uses pylint:

pipenv install --dev pylint
pylint --recursive=y .

pylint integrates with VS Code (highlights warnings inline) and — importantly — exits with a non-zero code when warnings exist, so it can gate commits and CI. Some warnings are genuinely useful: it flagged a real cross-platform bug where files were opened without encoding='utf-8'.

Suppress warnings deliberately, not blindly — disable globally for conventions you accept (e.g. missing-docstring), or locally for one class with an inline # pylint: disable=... comment.

⚙️ Configuring via pyproject.toml

Rather than scattering config files, modern tools share a single pyproject.toml. Disable accepted warnings there:

[tool.pylint.message_control]
disable = [
    "missing-function-docstring",
    "missing-module-docstring",
    "missing-class-docstring",
    "too-few-public-methods",
]

Long inline data (like base64 strings) that triggered line-too-long is moved into separate files (data.b64, event.json) and read at runtime — cleaner code and no warning. Use pathlib.Path(__file__).parent so the file is found relative to the script.

8 · Code Quality: Formatting

🎨 black & isort

Two tools automate formatting so you never do it by hand:

Formatter

black

Reformats code to a consistent style — spacing, line length, trailing commas. Opinionated, but configurable.

Import sorter

isort

Sorts and groups imports (standard library, then third-party, then named imports).

Preview before applying with black --diff . and isort --diff .. Commit first so changes are reversible. Configure both in pyproject.toml:

[tool.black]
line-length = 88
target-version = ['py39']
skip-string-normalization = true     # keep single quotes

[tool.isort]
line_length = 88
length_sort = true                    # sort imports by length
A trailing comma in a multi-line collection tells black to leave it multi-line. The full quality run is: isort . → black . → pylint --recursive=y . → pytest tests/.
9 · Pre-commit Hooks

🪝 Run Checks Before Code Enters Git

It's easy to forget to format or test. A git pre-commit hook runs checks automatically before every commit, blocking it if anything fails. The pre-commit tool manages these hooks:

pipenv install --dev pre-commit
pre-commit sample-config > .pre-commit-config.yaml
pre-commit install                    # writes .git/hooks/pre-commit

Since the hook lives in .git/hooks (never committed), every teammate must run pre-commit install once after cloning. The config chains the same quality tools:

repos:
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.3.0
    hooks:
      - id: trailing-whitespace
      - id: end-of-file-fixer
      - id: check-yaml
      - id: check-added-large-files
  - repo: https://github.com/pycqa/isort
    rev: 5.10.1
    hooks: [{ id: isort }]
  - repo: https://github.com/psf/black
    rev: 22.6.0
    hooks: [{ id: black }]
  - repo: local
    hooks:
      - id: pylint
        name: pylint
        entry: pylint
        language: system
        types: [python]
        args: ["--recursive=y"]
      - id: pytest
        name: pytest
        entry: pytest
        language: system
        pass_filenames: false
        args: ["tests/"]
If black reformats a file, the commit fails but the file is already fixed — just git add and commit again. This saves you from ever forgetting to test or format.
10 · Makefiles

📋 One Command Instead of Many

A Makefile defines named aliases (targets) and dependencies between them, forming a graph of steps. Instead of remembering long commands, run make test or make publish — with tab auto-completion:

LOCAL_TAG := $(shell date +"%Y-%m-%d-%H-%M")
LOCAL_IMAGE_NAME := stream-model-duration:${LOCAL_TAG}

test:
	pytest tests/

quality_checks:
	isort .
	black .
	pylint --recursive=y .

build: quality_checks test
	docker build -t ${LOCAL_IMAGE_NAME} .

integration_test: build
	LOCAL_IMAGE_NAME=${LOCAL_IMAGE_NAME} bash integration-test/run.sh

publish: build integration_test
	bash scripts/publish.sh ${LOCAL_IMAGE_NAME}

setup:
	pipenv install --dev
	pre-commit install
Use $(shell ...), not $(...), so the date is evaluated once in bash and reused — otherwise each target re-evaluates it and the tag can drift between build and test. make setup bootstraps a fresh clone in one step.
11 · Infrastructure as Code with Terraform

🏗️ Defining the Pipeline as Code

Infrastructure as Code (IaC) lets you define and automate cloud resources in version-controlled files instead of clicking through a console. Terraform is the chosen tool — open source and cloud-agnostic. The same streaming pipeline is provisioned in AWS:

Kinesis
(source)
→
Lambda
→
S3 model
→
Kinesis
(output)

An ECR repository holds the Lambda's Docker image, and a CloudWatch event source mapping triggers the Lambda whenever a record arrives in the source stream.

Terraform stores its state (a snapshot of managed resources) in a backend. Here it's a remote S3 bucket you must create manually first — Terraform won't create its own state bucket.

🧩 Modules & Output Variables

A Terraform project needs at least a main.tf; variables.tf holds inputs and defaults. The core building blocks:

  • Provider — installs the AWS plugin (analogous to importing a library); configures region and credentials.
  • Resource — a physical or logical component (a Kinesis stream, an S3 bucket).
  • Module — a reusable, black-box component, ideal for avoiding duplication across staging and production.
  • Output variable — a value a module returns after creation (e.g. a stream's ARN), used to wire one resource into another and manage dependencies.
# modules/kinesis/main.tf
resource "aws_kinesis_stream" "stream" {
  name             = var.stream_name
  shard_count      = var.shard_count
  retention_period = var.retention_period
  tags = { CreatedBy = var.tags }
}

output "stream_arn" {
  value = aws_kinesis_stream.stream.arn   # returned for use elsewhere
}
# main.tf — referencing the module
module "source_kinesis_stream" {
  source           = "./modules/kinesis"
  stream_name      = "${var.source_stream_name}-${var.project_id}"
  shard_count      = 2
  retention_period = 48
}

⛓️ The Tricky Lambda Dependencies

Modules are created in parallel, but the Lambda depends on the Kinesis streams, the S3 model, and an existing image in ECR. In a monorepo you must force this ordering. A null_resource builds and pushes the image first, with a local-exec provisioner and an MD5 trigger so the image rebuilds only when the Dockerfile or handler changes:

resource "null_resource" "ecr_image" {
  triggers = {
    python_file = md5(file(var.lambda_function_local_path))
    docker_file = md5(file(var.docker_image_local_path))
  }
  provisioner "local-exec" {
    command = <<EOF
      ... docker build & push to ECR ...
    EOF
  }
}

The standard workflow is terraform init → plan (a dry run against state) → apply. Per-environment values live in .tfvars files (stg.tfvars, prod.tfvars), passed with -var-file, and a suffix on every resource keeps staging and prod clusters separate.

An IAM role grants the Lambda permission to read the input stream, write the output stream, fetch the S3 model, and log to CloudWatch. The CloudWatch event source mapping is what actually triggers the function on each new record.
12 · CI/CD with GitHub Actions

🚦 Continuous Integration & Delivery

CI/CD shortens and de-risks delivery. CI develops, tests, and packages code; CD ships it to the environments that depend on it. Workflows live in .github/workflows/ and are triggered by repo events. The module builds two:

Workflow 1 · CI

On pull request

Runs unit tests, integration tests (against LocalStack), and linting; runs terraform plan to validate infrastructure changes.

Workflow 2 · CD

On merge / push

Runs terraform apply, builds & pushes the Docker image to ECR, then updates the Lambda config.

Each job runs in its own container, so every job re-checks out the repo and re-configures AWS. Jobs run in parallel unless you declare dependencies. Secrets (AWS keys) come from the repo's GitHub Secrets.

📝 The CI Workflow

The CI workflow triggers on a pull request to develop, scoped to the module's path. Its test job sets up Python, installs dependencies, then runs the full local test suite:

name: CI-Tests
on:
  pull_request:
    branches: ['develop']
    paths: ['06-best-practices/code/**']

env:
  AWS_DEFAULT_REGION: 'eu-west-1'

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-python@v2
        with: { python-version: '3.9' }
      - name: Install dependencies
        run: pip install pipenv && pipenv install --dev
      - name: Run unit tests
        run: pipenv run pytest tests/
      - name: Lint
        run: pipenv run pylint --recursive=y .
      - uses: aws-actions/configure-aws-credentials@v1
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
      - name: Integration Test
        run: bash integration-test/run.sh

  tf-plan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: aws-actions/configure-aws-credentials@v1
        with: { ... }
      - uses: hashicorp/setup-terraform@v2
      - name: TF plan
        run: terraform init && terraform plan -var-file=vars/prod.tfvars
A small flag in run.sh detects the GitHub CI environment so the working-directory logic behaves both locally and in CI. LocalStack in CI also needs dummy AWS keys set in the integration test's docker-compose.

🚀 The CD Workflow

The CD workflow triggers on push to develop (i.e. when a PR merges) and runs as one sequential job with three sections:

  1. Define infrastructure — terraform init + plan, then apply -auto-approve (only if the plan step succeeded). Output variables expose the ECR repo, prediction stream, model bucket, and Lambda name.
  2. Build & push — log in to ECR, build the Lambda image, and push it (tagged latest, or a git SHA for blue-green style versioning).
  3. Deploy — fetch the latest model run_id, copy artifacts into the Terraform-created bucket, then aws lambda update-function-configuration with the fresh environment variables.
The image is first deployed by Terraform, but the CD deploy step keeps the application logic fresh independently of infrastructure changes — that's why the two are decoupled. Add a short sleep so the Lambda finishes bootstrapping before you update its config.
13 · Module Recap

🎓 What Good Engineering Looks Like

Reliability
  • Unit tests for isolated functions
  • Integration tests for the whole service
  • LocalStack to test cloud services offline
Automation & Hygiene
  • Linting & formatting (pylint, black, isort)
  • Pre-commit hooks & Makefiles
  • Terraform IaC + GitHub Actions CI/CD

Together these practices take a fragile script and turn it into a service you can change with confidence — tests catch regressions, formatters keep it readable, and automation deploys it the same way every time.

The natural next step is Continuous Training (CT) — automating model retraining and registry integration so new model versions flow into production through the same CI/CD machinery.
