1 · What Is MLOps?

🛠️ A Set of Best Practices for Production ML

Search any engine for "MLOps" and you will land on Wikipedia, where you read that MLOps is a set of best practices for putting machine learning into production. That single idea is the entire focus of this course — the practices, and the tools that implement them, for taking a model out of a notebook and into the real world.

MLOps is not one tool. It is a collection of practices for automating everything and for working together as a team across the whole lifecycle of a machine learning project.

Concretely, MLOps helps at every stage: experimenting without losing your mind, reproducing results, retraining a model with one click, deploying it easily, and monitoring its quality once it is live.

3 · A Simplified ML Project Process

🔄 Design → Train → Operate

To see where best practices fit, it helps to look at a very simplified view of an ML project. In reality there are more stages, but in the simplest form there are three.

1 · Design
→
2 · Train
→
3 · Operate
  • Design — decide whether machine learning is even the right tool. Do we actually need ML to predict trip duration, or is something simpler enough? If ML is warranted, move on.
  • Train — run experiments to find the best possible model. The output of this stage is a trained model ready to apply to new data.
  • Operate — put the model to work. Typically deploy it as a web service behind an API so the customer's phone can send pickup/drop-off and get back a predicted duration.
Deployment lives inside the operate stage — but operating doesn't stop there. Once the model is live we must make sure it keeps performing well and does not degrade over time, which is where monitoring comes in.
4 · Reading the NYC Taxi Data (Parquet)

📦 CSV Is Gone — Everything Is Parquet Now

The NYC TLC trip-record dataset used to be distributed as CSV, but it has since switched to the Parquet format. Parquet files are much smaller than the equivalent CSV, which is almost certainly why the switch was made.

Download the files with wget (the February URL follows the same pattern — just swap the month):

# inside the data folder
wget https://.../green_tripdata_2021-01.parquet
wget https://.../green_tripdata_2021-02.parquet

Read them with Pandas using read_parquet instead of read_csv. This needs an engine — install pyarrow (or fastparquet) first:

# pip install pyarrow
import pandas as pd

df = pd.read_parquet("data/green_tripdata_2021-01.parquet")
A pleasant side effect: with Parquet the datetime columns arrive already typed as datetime, so the manual pd.to_datetime(...) parsing step needed for CSV is no longer required.
5 · A Baseline Duration Model (Optional)

🧪 Building the Model the Data-Science Way

This walkthrough is optional — you can skip to the result. It shows how a data scientist typically works: open a notebook, import, experiment, and execute cells in a very particular order.

Compute the target

The target is trip duration in minutes, derived from the drop-off and pickup timestamps:

df["duration"] = df.lpep_dropoff_datetime - df.lpep_pickup_datetime
df["duration"] = df.duration.apply(lambda td: td.total_seconds() / 60)

The raw distribution has a very long tail — some trips last almost a full day, others under a minute. Filtering to a sensible business range (1 to 60 minutes) removes the noise:

df = df[(df.duration >= 1) & (df.duration <= 60)]

Vectorize the features

Use pickup and drop-off location IDs as categorical features and trip distance as a numerical feature. Cast the categoricals to strings so DictVectorizer one-hot encodes them instead of treating them as numbers:

from sklearn.feature_extraction import DictVectorizer

categorical = ["PULocationID", "DOLocationID"]
numerical = ["trip_distance"]
df[categorical] = df[categorical].astype(str)

train_dicts = df[categorical + numerical].to_dict(orient="records")
dv = DictVectorizer()
X_train = dv.fit_transform(train_dicts)
y_train = df["duration"].values

Train and evaluate

Fit a plain linear regression and measure the root mean squared error:

from sklearn.linear_model import LinearRegression
from sklearn.metrics import mean_squared_error

lr = LinearRegression()
lr.fit(X_train, y_train)

y_pred = lr.predict(X_train)
mean_squared_error(y_train, y_pred, squared=False)   # ≈ 9 min

Around 9 minutes of error on train, ~10 on validation (February). Lasso and Ridge with various alpha values did not beat plain linear regression on this data.

A better feature: the pickup–dropoff pair

Instead of using the two locations separately, combine them into a single interaction feature. This captures "the typical duration from this pickup to this drop-off" and improves the model significantly:

df["PU_DO"] = df["PULocationID"] + "_" + df["DOLocationID"]
categorical = ["PU_DO"]

Save the model

Persist both the vectorizer and the model together with pickle — you need both to make predictions later:

import pickle

with open("models/lin_reg.bin", "wb") as f_out:
    pickle.dump((dv, lr), f_out)
This notebook is intentionally messy — you cannot just run it top to bottom and expect the best model. That fragility is exactly the motivation for the rest of the course.

The pipeline naturally decomposes into steps that don't all need re-running every time: load & prepare data → vectorize → train model. Once parameterized (which month, which model), the whole thing reduces to a single command like python pipeline.py.

At the highest level of automation, monitoring detects a performance drop, automatically triggers the pipeline to retrain on fresh data, and automatically deploys the new model — no human in the loop.
7 · The MLOps Maturity Model

📈 Five Levels, From No MLOps to Full Automation

Borrowed from Microsoft's MLOps maturity article, the model defines five levels — from level 0 (no MLOps) to level 4 (full automation). The numbering starts at zero, for fun.

  • 0

    No MLOps automation

    All work in sloppy Jupyter notebooks — no pipelines, no experiment tracking, no model metadata. Data scientists work alone and hand notebooks to engineers to re-implement. Fine for a proof of concept.

  • 1

    DevOps, but no MLOps

    Solid software engineering: automated releases, unit & integration tests, CI/CD, and operational metrics (requests/sec, network). But the system is not ML-aware — no experiment tracking, hard to reproduce a model.

  • 2

    Automated training

    A parameterized training pipeline (even just python train.py), plus experiment tracking and a model registry. Deployment is still manual but low-friction. Data scientists and engineers work together. Worthwhile once you have 2–3 proven use cases.

  • 3

    Automated deployment

    Deploying a model is effortless — often a single API call to an ML platform — and can be part of the training pipeline. Adds A/B testing between model versions and model monitoring.

  • 4

    Full MLOps automation

    Training and deployment fully automated together. Monitoring detects drift, triggers retraining, deploys (and A/B tests) the new model, and rolls it out — with no human involved. Requires deep trust in the system.

Be pragmatic. Not every project needs level 4 — for many models level 2 (perhaps with monitoring) is plenty. Match the maturity level to the value and risk of the use case rather than chasing full automation everywhere. This course aims to get you to a solid level 2 touching level 3.
