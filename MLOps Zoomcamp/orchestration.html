MLOps Zoomcamp · Module 3

ML Pipelines & Orchestration

Turning a messy Jupyter notebook into a reproducible training pipeline — from a Python script to workflow orchestration with Airflow (or alternatives).

1 · Why Pipelines?

📓 From Notebook Mess to Something Runnable

In Modules 1 and 2 you built a ride-duration model inside a long Jupyter notebook — experiment tracking, Hyperopt tuning, XGBoost training, MLflow logging. It works, but the notebook is a mess: cells run in fragile order, blocks are hard to identify, and rerunning it means opening your laptop and clicking through cells one by one.

A training pipeline is, in simple terms, a sequence of steps executed in order to train a model. The notebook is loosely a pipeline (execute everything → get a model in the registry), but it is not maintainable, reliably re-executable, or parameterized.

The goal of this module: take that notebook and turn it into something reproducible, runnable, and parameterized — and then layer a workflow orchestrator on top.
2 · Pipeline vs Orchestration

🔄 Two Related Terms

  • Workflow orchestration — a general data-engineering term for how you schedule and organize steps that must run in order.
  • Machine learning pipeline — the ML-specific case: an ordered sequence of steps whose output is a trained model.

An ML pipeline is a kind of workflow; orchestration is the broader idea of running any multi-step job reliably.

4 · Plain Python Script

🐍 The First Good Step

The easiest upgrade from a notebook is a single Python script with clean functions called in order — already a maintainable pipeline you can test and rerun:

# rough structure of duration_prediction.py
def read_dataframe(year, month): ...
def create_X(df, dv=None): ...
def train_model(X_train, y_train, X_val, y_val, dv): ...

def run(year, month):
    df_train = read_dataframe(year, month)
    next_year, next_month = (year, month + 1) if month < 12 else (year + 1, 1)
    df_val = read_dataframe(next_year, next_month)
    X_train, dv = create_X(df_train)
    X_val, _ = create_X(df_val, dv)
    y_train, y_val = df_train.duration.values, df_val.duration.values
    return train_model(X_train, y_train, X_val, y_val, dv)

Refinements from the course walkthrough:

  • Convert the notebook with nbconvert: jupyter nbconvert --to script duration_prediction.ipynb
  • Parameterize with argparse — --year and --month instead of hard-coded January/February
  • Build the Parquet URL from year/month: green_tripdata_{year}-{month:02d}.parquet
  • Point MLflow at a running instance: mlflow.set_tracking_uri("http://localhost:5000")
  • Reuse known-best XGBoost params (skip Hyperopt in the script); ~30 boost rounds is enough for the demo
  • Return and persist the run ID (e.g. JSON file) so you can promote the model to staging later
  • Create the models/ folder programmatically with Path("models").mkdir(exist_ok=True)
Even without an orchestrator, python duration_prediction.py --year 2021 --month 1 is already a training pipeline — reproducible, parameterized, and logged to MLflow.
5 · Why You Still Need an Orchestrator

⚠️ Limits of a Bare Script

A script on your laptop breaks down in production:

  • Scheduling — who runs the cron job? What if you have many pipeline files?
  • Centralization — code shouldn't live on one developer's machine; the team needs a shared place to deploy and update it.
  • Scalability — one machine won't handle growing job volume.
  • Failure & retries — a 2-second network blip during download kills the whole run; hand-rolling retry logic makes the code complex.
  • Dependency management — steps must run in order; if step 1 fails, step 2 must not start.

Workflow orchestrators solve this: centralized server, team collaboration, scaling, monitoring/alerting, and built-in dependency + retry handling.

6 · Orchestrator Landscape

🛠️ General-Purpose vs ML-Specific

TypeToolsNotes
General-purposeAirflow, Prefect, Dagster, Kestra, Mage, LuigiGood for data engineering and ML; flexible DAG/workflow definitions.
ML-specificKubeflow Pipelines, MLflow ProjectsLess flexible, tightly focused on ML workflows.
Airflow (recommended here)
  • Industry standard — huge community, docs, and hiring signal
  • Mature scheduling, backfill, and operational tooling
  • Integrates with cloud providers (MWAA, Cloud Composer)
Airflow trade-offs
  • Steeper learning curve — DAG boilerplate, scheduler/worker setup
  • Heavier locally than Prefect or Mage
  • Less Python-native than Prefect/Dagster for quick experiments

Prefect and Dagster are lighter and more Pythonic for new projects; Mage blends notebooks with pipelines. For this course the pattern is the same — pick one tool and wire the script's steps into it. Airflow is the default choice when you want maximum portability and ops maturity.

7 · Using an Orchestrator

🚀 Hands-On Path (No Dedicated Video — Build It Yourself)

There is no walkthrough video for this unit. Take the duration_prediction.py script from the previous section and turn it into a production pipeline with your chosen orchestrator. ChatGPT or the course repo can help with boilerplate.

Step 1 · Choose the tool

Pick an orchestrator. Examples:

  • Airflow — recommended; best if you care about industry standard and cloud deployment paths
  • Prefect, Dagster, Kestra, Mage, Luigi — valid alternatives with different ergonomics

Step 2 · Run the tool locally

  1. Install and configure the orchestrator on your machine (for Airflow: pip install apache-airflow, init DB, start webserver + scheduler).
  2. Run the simplest "hello world" workflow to confirm the setup works before touching the ML code.

Step 3 · Orchestrate the workflow

Split duration_prediction.py into orchestrator tasks that mirror the pipeline steps:

ingest
→
transform
→
prepare_features
→
train_and_log

In Airflow, each step becomes a PythonOperator (or @task in TaskFlow) with explicit upstream/downstream dependencies so ingest must succeed before transform, and so on. The orchestrator handles retries if read_dataframe fails on a transient network error.

Step 4 · Parametrize & schedule

Schedule the workflow to run monthly. Derive data windows from the execution date:

Monthly data windows
train = 2 months ago  ·  validation = 1 month ago
If the DAG runs on 2021-03-01, train on 2021-01, validate on 2021-02.
Pass year and month as DAG params or compute them from {{ ds }} / logical_date in Airflow. 
# Airflow-style param derivation (conceptual)
from dateutil.relativedelta import relativedelta

execution_date = context["logical_date"]
train_date = execution_date - relativedelta(months=2)
val_date   = execution_date - relativedelta(months=1)

train_year, train_month = train_date.year, train_date.month
val_year, val_month     = val_date.year, val_date.month

Step 5 · Backfilling

Learn to backfill — run the workflow for past months you missed. In Airflow:

airflow dags backfill taxi_training \
  --start-date 2021-01-01 \
  --end-date 2021-06-01

This replays the DAG for each scheduled interval in the range, training on rolling two-month-old / one-month-old windows.

Step 6 · Deployment (optional)

Deploy the orchestrator to the cloud so the team shares one scheduler:

  • Airflow → AWS MWAA, Google Cloud Composer, or self-hosted on Kubernetes
  • Prefect → Prefect Cloud or self-hosted server
  • Dagster → Dagster Cloud or Docker on a VM
After training, use the saved run ID from MLflow to promote the model to staging in the registry — the orchestrator trains; the registry + CI/CD handles deployment.
8 · The Script — What Airflow Will Run

📄 duration_prediction.py Anatomy

Hard dependencies the DAG must respect:

  • MLflow tracking server at http://localhost:5000 must be up before training.
  • DictVectorizer must be fit on train before transforming validation — order matters.
  • The script writes models/preprocessor.b locally before logging it to MLflow.

9 · Local Airflow Setup

⚙️ Run Airflow Beside MLflow

  1. Start MLflow in one terminal: mlflow ui --backend-store-uri sqlite:///mlflow.db (or your existing setup on port 5000).
  2. Install Airflow in the same env as the script: pip install "apache-airflow==2.10.4" pandas pyarrow xgboost scikit-learn mlflow
  3. Init and start Airflow (Airflow 2.x standalone is enough locally):
    airflow standalone — note the admin password printed to the terminal.
  4. Put duration_prediction.py on the Python path. Easiest: place the DAG file in the same folder and import from there, or set PYTHONPATH in the DAG.
  5. Copy the DAG below into $AIRFLOW_HOME/dags/taxi_training_dag.py (standalone sets AIRFLOW_HOME=~/airflow by default).
Airflow runs tasks from the scheduler's environment — not your notebook kernel. If imports fail, fix PYTHONPATH or install deps into the same conda env Airflow uses.
10 · Approach A — Single-Task DAG

🌬️ Wrap the Existing Script (Start Here)

The fastest path: one task imports run() and derives train month from the DAG's logical date. Retries cover transient Parquet download failures without rewriting the script.

# dags/taxi_training_dag.py
from datetime import datetime

import pendulum
from airflow.decorators import dag, task

# adjust import path if duration_prediction.py lives elsewhere
from duration_prediction import run as train_pipeline


@dag(
    dag_id="taxi_duration_training",
    start_date=datetime(2021, 3, 1, tzinfo=pendulum.timezone("UTC")),
    schedule="@monthly",
    catchup=True,
    tags=["mlops", "nyc-taxi"],
    default_args={"retries": 2, "retry_delay": pendulum.duration(minutes=5)},
)
def taxi_duration_training():
    @task
    def train_and_log(**context):
        logical = context["logical_date"]
        train_dt = logical.subtract(months=2)
        run_id = train_pipeline(year=train_dt.year, month=train_dt.month)
        with open("run_id.txt", "w") as f:
            f.write(run_id)
        return run_id

    train_and_log()


taxi_duration_training()

Equivalent one-liner from the shell (what the DAG wraps):

python duration_prediction.py --year 2021 --month 1

Trigger manually for a specific interval:

airflow dags trigger taxi_duration_training --logical-date 2021-03-01
11 · Approach B — Multi-Task DAG

🔗 Split Steps for Granular Retries

When you want Airflow to retry only the download or only training, split the pipeline. Don't push DataFrames through XCom — they're too large. Pass file paths under a run-scoped temp directory instead.

load_train
→
load_val
→
prepare_features
→
train_and_log
→
save_run_id
import pickle
from pathlib import Path

import pendulum
from airflow.decorators import dag, task
from datetime import datetime

from duration_prediction import read_dataframe, create_X, train_model


def _run_dir(context) -> Path:
    p = Path("/tmp/taxi_training") / context["run_id"]
    p.mkdir(parents=True, exist_ok=True)
    return p


@dag(
    dag_id="taxi_duration_training_split",
    start_date=datetime(2021, 3, 1, tzinfo=pendulum.timezone("UTC")),
    schedule="@monthly",
    catchup=True,
    default_args={"retries": 2},
)
def taxi_duration_training_split():
    @task
    def load_train(**context):
        logical = context["logical_date"].subtract(months=2)
        df = read_dataframe(logical.year, logical.month)
        path = _run_dir(context) / "train.parquet"
        df.to_parquet(path)
        return str(path)

    @task
    def load_val(**context):
        logical = context["logical_date"].subtract(months=2)
        y, m = logical.year, logical.month
        next_y = y if m < 12 else y + 1
        next_m = m + 1 if m < 12 else 1
        df = read_dataframe(next_y, next_m)
        path = _run_dir(context) / "val.parquet"
        df.to_parquet(path)
        return str(path)

    @task
    def prepare_features(train_path, val_path, **context):
        import pandas as pd
        df_train = pd.read_parquet(train_path)
        df_val = pd.read_parquet(val_path)
        X_train, dv = create_X(df_train)
        X_val, _ = create_X(df_val, dv)
        out = _run_dir(context) / "matrices.pkl"
        with open(out, "wb") as f:
            pickle.dump({
                "X_train": X_train, "X_val": X_val,
                "y_train": df_train.duration.values,
                "y_val": df_val.duration.values,
                "dv": dv,
            }, f)
        return str(out)

    @task
    def train_and_log(matrices_path):
        with open(matrices_path, "rb") as f:
            data = pickle.load(f)
        return train_model(
            data["X_train"], data["y_train"],
            data["X_val"], data["y_val"],
            data["dv"],
        )

    @task
    def save_run_id(run_id: str):
        with open("run_id.txt", "w") as f:
            f.write(run_id)

    train_path = load_train()
    val_path = load_val()
    matrices = prepare_features(train_path, val_path)
    run_id = train_and_log(matrices)
    save_run_id(run_id)


taxi_duration_training_split()

Dependency chain: load_train and load_val can run in parallel; both must finish before prepare_features; training waits on prepared matrices.

12 · Scheduling & Backfill

📅 Monthly Runs for Past Months

With schedule="@monthly" and catchup=True, Airflow creates one DAG run per month from start_date forward. Each run trains on data two months before that run's logical date.

Logical date (DAG run)run(year, month)Train dataVal data (script: month+1)
2021-03-012021, 1Jan 2021Feb 2021
2021-04-012021, 2Feb 2021Mar 2021
2021-05-012021, 3Mar 2021Apr 2021

Backfill a range of months in one command:

airflow dags backfill taxi_duration_training \
  --start-date 2021-03-01 \
  --end-date 2021-06-01

After each successful run, open MLflow UI → nyc-taxi-experiment → find the new run → use run_id.txt (or the task return value) to register or promote the model in the registry.

Prerequisites checklist: MLflow on :5000 · green trip Parquet available for train/val months · models/ writable · same Python deps in the Airflow worker env.
