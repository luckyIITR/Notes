2 · The Three Deployment Modes

🔀 Do You Need Predictions Now?

The first question: do predictions need to be immediate, or can they wait an hour / day / week? The answer splits deployment into two families — and the online family splits again.

  • 📦

    Batch (offline)

    The model is not running all the time. A scoring job runs at regular intervals (every 10 min, hourly, daily, weekly), pulls the last interval's data, applies the model, and writes predictions somewhere.

  • 🌐

    Online — Web Service

    The model is up all the time, reachable over HTTP. A client sends a request and holds the connection open until it gets the prediction back. One-to-one, client–server.

  • 📡

    Online — Streaming

    Producers push events to a stream; one or many consumers listen and react. The producer doesn't know or care who consumes — an implicit, one-to-many (or many-to-many) relationship.

3 · Batch Mode — Concept

📦 The Scoring Job Pattern

A typical batch setup: a database holds the data; a scoring job (with the model inside) pulls the last interval's rows, applies the model, and writes results to another database. A downstream consumer — a report, say — reads the predictions and reacts.

Churn prediction is the classic batch use case. You don't need to know the instant a user decides to leave — a daily or weekly job identifies likely churners, and a marketing job sends them an incentive (a push, a reward).
4 · Web Service vs Streaming — Concept

🔗 Connection vs Events

The ride-duration model is a great web-service example: the app's backend asks the duration service for a prediction, the user sees "≈ 30 min", and decides whether to hire the taxi. The decision is needed immediately on screen — you can't wait five minutes.

Streaming is different. When the ride starts, the backend becomes a producer and pushes a ride_started event. Independent consumers react: a tip-prediction model, a more accurate duration model that updates the earlier estimate ("actually 20 min, not 30"), and so on.

AspectWeb ServiceStreaming
RelationshipOne-to-one (client ↔ server)One-to-many / many-to-many
ConnectionExplicit, kept alive until responseImplicit — producer fires and forgets
Producer knows consumers?YesNo (doesn't care who/how many)
Good forImmediate single decisionMany independent reactions to one event
Content moderation is the canonical streaming example: a video-upload event fans out to copyright, NSFW, and violence/hate-speech consumers, each pushing a verdict to a predictions stream that a decision service acts on. Adding a new check is just adding a new consumer — the system scales to arbitrarily many models.
5 · Web Service with Flask

🧪 Wrapping the Pickle in Flask

Module 1 left us a pickle file (lin_reg.bin) holding the DictVectorizer and the linear regression. The plan: isolated env with Pipenv → script → Flask app → Docker.

Pin the environment

Match the exact scikit-learn version used to create the pickle, or unpickling may fail. Pin Python too (3.9 to be safe):

pip freeze | grep scikit-learn        # e.g. scikit-learn==1.0.2
pipenv install scikit-learn==1.0.2 flask --python=3.9
pipenv shell

predict.py — load, prepare, predict, serve

import pickle
from flask import Flask, request, jsonify

with open('lin_reg.bin', 'rb') as f_in:
    (dv, model) = pickle.load(f_in)

def prepare_features(ride):
    features = {}
    features['PU_DO'] = '%s_%s' % (ride['PULocationID'], ride['DOLocationID'])
    features['trip_distance'] = ride['trip_distance']
    return features

def predict(features):
    X = dv.transform(features)
    preds = model.predict(X)
    return float(preds[0])

app = Flask('duration-prediction')

@app.route('/predict', methods=['POST'])
def predict_endpoint():
    ride = request.get_json()
    features = prepare_features(ride)
    pred = predict(features)
    return jsonify({'duration': pred})

if __name__ == "__main__":
    app.run(debug=True, host='0.0.0.0', port=9696)
Feature engineering must match training. The PU_DO interaction feature is built by string-concatenating the location IDs — do it identically in serving, or predictions will be wrong.

Production server + test client

Flask's built-in server warns it's for development only. Use gunicorn in production. Keep requests as a dev-only dependency since it's only needed for tests:

--system --deploy installs straight into the container's system Python (no nested virtualenv) using the locked versions. With a Docker image you can deploy anywhere Docker runs — Elastic Beanstalk, Kubernetes, etc.

6 · Web Service + MLflow Registry

🗂️ Fetching the Model Instead of Emailing It

Rather than shipping a pickle by email, pull the model from the MLflow registry by run_id. The naive version uses the client to download the vectorizer artifact separately:

import mlflow
from mlflow.tracking import MlflowClient

MLFLOW_TRACKING_URI = 'http://localhost:5000'
RUN_ID = 'abc123...'

mlflow.set_tracking_uri(MLFLOW_TRACKING_URI)
client = MlflowClient(tracking_uri=MLFLOW_TRACKING_URI)

path = client.download_artifacts(run_id=RUN_ID, path='dict_vectorizer.bin')
with open(path, 'rb') as f_out:
    dv = pickle.load(f_out)

logged_model = f'runs:/{RUN_ID}/model'
model = mlflow.pyfunc.load_model(logged_model)

Cleaner: bundle DV + model in a pipeline

Carrying the vectorizer as a separate artifact is ugly. Put both in a scikit-learn Pipeline at training time, then the registry stores one self-contained model:

from sklearn.pipeline import make_pipeline

pipeline = make_pipeline(
    DictVectorizer(),
    RandomForestRegressor(**params, n_jobs=-1)
)
pipeline.fit(dicts, y_train)
mlflow.sklearn.log_model(pipeline, artifact_path="model")

Serving then collapses to a single load_model — no separate vectorizer download.

Remove the tracking-server dependency

Setting set_tracking_uri makes serving depend on the tracking server being up — a problem when an autoscaled instance can't start because the server is down. Point directly at the S3 artifact location instead, and make the run ID configurable via an env var:

import os
RUN_ID = os.getenv('RUN_ID')

logged_model = f's3://mlflow-models-alexey/1/{RUN_ID}/artifacts/model'
model = mlflow.pyfunc.load_model(logged_model)
Returning the model_version (run ID) in the response is good practice — you always know which model produced a given prediction. In Kubernetes you just set a different RUN_ID env var to roll the model forward.
7 · Streaming with Kinesis + Lambda

📡 Serverless Stream Consumer (Optional)

AWS Kinesis is the event stream (like Kafka); AWS Lambda is the consumer — "run code without thinking about servers". The flow: ride-started events land on a Kinesis stream, Lambda reads them, applies the model, and pushes predictions to a second stream.

  1. Create an IAM role for Lambda with permission to read from Kinesis (and write CloudWatch logs); attach the policy.
  2. Create the Lambda function (Python 3.9) using the role; test with a sample event.
  3. Create the input Kinesis stream (ride_events, provisioned, 1 shard) and wire it as a Lambda trigger.
  4. Decode records — Kinesis delivers a list of records, each base64-encoded.
  5. Apply the model, then put results on the output stream (ride_predictions) with boto3.
import os, json, base64
import boto3, mlflow

kinesis_client = boto3.client('kinesis')
PREDICTIONS_STREAM_NAME = os.getenv('PREDICTIONS_STREAM_NAME', 'ride_predictions')
RUN_ID = os.getenv('RUN_ID')
TEST_RUN = os.getenv('TEST_RUN', 'False') == 'True'

model = mlflow.pyfunc.load_model(f's3://.../{RUN_ID}/artifacts/model')

def lambda_handler(event, context):
    predictions = []
    for record in event['Records']:
        encoded = record['kinesis']['data']
        decoded = base64.b64decode(encoded).decode('utf-8')
        ride_event = json.loads(decoded)

        ride = ride_event['ride']
        ride_id = ride_event['ride_id']
        features = prepare_features(ride)
        prediction = predict(features)

        prediction_event = {
            'model': 'ride_duration_prediction_model',
            'version': RUN_ID,
            'prediction': {'ride_duration': prediction, 'ride_id': ride_id},
        }
        if not TEST_RUN:
            kinesis_client.put_record(
                StreamName=PREDICTIONS_STREAM_NAME,
                Data=json.dumps(prediction_event),
                PartitionKey=str(ride_id),
            )
        predictions.append(prediction_event)
    return {'predictions': predictions}
Correlation IDs matter in streaming. Unlike a web service, the producer never sees the response — so every event carries a ride_id, and every prediction echoes the model name + version so downstream consumers know what produced it.

Packaging uses the AWS Lambda base image (from ECR Public), installs deps, and sets the handler. Push to ECR, then create a container-image Lambda with RUN_ID / PREDICTIONS_STREAM_NAME env vars. Grant the role S3 get/list so it can load the model, give it enough memory/timeout (first invocation is slow), and pin scikit-learn (and ideally cloudpickle) to avoid version-mismatch warnings.

FROM public.ecr.aws/lambda/python:3.9
RUN pip install -U pip
RUN pip install pipenv
COPY [ "Pipfile", "Pipfile.lock", "./" ]
RUN pipenv install --system --deploy
COPY [ "lambda_function.py", "./" ]
CMD [ "lambda_function.lambda_handler" ]
8 · Batch Scoring Script

📝 Notebook → Parameterized score.py

For batch, repurpose the registry notebook from training to applying the model. The use case: compare actual vs predicted duration for analytics (spotting traffic jams, driver deviation). Add an artificial ride_id (UUID) so each prediction is traceable.

import uuid
import pandas as pd
import mlflow

def generate_uuids(n):
    return [str(uuid.uuid4()) for _ in range(n)]

def read_dataframe(filename):
    df = pd.read_parquet(filename)
    df['duration'] = df.lpep_dropoff_datetime - df.lpep_pickup_datetime
    df['duration'] = df.duration.dt.total_seconds() / 60
    df = df[(df.duration >= 1) & (df.duration <= 60)].copy()
    df['ride_id'] = generate_uuids(len(df))
    return df

def apply_model(input_file, run_id, output_file):
    df = read_dataframe(input_file)
    dicts = prepare_dictionaries(df)
    model = load_model(run_id)
    y_pred = model.predict(dicts)

    df_result = pd.DataFrame()
    df_result['ride_id'] = df['ride_id']
    df_result['lpep_pickup_datetime'] = df['lpep_pickup_datetime']
    df_result['PULocationID'] = df['PULocationID']
    df_result['DOLocationID'] = df['DOLocationID']
    df_result['actual_duration'] = df['duration']
    df_result['predicted_duration'] = y_pred
    df_result['diff'] = df_result['actual_duration'] - df_result['predicted_duration']
    df_result['model_version'] = run_id

    df_result.to_parquet(output_file, index=False)

Parameterize input/output paths by taxi type, year, and month — pandas can read a URL directly, so no manual download:

def run():
    taxi_type = sys.argv[1]   # green
    year = int(sys.argv[2])   # 2021
    month = int(sys.argv[3])  # 4
    run_id = sys.argv[4]

    input_file = f'https://.../{taxi_type}_tripdata_{year:04d}-{month:02d}.parquet'
    output_file = f'output/{taxi_type}/{year:04d}-{month:02d}.parquet'
    apply_model(input_file=input_file, run_id=run_id, output_file=output_file)

9 · Batch Scoring with Mage

🧱 Orchestrating the Batch Job (No Video — You Already Know This)

There is no walkthrough video for this unit — it reuses what you learned about Mage in the orchestration module plus the score.py logic above. The idea: instead of running the script by hand, turn batch scoring into a Mage pipeline whose blocks pull the model from the MLflow registry and apply it.

  1. Connect to MLflow — point Mage at your tracking server / artifact store so blocks can reach the registry.
  2. Create a transformation block — the block that does the scoring work.
  3. Get the model from the registry and apply it — load by run_id (or stage), run predict, return the results.
load_data
→
transform · score
→
export predictions

A transformation block that fetches the model and scores the data:

import os
import mlflow

if 'transformer' not in globals():
    from mage_ai.data_preparation.decorators import transformer

MLFLOW_TRACKING_URI = os.getenv('MLFLOW_TRACKING_URI', 'http://localhost:5000')
RUN_ID = os.getenv('RUN_ID')

mlflow.set_tracking_uri(MLFLOW_TRACKING_URI)

@transformer
def transform(df, *args, **kwargs):
    # model is a pipeline (DictVectorizer + regressor) from the registry
    model = mlflow.pyfunc.load_model(f'runs:/{RUN_ID}/model')

    dicts = prepare_dictionaries(df)
    df['predicted_duration'] = model.predict(dicts)
    df['model_version'] = RUN_ID
    return df
Because the model is logged as a pipeline, the block doesn't need a separate vectorizer — connect to MLflow, load by run ID, and apply. The upstream data-loader block feeds df; a downstream exporter block writes predictions to Parquet / S3.
10 · Choosing a Mode

⚖️ Quick Decision Guide

Reach for batch when…
  • Predictions can wait (hours/days)
  • Work is periodic — marketing, churn, analytics
  • You want the simplest ops (a scheduled job)
Reach for online when…
  • Web service: one immediate decision per request (ride duration)
  • Streaming: one event, many independent reactions (moderation, recs)
  • Latency matters to the end user
