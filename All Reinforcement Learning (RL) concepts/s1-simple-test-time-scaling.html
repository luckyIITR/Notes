Test-Time Scaling · Reasoning · SFT

s1: Simple Test-Time Scaling

How 1,000 carefully curated examples and a decoding trick called budget forcing match o1-preview on reasoning benchmarks.

1 · The Problem

🧩 What Problem Are They Solving?

Language models have traditionally gotten better by training on more data with more compute — what's called train-time scaling. OpenAI's o1 introduced a new paradigm: test-time scaling, where the model uses extra compute at inference time to think longer and reason better. But OpenAI didn't publish their method, sparking a wave of replication attempts using complex techniques like Reinforcement Learning (RL), Monte Carlo Tree Search, and multi-agent systems.

The authors ask a pointed question: what is the simplest possible approach to achieve test-time scaling and strong reasoning?

Their answer: just 1,000 carefully chosen training examples and a trick called budget forcing.
2 · Data Curation — The s1K Dataset

🗄️ Starting Pool (59K Samples)

They collect 59,029 questions from 16 sources following three guiding principles:

  • Quality: Only well-formatted, clean datasets
  • Difficulty: Hard problems that require significant reasoning
  • Diversity: Questions from many different fields (math Olympiads, biology, CS, physics, economics, etc.)

For each question, they use the Gemini Flash Thinking API to generate a long reasoning trace and a solution — producing 59K (question, reasoning trace, answer) triplets.

🔽 Filtering Down to 1K

They apply three filtering stages:

Quality 59K → 51K Remove questions with formatting issues — ASCII art, broken image references, numbering errors.
Difficulty 51K → 24K Evaluate Qwen2.5-7B and Qwen2.5-32B on each question. Remove any question either model gets right — keeping only the genuinely hard ones. Reasoning trace length also serves as a difficulty proxy: harder problems require more thinking tokens.
Diversity 24K → 1K Classify questions into domains using the Mathematics Subject Classification (MSC) system. Sample one domain at a time (uniformly at random), pick one problem from that domain (favoring longer traces), repeat until exactly 1,000 samples spanning 50 domains.
A key finding: ~46% of the answers in s1K are actually wrong, but the model still learns well from them — suggesting the reasoning process matters more than the correct answer.
3 · Budget Forcing — The Core Technique

⏱️ Controlling How Long the Model Thinks

Budget forcing is a simple decoding-time trick to control how much the model "thinks." No retraining required.

  • To make the model think less: Append the end-of-thinking delimiter token early, forcing the model to stop reasoning and give its best current answer.
  • To make the model think more: Suppress the end-of-thinking token and instead append the word "Wait" to the model's current reasoning trace, encouraging more exploration. This can lead the model to double-check its answer, often fixing incorrect reasoning steps.
Figure 3 Example — "How many r's in raspberry?"
Model (first pass): …the answer is 2. [attempts to emit stop token]
Budget forcing: Suppresses stop token → appends Wait
Model (second pass): Let me re-examine… r-a-s-p-b-e-r-r-y… the answer is 3.

💡 Why "Wait" Works

Appending "Wait" triggers the model's trained pattern of self-reflection. It's not magic — the model has seen patterns like this in its training data where a second look yields a better answer. The key insight is that this is controllable: you can force the model to think more without retraining.

Budget forcing separates how long to think from the model's weights entirely — it's a pure inference-time intervention with no gradient required.

Standard supervised fine-tuning on the 1K examples — no RL, no reward model, no special training objective. The resulting model is called s1-32B.

🔀 Sequential vs. Parallel Scaling

Sequential
Budget Forcing
Later steps build on earlier ones — longer reasoning traces. Each "Wait" extends the chain of thought, allowing self-correction.
Parallel
Majority Voting
Run multiple independent attempts and take the most common answer. Scales compute without extending any single reasoning chain.

After training on 1,000 samples and equipping s1-32B with budget forcing, scaling test-time compute on the base model via majority voting does not catch up with s1-32B's performance — validating that sequential scaling is more efficient.

⚖️ Budget Forcing vs. Other Methods

Budget forcing wins on all metrics. Other methods fail in interesting ways:

  • Token-conditional control: The model can't reliably count tokens in its own output.
  • Step-conditional control: The model "hacks" the constraint by adjusting token count per step.
  • Rejection sampling: Shows inverse scaling — shorter generations tend to be where the model was on the right track from the start, so filtering for short outputs actually selects better answers paradoxically.
7 · Limitations & Future Directions

⚠️ Where the Method Falls Short

Promising Extensions
  • Combining budget forcing with tree search (REBASE) scales even further — hybrid sequential + parallel approaches are promising
  • Human experts could theoretically replace the Gemini teacher model for trace generation
  • Works well for any problem with a definite, verifiable answer
Current Limitations
  • Flattening: Budget forcing eventually plateaus; the context window constrains further scaling. Too many "Wait" appends leads to repetitive loops
  • Abstract tasks: Unclear how to apply to creative writing or open-ended tasks without a ground truth
  • Distillation dependency: Reasoning traces come from a powerful teacher (Gemini Flash Thinking) — access to a strong model is assumed
8 · The Big Picture

🌐 A Direct Challenge to RL-Heavy Pipelines

The paper's central thesis is elegantly simple: you don't need RL, you don't need millions of samples, you don't need complex search algorithms. Careful data curation (1K examples selected for quality, difficulty, and diversity) plus a trivial decoding intervention ("Wait") is enough to match or beat much more expensive approaches.

This is a direct challenge to the prevailing assumption that o1-level reasoning requires massive RL pipelines.

The fact that ~half the training answers are wrong and the model still learns is a profound insight — the model is learning how to think, not what to answer.
