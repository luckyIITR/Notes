Foundations · Implementation

How RL Code Is Structured

The shared backbone behind every RL algorithm — from the universal agent–environment loop down to tabular Q-Learning, DQN, and policy gradients, and how it all maps onto real project files.

Reinforcement Learning code almost always follows the same backbone: an agent interacts with an environment in a loop, collects experience, and uses that experience to improve a policy. Below is the structure, from the generic skeleton down to specific algorithms.

1 · The Universal RL Loop

🔁 The Core Every Algorithm Shares

Every RL solution, regardless of algorithm, has this core at its center:

Generic RL Skeleton
initialize environment
initialize agent (policy / value function with random parameters)

for each episode:
    state = environment.reset()
    done = False

    while not done:
        action = agent.select_action(state)        # decide
        next_state, reward, done = environment.step(action)  # act
        agent.store(state, action, reward, next_state, done) # remember
        agent.learn()                              # improve
        state = next_state

The four verbs are the heart of it. Different algorithms just fill in select_action and learn differently.

Decide
select_action()
Act
env.step()
Remember
store()
Improve
learn()
Master this loop once and every algorithm becomes a variation on the same theme — you only ever swap out how the agent decides and how it learns.
2 · Tabular Q-Learning

🎯 The Classic Starting Point

Used when states and actions are small and discrete. No neural network needed — the agent's entire "brain" is a lookup table.

Tabular Q-Learning
initialize Q[states][actions] = 0
set hyperparameters: alpha (learning rate), gamma (discount), epsilon (exploration)

for each episode:
    state = env.reset()
    while not done:
        # epsilon-greedy action selection
        if random() < epsilon:
            action = random_action()
        else:
            action = argmax(Q[state])

        next_state, reward, done = env.step(action)

        # Q-update (Bellman equation)
        best_next = max(Q[next_state])
        Q[state][action] += alpha * (reward + gamma * best_next - Q[state][action])

        state = next_state
3 · Deep Q-Network (DQN)

🧠 When States Are Large or Continuous

Replaces the Q-table with a neural network, plus two key tricks: a replay buffer and a target network.

Deep Q-Network
initialize Q_network with random weights θ
initialize target_network with weights θ⁻ = θ
initialize replay_buffer (fixed capacity)

for each episode:
    state = env.reset()
    while not done:
        action = epsilon_greedy(Q_network, state)
        next_state, reward, done = env.step(action)
        replay_buffer.add(state, action, reward, next_state, done)
        state = next_state

        # learn from a random batch (breaks correlation between samples)
        if replay_buffer.size > batch_size:
            batch = replay_buffer.sample(batch_size)

            # target uses the SEPARATE target network for stability
            target = reward + gamma * max(target_network(next_state)) * (1 - done)
            prediction = Q_network(state)[action]
            loss = (target - prediction)²

            gradient_descent_step(loss, θ)

    # periodically sync target network
    every N steps: θ⁻ = θ
Replay Buffer
  • Stores past transitions and samples random batches
  • Breaks temporal correlation between consecutive samples
  • Lets each experience be reused many times
Target Network
  • A frozen copy of the Q-network used to compute targets
  • Prevents the "moving target" instability
  • Synced to the online network every N steps
4 · Policy Gradient / REINFORCE

📈 Learn the Policy Directly

Instead of estimating values, this directly optimizes a policy network. The structure shifts to collect a full episode, then update.

REINFORCE
initialize policy_network πθ

for each episode:
    trajectory = []
    state = env.reset()

    # roll out a full episode
    while not done:
        action_probs = policy_network(state)
        action = sample(action_probs)
        next_state, reward, done = env.step(action)
        trajectory.append((state, action, reward))
        state = next_state

    # compute discounted returns, then update
    for t in trajectory:
        G_t = discounted_return_from(t)             # sum of future rewards
        loss = -log(π(action_t | state_t)) * G_t    # push up good actions
        gradient_descent_step(loss, θ)
The key contrast: value-based methods (Q-Learning, DQN) update after every step, while REINFORCE waits for a complete episode before it can compute returns and update.
5 · Mapping Onto a Real Project

🗂️ How the Pieces Become Files

A typical RL codebase organizes the above into files like:

Project Layout
environment.py    # reset(), step(action) -> (next_state, reward, done)
agent.py          # select_action(), learn(), the policy/value model
replay_buffer.py  # store and sample experience (for off-policy methods)
networks.py       # neural net definitions (for deep RL)
train.py          # the main loop that ties it together
config.py         # hyperparameters: alpha, gamma, epsilon, batch_size...

If you're using a framework, the environment side is usually Gymnasium — the reset/step interface is the de-facto standard. The agent side comes from libraries that implement DQN, PPO, SAC, and more, so you mostly write the environment and tune hyperparameters.

LibraryRole
GymnasiumEnvironment interface — reset() / step() standard
Stable-Baselines3Batteries-included agent implementations (DQN, PPO, SAC…)
CleanRLSingle-file, readable reference implementations
RLlibScalable, distributed RL for large workloads
Whichever path you take, the four verbs — decide, act, remember, improve — stay in place. Frameworks just hand you battle-tested versions of select_action and learn.
