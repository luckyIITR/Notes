Fine-tuning with ms-swift

A complete ground-up guide to ms-swift — what it is, how it works, and how to go from a base model to a fine-tuned, deployed LLM.

1 · What Is ms-swift?

🚀 The One-Stop LLM Training Framework

ms-swift (SWIFT — Scalable lightWeight Infrastructure for Fine-Tuning) is an open-source framework from the ModelScope community (Alibaba) for fine-tuning, aligning, and deploying large language models and multimodal models. It was published at AAAI 2025.

The core promise is a single unified framework that handles the entire lifecycle: pre-training continuation → supervised fine-tuning → human alignment → inference → deployment — without needing to wire together separate libraries for each stage.

ms-swift supports 600+ LLMs (Qwen3, DeepSeek-R1, LLaMA 4, GLM-5, InternLM3…) and 300+ multimodal models (Qwen3-VL, InternVL, LLaVA, Phi-4…) out of the box, with a single consistent CLI.

Install it:

Installation
# Standard install (recommended)
pip install ms-swift -U

# Full install with all optional extras
pip install 'ms-swift[all]' -U

# Add DeepSpeed for multi-GPU ZeRO training
pip install deepspeed -U
2 · Why ms-swift Exists

⚖️ The Problem It Solves

Before frameworks like ms-swift, fine-tuning an LLM meant cobbling together a stack of separate tools: Hugging Face Transformers for the model, PEFT for LoRA, TRL for DPO/PPO, DeepSpeed or FSDP for distributed training, vLLM for inference, and a custom evaluation pipeline on top. Each had its own API, data format, and configuration quirks.

The friction is especially bad for multimodal models (images + text), where there was previously no unified training standard across model families.

What ms-swift provides
  • One CLI, one data format, every training method
  • First framework with systematic multimodal fine-tuning support
  • Consistent model templates so you never worry about chat format mismatches
  • Post-training (inference, evaluation, quantization, deployment) built in
  • Web-UI for those who prefer no-code fine-tuning
What you still need to know
  • GPU hardware — ms-swift doesn't abstract away VRAM constraints
  • Dataset preparation — you curate and format your own training data
  • Hyperparameter choices — the framework runs what you tell it to
3 · The Three-Stage Training Ladder

🪜 Pre-training → SFT → Alignment

Most production LLM pipelines follow a three-stage ladder. ms-swift supports all three with a matching CLI command for each:

Stage 1swift pt
→
Stage 2swift sft
→
Stage 3swift rlhf
→
Deployswift deploy

Stage 1 — Continued Pre-training (swift pt)

You have a general base model (e.g. Qwen2.5-7B) but want it to absorb domain-specific text — medical literature, legal documents, code in a new language. swift pt runs a standard causal language modelling objective on your corpus, updating all (or a subset of) weights. No chat template is applied; the model just learns to predict next tokens.

Stage 2 — Supervised Fine-tuning (swift sft)

You have labelled (instruction, response) pairs and want the model to follow instructions. swift sft applies the model's chat template, formats the conversation correctly, and trains on your dataset — with LoRA, full parameters, or anything in between.

Stage 3 — RLHF / Alignment (swift rlhf)

You have an SFT checkpoint and want to improve its responses using human preference data or a reward signal. swift rlhf dispatches to the alignment method you choose — DPO, GRPO, PPO, KTO, and more.

Important order: always do SFT first on a base model before running RLHF. A base model has no concept of conversation; asking it to do DPO without SFT will produce garbage.
4 · Lightweight Tuning Methods

🪶 Training Without Touching Every Weight

Full fine-tuning updates every parameter in the model. For a 70B model that is tens of gigabytes of gradients and optimizer state per step — impractical on consumer hardware. Parameter-efficient fine-tuning (PEFT) methods solve this by training only a small subset of new parameters.

LoRA — Low-Rank Adaptation

LoRA is the most widely used PEFT method. Instead of updating the original weight matrix W, it learns two small low-rank matrices A and B and adds their product as a residual:

LoRA Weight Update
W' = W + α · (B · A)
W — frozen original weight matrix  ·  A — low-rank matrix (r × din), randomly initialised  ·  B — low-rank matrix (dout × r), zero-initialised  ·  α — scaling factor (lora_alpha / rank)  ·  r — rank (typical: 8–64)

Since r ≪ d, the number of trainable parameters drops dramatically. A 7B model with rank 8 LoRA trains ~0.1% of the total parameters instead of 100%.

QLoRA — Quantised LoRA

QLoRA takes LoRA further: the base model weights are quantised to 4-bit (using NF4 or other quantisation formats) before training begins, then LoRA adapters are trained in full precision on top. This lets you fine-tune a 7B model on a single GPU with just 9 GB of VRAM.

Other PEFT variants in ms-swift

LoRA QLoRA DoRA LoRA+ LoRA-GA RS-LoRA LLaMAPro LongLoRA ReFT Adapter LISA
MethodKey ideaWhen to pick it
LoRALow-rank residual adapters on attention/MLP layersDefault choice — fast, mergeable, works everywhere
QLoRA4-bit quantised base + LoRA adaptersVery limited VRAM; note: cannot merge into full weights
DoRADecomposes weight into magnitude + direction; updates bothWhen LoRA underfits; slightly higher cost

Rule of thumb: use LoRA by default. Only switch to QLoRA when VRAM is the hard constraint, and avoid QLoRA if you plan to serve the model with vLLM (QLoRA adapters can't be merged).
5 · RLHF & Alignment Methods

🧑‍🏫 Moving Beyond SFT

SFT teaches a model to imitate a dataset. RLHF methods go further: they use preference signals — either from human raters or from a reward model — to pull the model towards outputs humans actually prefer, not just outputs that appeared in training data.

The full menu of alignment methods in ms-swift:

PPO DPO GRPO KTO SimPO CPO ORPO Reward Model GKD
MethodData neededWhat it optimises
PPOReward model + promptsPolicy gradient with a learned reward; the classic RLHF pipeline
DPOPreference pairs (chosen, rejected)Directly trains on preference pairs — no reward model needed
GRPOPrompts + verifiable rewardsGroup-normalised reward baseline; DeepSeek-R1's training algorithm
KTOBinary good/bad labelsKahneman-Tversky optimisation; works without paired comparisons
SimPOPreference pairsLength-normalised DPO — avoids the need for a reference model
ORPOPreference pairsFuses SFT and preference tuning into one loss — no reference model
CPOPreference pairsContrastive preference optimisation
GKDTeacher model outputsGeneralised knowledge distillation — align student to teacher

Choosing an alignment method:

  • Have human preference data (chosen/rejected pairs)? → Start with DPO. It's simple, no reward model to train.
  • Want reasoning/thinking to emerge (like DeepSeek-R1)? → Use GRPO with a verifiable reward (e.g. math answer correctness).
  • Only have thumbs-up/thumbs-down labels, not pairs? → Use KTO.
  • Need a single-stage SFT + preference training? → Try ORPO.
  • Full classic RLHF pipeline? → PPO (but it's the most complex to set up).
6 · Distributed Training

🖥️ Scaling Across Multiple GPUs

Large models don't fit in a single GPU's memory. ms-swift supports four distributed strategies that trade memory savings against training speed:

StrategyWhat it shardsBest for
DDPNothing — each GPU holds the full modelModels that fit in one GPU; fastest throughput
device_mapModel layers split across GPUs sequentiallyModel is too big for one GPU; simplest to enable (automatic)
DeepSpeed ZeRO-2Optimizer states + gradientsReduces memory ~4× vs DDP; modest speed cost
DeepSpeed ZeRO-3Optimizer states + gradients + model parametersMaximum memory savings; significant speed cost
FSDP + QLoRAFull model sharded + quantised adapters70B models on consumer GPUs (e.g. two 3090s)
Example — ZeRO-3 on 4 GPUs
NPROC_PER_NODE=4 \
CUDA_VISIBLE_DEVICES=0,1,2,3 \
swift sft \
    --model Qwen/Qwen2.5-7B-Instruct \
    --dataset your-dataset \
    --deepspeed default-zero3 \
    --output_dir output

Megatron support is also available for very large scale — tensor parallelism, pipeline parallelism, sequence parallelism — via the swift megatron sub-commands. This is the approach used to train 70B+ parameter models across hundreds of GPUs.

For a single machine with multiple GPUs and a model that fits in one GPU: use DDP. If the model doesn't fit: enable device_map (happens automatically). If you still run out: add DeepSpeed ZeRO-3.
7 · CLI Workflow: Start to Finish

⌨️ The Four Commands You Actually Use

The entire ms-swift workflow maps to four CLI entry-points. Every command shares the same argument style and the same output directory structure.

Train — SFT with LoRA on a single GPU

swift sft (LoRA, single GPU)
CUDA_VISIBLE_DEVICES=0 \
swift sft \
    --model              Qwen/Qwen2.5-7B-Instruct \
    --dataset            AI-ModelScope/alpaca-gpt4-data-en \
    --tuner_type         lora \
    --lora_rank          8 \
    --lora_alpha         32 \
    --num_train_epochs   3 \
    --output_dir         output/

Train — RLHF with DPO

swift rlhf (DPO)
CUDA_VISIBLE_DEVICES=0 \
swift rlhf \
    --rlhf_type          dpo \
    --model              Qwen/Qwen2.5-7B-Instruct \
    --dataset            your-preference-dataset \
    --tuner_type         lora \
    --output_dir         output/

Train — GRPO (reasoning-style RL)

swift rlhf (GRPO)
CUDA_VISIBLE_DEVICES=0,1,2,3 \
swift rlhf \
    --rlhf_type          grpo \
    --model              Qwen/Qwen2.5-7B-Instruct \
    --dataset            your-math-dataset \
    --reward_funcs       accuracy format \
    --deepspeed          default-zero3 \
    --output_dir         output/

Inference — test your checkpoint

swift infer
CUDA_VISIBLE_DEVICES=0 \
swift infer \
    --adapters           output/vx-xxx/checkpoint-xxx \
    --infer_backend      transformers \
    --stream             true \
    --temperature        0 \
    --max_new_tokens     2048

# For full-parameter checkpoints, use --model instead of --adapters

Deploy — OpenAI-compatible server

swift deploy
CUDA_VISIBLE_DEVICES=0 \
swift deploy \
    --adapters           output/vx-xxx/checkpoint-xxx \
    --infer_backend      vllm \
    --served_model_name  my-model

# Now call it just like OpenAI's API:
# POST http://localhost:8000/v1/chat/completions
The --adapters folder contains a saved args.json with all training config. Swift reads it automatically — you never have to re-specify --model or --system when doing inference or deployment.
8 · Key Hyperparameters

🎛️ What to Tune and Why

ParameterDefault / TypicalWhat it controls
tuner_typeloraWhich PEFT method to use: lora, full, qlora, dora, …
lora_rank8Rank of the low-rank matrices. Higher = more capacity, more params. Common: 8, 16, 32, 64.
lora_alpha32Scaling factor. Effective learning rate scale = alpha / rank. Usually set to 2× rank.
lora_dropout0.0–0.05Dropout on LoRA layers. Small regularisation for short training runs.
learning_rate1e-4 (LoRA), 1e-5 (full)Step size. LoRA tolerates higher LR than full fine-tuning.
num_train_epochs2–5Number of passes through the dataset. Overfitting is common beyond 3 epochs on small datasets.
max_length2048Max token sequence length. Longer = more memory. Match to your data's average length.
gradient_checkpointingtrueRecompute activations during backward pass — saves memory at ~20% speed cost.
torch_dtypebfloat16Training precision. bfloat16 is standard on A100/H100. Use float16 on older GPUs.
rlhf_type—For swift rlhf: dpo, ppo, grpo, kto, simpo, orpo, cpo

ms-swift sets --gradient_checkpointing true by default to save memory. For maximum speed when VRAM is not the bottleneck, you can disable it with --gradient_checkpointing false.

10 · Multimodal Fine-tuning

🖼️ Images, Video, and Audio

ms-swift is the first open-source framework to provide systematic support for fine-tuning multimodal large language models (MLLMs). It supports three modalities — images, video, and audio — and handles the messy engineering (aligning tokenisers, managing visual encoders, balancing learning rates between the visual aligner and the language backbone).

Supported multimodal tasks:

  • Captioning — describe an image or video in natural language
  • VQA — visual question answering ("What object is in the red circle?")
  • OCR — extract text from images/documents
  • Grounding — localise objects mentioned in text (output bounding boxes)

A typical multimodal LoRA setup uses:

  • Full-parameter training for the visual encoder and aligner (few parameters, high impact)
  • LoRA for the language backbone (billions of parameters, keep frozen with adapters)
  • Separate, higher learning rate for the visual components
Fine-tuning multimodal models on custom data (e.g. your company's diagrams or medical scans) is one of the highest-leverage applications of ms-swift — the visual understanding transfers, you just teach it your domain's output format.
11 · Post-Training: Merging, Inference & Export

📦 What Happens After Training

Training produces a checkpoint directory. Here's the typical post-training flow:

  1. Merge LoRA adapters into the base model (if using LoRA, not QLoRA) so you have a single clean weight file. Use swift export --merge_lora true.
  2. Evaluate the merged model on your held-out test set using swift infer --val_dataset .... Check quality before deploying.
  3. Quantise (optional) the merged model to GPTQ or AWQ 4-bit to cut serving memory in half with minimal quality loss. Use swift export --quant_method gptq.
  4. Deploy via swift deploy with --infer_backend vllm for high-throughput serving. This exposes an OpenAI-compatible REST API at localhost:8000/v1.

The deployed server supports the full OpenAI Chat Completions API format, so any existing code that calls OpenAI can point to your local model with no other changes.

QLoRA trap: QLoRA adapters cannot be merged back into a full-precision model. If you want to use vLLM or export the model, use regular LoRA (not QLoRA) even if it requires slightly more VRAM during training.
12 · Strengths & Limitations

⚖️ When to Use ms-swift

Strengths
  • The broadest model coverage of any open-source fine-tuning framework (600+ LLMs, 300+ MLLMs)
  • Single CLI for the entire lifecycle — no glue code between stages
  • First-class multimodal training support with correct template handling per model family
  • Every alignment method (DPO, GRPO, PPO, KTO…) in one place
  • Web-UI available for quick experiments without writing any code
  • Active development — new models and methods land within days of paper release
  • AAAI 2025 peer-reviewed; used in dozens of published research papers
Limitations
  • Documentation is primarily in Chinese; English docs lag behind occasionally
  • QLoRA adapters cannot be merged — limits deployment options
  • PPO multimodal not yet supported
  • Large API surface can be overwhelming; fewer examples vs Hugging Face TRL for common cases
  • ModelScope ecosystem focus — some integrations assume ModelScope Hub over Hugging Face Hub

Fine-tune a 7B chat model on a custom datasetswift sft + LoRA on a single A100/3090
Reproduce DeepSeek-R1 style reasoning trainingswift rlhf --rlhf_type grpo with a verifiable reward function
Fine-tune a vision-language model on your imagesswift sft with multimodal dataset format
Align a model to preferences without a reward modelswift rlhf --rlhf_type dpo
Fit a 70B model on consumer GPUsFSDP + QLoRA or DeepSpeed ZeRO-3 + LoRA
