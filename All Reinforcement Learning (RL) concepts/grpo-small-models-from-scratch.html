Training Small LMs to Reason with GRPO from Scratch

End-to-end walkthrough — building a GRPO trainer by hand to teach 135M-parameter models step-by-step reasoning, with the policy-gradient math, the code internals, an SFT warm-up, results across model sizes, and ten implementation lessons.

1 · Why This Project Exists

🚀 Reasoning, But Tiny

Every major lab — DeepSeek, Google, OpenAI, Anthropic — is shipping a new reasoning model almost every month. These models first generate a chain of thought before producing the final answer, and that single change drastically raises accuracy on math and logic. The question this project answers is whether the same behaviour can be taught to a 135M-parameter model — roughly 1,000× smaller than GPT-3, and many orders of magnitude smaller than GPT-4.

The vehicle is GRPO (Group Relative Policy Optimization) — the exact algorithm DeepSeek used for the seminal R1 models — implemented from scratch instead of through a library. The training budget is roughly $5–6 of cloud GPU.

Deep RL is finicky and sensitive to implementation details. Small LMs are even more finicky because they lack the world knowledge of larger ones. The recipe in this page is the result of digging through papers, reading open-source code, and burning through GPU credits until things actually worked.

Roadmap of this page:

  1. Set up text-based RL tasks using the reasoning-gym library.
  2. Visually unpack how GRPO works and walk through the policy-gradient objective term by term.
  3. Implement experience collection, advantages, log-probabilities, and the GRPO loss in code.
  4. Add a small supervised fine-tuning warm-up and look at results across model sizes.
  5. Ten practical takeaways covering hyperparameters and implementation gotchas.
2 · The Tasks — Syllogism & Propositional Logic

🧩 Two Logic Puzzles

Two simple-but-non-trivial reasoning tasks are used to fine-tune and evaluate the models:

  • Syllogism. A deductive-reasoning puzzle. The model is given two premises and a candidate conclusion, then must answer Yes or No based on whether the conclusion logically follows.
  • Propositional logic. A symbolic-reasoning task. The model is given symbols and rules and must generate the correct conclusion. This is harder than syllogism — it is not a binary choice.

Both tasks come from the reasoning-gym library, which makes generating prompts and validating answers a one-liner:

Generate a propositional-logic dataset from reasoning_gym import create_dataset dataset = create_dataset(name="propositional_logic", size=200)

Reasoning-gym also provides a scoring function — given the model's response and the ground-truth metadata, it returns a numeric reward. That removes the need for a separately trained reward model.

3 · The Output Format — Think + Answer Tags

🏷️ A Structured Response

To give the model a clean signal of where the reasoning ends and the answer begins, the system prompt asks it to wrap thinking in <think>…</think> and the final answer in <answer>…</answer>. The expected output for any task looks like this:

Expected response format
<think>
step-by-step reasoning here…
</think>
<answer> final answer here </answer>

Pulling the answer out of the response is a tiny regex:

Extract answer from XML def extract_answer(response): answer = re.search(r"<answer>(.*?)</answer>", response, re.DOTALL) if answer is not None: return answer.group(1).strip() return ""

The full reward later combines two things: a small format reward (does the response have the four tags?) and a much larger correctness reward (is the extracted answer right?).

4 · Big Model vs. Small Model — The Gap

🆚 What 4.1 Mini Solves That a 135M Model Cannot

Plugging GPT-4.1-mini into this loop via the OpenAI API works on the first try — it produces a clean think/answer trace and the score function returns a reward of 1.0. The same prompt fed to SmolLM-135M-Instruct via Hugging Face transformers fails on two dimensions at once:

  • The reasoning is incorrect.
  • The XML format is broken — the <answer> tags often don't appear, so the regex returns an empty string.
The second failure is what RL is being asked to fix first. Without correct formatting there is nothing to score, and without something to score there is no learning signal.
5 · GRPO at a Glance — Two Phases in a Loop

🔁 Collect, Then Train

GRPO alternates between two phases. During experience collection the current LLM acts in the environment and records what happened. During training the LLM's weights are updated to increase the probability of high-reward behaviour.

Phase 1

Experience Collection

  • Sample a batch of questions
  • Generate G diverse responses per question
  • Score each response with the environment
  • Compute group-relative advantages
  • Save (sequence, mask, advantage, log-probs) into a buffer
Phase 2

Training

  • Sample a mini-batch of experiences
  • Recompute current log-probs through the LLM
  • Maximize the GRPO policy-gradient objective
  • Backprop, optimizer step, zero grad
  • Refill the buffer and repeat

The group-relative bit is what gives GRPO its name. For each question, multiple responses are generated, and the advantage of each response is its reward standardised against the other responses to the same question. There is no learned critic — the group itself acts as the baseline.

6 · The Experience Tuple

📦 What Gets Saved Per Response

Every entry in the experience buffer contains exactly four things:

Field 1
Full sequence
System prompt + user question + assistant response, as token IDs.
Field 2
Response mask
Binary mask marking which tokens were generated by the LLM. Loss is only applied here.
Field 3
Group-relative advantage
Standardised reward — positive if better than peers in the same group, negative if worse.
Field 4
Old log-probs
Per-token log-probabilities under the policy that produced the response. Used in the importance-sampling ratio.

Field 4 is the subtle one: by the time training runs, the LLM has already changed, so its current probabilities differ from the ones at generation time. Saving the old log-probs lets the loss correct for that drift.

7 · The GRPO Objective — Term by Term

🧮 Don't Be Scared of the Equation

The full GRPO objective being maximised, per question q with a group of G sampled responses oi:

GRPO Policy-Gradient Objective
𝒥(θ) = (1/G) Σi=1G (1/|oi|) Σt=1|oi| min( ri,t(θ) · Âi,t ,  clip(ri,t(θ), 1−ε, 1+ε) · Âi,t )
ri,t(θ) = πθ(oi,t | q, oi,<t) ÷ πθold(oi,t | q, oi,<t) — importance-sampling ratio  ·  Âi,t — token-level advantage (broadcast from the response advantage)  ·  ε — clip range, typically 0.2  ·  πθ / πθold — current vs. data-collection policy

Reading the equation in pieces:

  • The two summations are just a nested loop — outer over the G responses in the group, inner over the tokens in each response. The leading 1/G · 1/|oi| turns those sums into means.
  • πθ(oi,t | q, oi,<t) is the current model's probability of token t in response i, given the question and all earlier tokens — exactly what the LLM outputs autoregressively.
  • Âi,t is the simplest possible token-level advantage: the response's group-relative advantage, broadcast to every token in that response. Every token is held equally responsible for the final reward.
  • The ratio ri,t compares the current policy to the one that collected the data. As the model trains, this ratio drifts away from 1.
  • The clip + min is a regulariser borrowed from PPO. It refuses to credit changes larger than ±ε in the direction that would help the loss — preventing destructively large updates.
Maximising 𝒥(θ) means: raise the log-probabilities of tokens with positive advantage and lower the log-probabilities of tokens with negative advantage — but only by a clamped amount per step.
8 · Building Intuition With a Toy Example

🔬 Two Sequences, ABC and DEF

Imagine the policy generated two responses, ABC and DEF. The environment scored ABC high (advantage +1) and DEF low (advantage −1). Each token has some current probability under πθ. The objective is the mean of πθ(token) · advantage(token) over all tokens.

  • Tokens in ABC get coefficient +1. To maximise the objective, the model wants their probabilities to go up.
  • Tokens in DEF get coefficient −1. To maximise, their probabilities must go down.

Advantages and rewards are fixed — they came from the environment. The only thing the optimiser can change is the policy. So gradient ascent on this objective is exactly: push log-probs of good tokens up, push log-probs of bad tokens down.

9 · Code — Generating Diverse Responses

🎲 Many Rollouts Per Question

Each batch of input prompts is left-padded so the tensors are rectangular, then passed to llm.generate with num_return_sequences = n_rollouts so the model emits multiple diverse responses for the same question.

grpo_utils.py — sample G responses per prompt def generate_responses(llm, input_ids, attention_mask, eos_token_id, n_rollouts=5, max_new_tokens=100, top_p=0.95, temperature=1.0): return llm.generate( input_ids=input_ids, attention_mask=attention_mask, max_new_tokens=max_new_tokens, do_sample=True, top_p=top_p, num_return_sequences=n_rollouts, temperature=temperature, eos_token_id=eos_token_id, pad_token_id=eos_token_id, )

Sampling knobs that shape the diversity of the rollouts:

  • do_sample=True — switch from greedy decoding to probabilistic sampling.
  • top_p=0.95 — nucleus sampling. Restrict to the smallest set of tokens whose cumulative probability ≥ 0.95. Lower this for safer outputs, raise toward 1.0 for more variety.
  • temperature — flattens or sharpens the logits before sampling. High temperature → more diversity; low temperature → near-greedy.
  • num_return_sequences — the group size G. Each prompt is duplicated and decoded in parallel.

With batch_size=2 and n_rollouts=3, the result tensor has shape (6, prompt_len + max_new_tokens). The first three rows are responses to question 1, the next three to question 2.

The whole rollout runs under torch.no_grad() — no gradients are needed during data collection, and skipping them saves a lot of compute and memory.

10 · Code — Rewards (Format + Correctness)

🏆 Two Reward Components

The reward function is a weighted blend: a tiny push for following the XML format and a much larger push for actually getting the answer right. The format reward exists primarily to bootstrap learning when the model can't yet answer correctly — it ensures the gradient signal is non-zero on responses that at least look like reasoning.

grpo_utils.py — format reward (shape only) FORMAT_REWARD_WEIGHT = 0.15 CORRECTNESS_REWARD_WEIGHT = 0.85 def calculate_format_reward(response): if ("<answer>" not in response and "</answer>" not in response and "<think>" not in response and "</think>" not in response): return -1 fr = 0 if "<think>" in response: fr += 0.15 if "</think>" in response: fr += 0.15 if "<answer>" in response and "</answer>" in response: return fr + 0.7 return fr
grpo_utils.py — combined reward def calculate_rewards(batch_responses, validation_objects): format_rewards = np.array([calculate_format_reward(r) for r in batch_responses]) correctness_rewards = np.array([ calculate_correctness_reward(extract_answer(r), v) for v, r in zip(validation_objects, batch_responses) ]) return (FORMAT_REWARD_WEIGHT * format_rewards + CORRECTNESS_REWARD_WEIGHT * correctness_rewards)

For a batch of two questions × three rollouts, the reward tensor reshapes into (2, 3). A typical row looks like [1.0, 0.15, 1.0] — a wrong answer that did follow the format still receives a small reward, while a totally malformed response gets a strong negative signal.

Correctness uses reasoning-gym's get_score_answer_fn. The validator is case- and format-sensitive — for the syllogism task, exactly "Yes"/"No" scores 1.0; "yes" scores 0.0. Worth knowing when debugging.
11 · Code — Group-Relative Advantages

📊 Standardise Within Each Group

Once each response has a scalar reward, the rewards are reshaped to (num_questions, G) and standardised per-row to zero mean and unit variance:

Group-Relative Advantage
Âi = ( ri − mean(rgroup) ) ÷ std(rgroup)
ri — reward for response i in the group  ·  rgroup — vector of rewards for all G responses to the same question

Two consequences of this design that matter in practice:

  • The mean across each row is exactly 0 and the std is roughly 1. No critic is needed — the group itself is the baseline.
  • If every response in a group has the same reward, the advantage is 0 for all of them. Including the cases "all correct" and "all wrong". Those groups produce no training signal at all.
That second point is the central failure mode of GRPO on small models. If the model is too weak, every rollout is wrong → all rewards equal → all advantages zero → nothing to learn. The fix (covered in §16) is a supervised warm-up so that some rollouts are correct.
12 · Code — Per-Token Log-Probabilities

📐 Picking Out the Tokens That Were Actually Generated

At training time the loss needs the log-probability of each generated token under the current policy. The trick is that logits is a giant tensor of shape (batch, seq_len, vocab_size) — for SmolLM, vocab_size = 49152. Storing every vocabulary entry's log-prob is wasteful; the only ones that matter are the log-probs of the tokens that were actually emitted.

grpo_utils.py — selected log-probs via gather def calculate_logits(llm, full_responses, attention_mask): logits = llm(input_ids=full_responses, attention_mask=attention_mask).logits log_probs = torch.log_softmax(logits, dim=-1) # (B, T, V) selected = torch.gather( input=log_probs, dim=2, index=full_responses.unsqueeze(-1), # (B, T, 1) ).squeeze(-1) # (B, T) return selected

Step by step:

  1. Forward pass: get logits of shape (B, T, V).
  2. Apply log_softmax on the last dim → log-probabilities over the entire vocab at every step.
  3. Use gather to pick — at each (b, t) — the log-prob corresponding to the token ID that was actually generated at that position. Result shape: (B, T).

Higher (closer to 0) means the model was more confident about that token. These per-token log-probs are saved during exploration as old_log_probs and recomputed during training as log_probs — the ratio of the two enters the GRPO objective.

13 · Code — The GRPO Loss

⚙️ Clipped Surrogate, Three Reductions

With log-probs old and new, advantages, and a response mask in hand, the loss is short:

grpo_utils.py — clipped objective def calculate_grpo_loss(log_probs, old_log_probs, response_mask, advantages, clip_epsilon=0.2, loss_implementation="grpo"): importance_sampling_ratio = torch.exp(log_probs - old_log_probs) clipped = importance_sampling_ratio * advantages unclipped = torch.clamp(importance_sampling_ratio, 1 - clip_epsilon, 1 + clip_epsilon) * advantages loss = -torch.min(clipped, unclipped) loss = loss * response_mask # mask out prompt tokens if loss_implementation == "grpo": rmsum = response_mask.sum(dim=1).clamp(min=1.0) return (loss.sum(dim=1) / rmsum).mean() # original GRPO elif loss_implementation == "dr_grpo": return loss.sum() / MAX_TOKENS # DAPO/Dr.GRPO elif loss_implementation == "bnpo": return loss.sum() / response_mask.sum().clamp(min=1.0)

Three things happen here:

  • Importance ratio. exp(log_probs − old_log_probs) = πθ/πold, computed elementwise per token.
  • Clipped surrogate. Take the elementwise minimum of ratio · A and clip(ratio, 1−ε, 1+ε) · A. The minus sign turns this maximisation into a loss.
  • Reduction. Three options trade off length-bias differently — the original GRPO mean-over-tokens-then-mean-over-responses, Dr.GRPO's divide-by-MAX_TOKENS, and BNPO's pooled mean over all valid tokens. The default "grpo" just works for these tasks.
14 · Why Clipping? — A Worked Walk-Through

✂️ Bounding How Much the Network Can Move

Inside a single mini-batch the model's weights are updated several times, so by the third or fourth step its log-probs can drift far from the old policy. Pick the worst-case token where log_probs − old_log_probs is large; the unclipped ratio explodes, the loss blows up, and the gradient is gigantic.

The clip term restricts ratio ∈ [1−ε, 1+ε] — typically [0.8, 1.2]. Below is what the loss looks like for two illustrative tokens:

old_lpnew_lpratioclip(ratio)Â−min(ratio·Â, clip·Â)

−4.0−4.10.900.90+1.0−0.90
−16.5−16.01.651.20−1.0+1.65

Read the rows:

  • Row 1 — positive advantage, ratio explodes. Without clipping the loss would be −1.65 (pulling new log-prob even further up); clipping caps it at −1.20.
  • Row 2 — small drift. Clipped and unclipped agree; the loss flows normally.
  • Row 3 — negative advantage, ratio explodes. Now the min picks the larger (less negative) loss, which is the unclipped term 1.65 — clipping doesn't protect the model from being pushed in the harmful direction. This asymmetry is intentional: trust-region behaviour where bad updates can still propagate gradient.
Always remember the sign: minimising the loss = maximising π·Â. Positive advantage → push log-prob up. Negative advantage → push it down. Clipping just bounds how aggressive a single step can be.
15 · Code — The Outer Training Loop

🔄 Buffer, Train, Empty, Repeat

Stitching it all together, the main loop is just the two-phase rhythm wrapped in a buffer:

  1. Sample a batch of questions from the dataloader.
  2. Call collect_experiences — generate G rollouts per question, score, compute advantages, save log-probs and masks. Append the new experiences to the buffer.
  3. If the buffer has reached its target size, call train: shuffle, mini-batch, run train_on_batch several times, then empty the buffer.
  4. Loop back to step 1.

Inside train_on_batch:

  • Left-pad all sequences in the mini-batch to the same length.
  • Run the LLM forward to get current log-probs (log_probs) — the old log-probs were already stored.
  • Call calculate_grpo_loss(log_probs, old_log_probs, response_mask, advantages).
  • accelerator.backward(loss) → optimizer.step() → optimizer.zero_grad().

Watching the loss across a few consecutive train steps on the same buffer shows the loss steadily decreasing — 1.15 → 0.8 → 0.7 → 0.6 — confirming the policy is moving toward the high-advantage tokens.

16 · Visualising the Effect on Log-Probs

📈 Blue = After Training, Orange = Before

Plotting the per-token log-probs of the last 25 generated tokens before and after a few rounds of training on a single buffer makes the optimiser's behaviour visible:

  • For a response that received a high advantage — say one that ended with the correct <answer> Yes </answer> — the blue (after) bars are shorter than the orange (before) bars on the negative axis. Log-probs have moved closer to zero. The model is more likely to emit the same tokens next time.
  • For a response that received a negative advantage, the blue bars are longer than the orange — log-probs went further negative, lowering the chance of regenerating those tokens.

That is gradient ascent on the GRPO objective in action: every passing batch, good-token probabilities rise and bad-token probabilities fall.

17 · The Cold-Start Problem & SFT Warm-Up

🥶 RL Needs a Non-Zero Signal

On a base 135M model, raw GRPO simply does not work — every rollout is malformed or wrong, every advantage is zero, and there is nothing for the policy gradient to optimise. The fix is a small dose of supervised fine-tuning before RL begins.

Step 1
Generate SFT data
Prompt GPT-4.1-mini for ~200 traces with the exact <think>/<answer> format
→
Step 2
LoRA SFT
Attach LoRA adapter to the 135M base, train cross-entropy on the traces
→
Step 3
GRPO RL
Continue training on top of the SFT'd model with the loop in §15

LoRA (Low-Rank Adaptation) keeps the base model frozen and adds small trainable matrices alongside it — cheaper memory, less risk of corrupting the base, and easy to swap on/off. After ~200 SFT examples the 135M model already follows the format and gets the syllogism answer right roughly 45% of the time. That is enough non-zero signal for GRPO to take over.

SFT teaches the format and a baseline level of competence. RL pushes past that floor by sharpening the reasoning over many rollouts. Doing them in this order is the difference between training and silence.

19 · Loss Variants — Vanilla GRPO vs. Dr.GRPO vs. BNPO

🧪 Three Ways to Aggregate the Loss

The original DeepSeek-Math GRPO paper observed that response lengths grew during training and read this as a sign of evolving intelligence. Follow-up work (DAPO, "Critical Perspectives on R1") argued that this is partly a length bias baked into the loss, and proposed alternative reductions:

VariantReductionLength bias
GRPO (vanilla)mean over tokens within a response, then mean over responsesImplicit upweighting of long responses
Dr.GRPOsum of (loss · mask) divided by a fixed MAX_TOKENSRemoved — fixed normaliser
BNPOsum of (loss · mask) divided by total valid tokens in the batchPooled across the batch

The DAPO paper also recommends removing the KL penalty that the original GRPO formulation kept against a reference model. The argument: reasoning policies should drift far from the base policy — the KL term is an unnecessary regulariser on small, simple tasks. The implementation in this page omits it for that reason.

For the syllogism / propositional-logic experiments here, vanilla GRPO worked perfectly. The other variants are worth keeping in the toolbox for harder tasks where length-bias artefacts start to dominate.
20 · Ten Practical Takeaways

🛠️ Lessons From the Trenches

These are the gotchas that make or break a small-model GRPO run. Skim them before starting, revisit them when training stalls.

  1. Not every task is a fit for every model. Small models lack the world knowledge of 14B+ models. Check the base accuracy on the target task first. If it is below 10–20%, the task is too hard — invest in supervised warm-up before any RL.
  2. SFT is absurdly sample-efficient on SLMs. A few hundred high-quality examples can move accuracy by 30–40 points. In many practical cases SFT alone is enough and RL is overkill. Measure first.
  3. Reasoning research is still in flux. The original GRPO paper interpreted growing response length as emerging intelligence; DAPO and "Critical Perspectives on R1" challenged that and proposed Dr.GRPO and BNPO loss reductions. Be ready to swap losses if length pathologies appear.
  4. The KL-against-reference term can be dropped. DAPO argues — and this implementation agrees — that on reasoning tasks the policy should diverge from the base model, so the KL regulariser is an unnecessary brake on intelligence.
  5. Diverse responses are the engine of RL. If every rollout in a group has the same reward, advantages collapse to zero and the gradient vanishes. Track the per-batch standard deviation of rewards. If it is low, raise temperature, raise num_return_sequences, or add more reward components.
  6. Drop zero-advantage experiences. Filtering out experiences whose advantages are near zero saves gradient steps and lets the optimiser focus on the contrastive cases. Several recent papers report consistent gains from this trick.
  7. Stability comes from compute, not tricks. The reliable knobs for stabilising training all cost more compute: more rollouts per group, larger replay buffer, smaller learning rate. They almost always work.
  8. Audit the SFT dataset's tokenisation. Trivial-looking choices (whitespace inside <answer> tags, trailing newlines, punctuation) change how the tokenizer splits sequences. Consistent formatting between SFT and RL makes the model's life dramatically easier.
  9. The system prompt is a first-class lever. Adding stricter format instructions or a couple of few-shot examples in the system prompt itself often beats spending more compute on training. Don't treat the prompt as fixed.
  10. Always evaluate during training and snapshot the best checkpoint. Once the model starts solving most questions, advantages flatten and gradients shrink — training can oscillate in place. Periodic eval on a held-out set captures the real best state. Anneal the learning rate, lower the clip ratio, and freeze LoRA params toward the end of training to keep things stable in the final stretch.

22 · Strengths & Limitations of This Recipe

⚖️ What It Is and Isn't For

Strengths
  • Cheap — sub-$10 cloud GPU runs end-to-end
  • From-scratch implementation makes every detail inspectable
  • Pairs cleanly with reasoning-gym for new task types
  • SFT + GRPO recipe transfers across SmolLM and Qwen sizes
  • No critic, no reward model, no KL term — minimal moving parts
Limitations
  • Cold start fails without SFT — pure RL is a dead end on 135M
  • Single-task training caps the model's general reasoning
  • Sensitive to tokenisation, prompt format, and sampling temperature
  • Vanilla GRPO has length bias — surfaces on harder tasks
  • Reward design (format vs. correctness weights) needs per-task tuning
