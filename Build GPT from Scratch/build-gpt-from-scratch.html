Let's Build GPT From Scratch

A complete walkthrough of building a character-level GPT — from raw Shakespeare text to a working decoder-only Transformer — covering tokenization, batching, the bigram baseline, the self-attention "math trick", multi-head attention, residual connections, LayerNorm, and how it all relates to ChatGPT.

1 · What We Are Building

💬 From ChatGPT to a Language Model

ChatGPT lets you give an AI text-based tasks, and it replies by generating words left to right, one after another. Ask it the same prompt twice and you get slightly different answers — it is a probabilistic system, so for any one prompt there are many possible completions.

Under the hood it is a language model: it models the sequence of words (or characters, or tokens) and knows how they tend to follow each other. From its point of view, it is simply completing a sequence — you give it the start, it produces the rest.

The neural network doing the heavy lifting is the Transformer, proposed in the 2017 paper "Attention Is All You Need". GPT stands for Generatively Pre-trained Transformer. The architecture was introduced for machine translation, but with minor changes it took over the rest of AI in the years after.

We will not reproduce production ChatGPT (trained on much of the internet, with extensive pre-training and fine-tuning). Instead we build a small but educational character-level Transformer language model trained on a toy dataset, and watch it generate infinite (fake) Shakespeare.

Roadmap of this page:

  1. Load the data and build a character-level tokenizer.
  2. Chunk the data into batches with a block size (context length).
  3. Train a bigram baseline, evaluate the loss, and generate.
  4. Learn the matrix-multiply "math trick" behind self-attention.
  5. Build one self-attention head, then multi-head attention.
  6. Add feed-forward layers, residual connections, LayerNorm, and Dropout.
  7. Scale up the full decoder-only Transformer and relate it to ChatGPT.
2 · The Data — Tiny Shakespeare

📜 One Megabyte of Shakespeare

We use the tiny Shakespeare dataset — a concatenation of all of Shakespeare's works in a single ~1 MB file, roughly 1 million characters. The model's job is to learn how the characters follow each other: given some context, predict the character that comes next.

Load the raw textwith open('input.txt', 'r') as f: text = f.read() print(len(text)) # ~1,000,000 characters print(text[:1000]) # first 1000 chars of Shakespeare

The set of unique characters that occur in the text becomes our vocabulary. Sorting them gives a fixed ordering, and counting them gives the vocabulary size — 65 for this dataset (a space, some special characters, and upper/lowercase letters).

3 · Tokenization

🔤 Turning Text Into Integers

To tokenize means to convert raw text into a sequence of integers according to some vocabulary. Because we work at the character level, we simply map each character to an integer and back. We build a lookup table both ways and an encoder / decoder pair.

Character-level encode / decodestoi = { ch:i for i,ch in enumerate(chars) } itos = { i:ch for i,ch in enumerate(chars) } encode = lambda s: [stoi[c] for c in s] # string -> list[int] decode = lambda l: ''.join([itos[i] for i in l]) # list[int] -> string print(encode("hi there")) # [46, 47, ...] print(decode(encode("hi there"))) # "hi there"

This is only one of many possible schemes. In practice people use subword tokenizers — Google's SentencePiece, or OpenAI's tiktoken (a byte-pair-encoding tokenizer used by GPT). These trade off codebook size against sequence length:

TokenizerLevelVocab size"hi there" →
Ours (char)Character658 integers (0–64)
tiktoken (GPT-2)Subword~50,2573 integers
Large vocabularies give short sequences; small vocabularies give long sequences. Character level keeps the tokenizer dead simple (tiny codebook, trivial encode/decode) at the cost of longer sequences — perfect for learning.
4 · Train / Validation Split

✂️ Encode Everything, Then Split

We encode the entire dataset and wrap it in a PyTorch tensor — the whole text becomes one very long sequence of integers. Then we hold out the last 10% as a validation split to monitor overfitting: we want a model that produces Shakespeare-like text, not one that memorizes the exact training file.

Tensor + 90/10 splitimport torch data = torch.tensor(encode(text), dtype=torch.long) n = int(0.9 * len(data)) train_data = data[:n] val_data = data[n:]
5 · Chunks, Block Size & Batches

🧱 The Time Dimension

We never feed the entire text into the Transformer at once — that would be far too expensive. Instead we sample random chunks of a maximum length called the block size (also known as context length). A chunk of block_size + 1 characters secretly contains block_size training examples, because every prefix predicts the next character.

Eight examples hidden in nine charactersblock_size = 8 x = train_data[:block_size] # inputs y = train_data[1:block_size+1] # targets (offset by one) for t in range(block_size): context = x[:t+1] target = y[t] print(f"when input is {context.tolist()} the target is {target}")
Training on contexts from length 1 up to block_size is not just for efficiency — it teaches the Transformer to predict given any amount of context. That matters at generation time, when we can kick off sampling with a single character.

📦 The Batch Dimension

For GPU efficiency we stack many chunks into a single tensor — the batch dimension. These chunks are processed completely independently; they never talk to each other. We grab batch_size random offsets and stack the chunks into a batch × time tensor.

get_batch — sampling random chunksbatch_size = 4 # independent sequences per pass block_size = 8 # max context length def get_batch(split): data = train_data if split == 'train' else val_data ix = torch.randint(len(data) - block_size, (batch_size,)) x = torch.stack([data[i : i+block_size ] for i in ix]) y = torch.stack([data[i+1 : i+block_size+1] for i in ix]) return x, y # both shape (4, 8) -> 32 examples total

A 4 × 8 input tensor therefore packs 32 independent examples. The targets y are fed in at the very end to compute the loss — they give the correct next character for every position in x.

6 · The Bigram Baseline

🎲 The Simplest Possible Model

The simplest language model is the bigram model: predict the next character from the identity of the current one alone — no context, tokens don't see each other. We implement it with a single nn.Embedding table of shape vocab_size × vocab_size. Each input integer plucks out a row, which we read directly as the logits (scores) for the next character.

Bigram language modelimport torch.nn as nn from torch.nn import functional as F class BigramLanguageModel(nn.Module): def __init__(self, vocab_size): super().__init__() self.token_embedding_table = nn.Embedding(vocab_size, vocab_size) def forward(self, idx, targets=None): logits = self.token_embedding_table(idx) # (B, T, C) if targets is None: loss = None else: B, T, C = logits.shape logits = logits.view(B*T, C) # reshape for cross_entropy targets = targets.view(B*T) loss = F.cross_entropy(logits, targets) return logits, loss

PyTorch arranges the output as a Batch × Time × Channel tensor (here 4 × 8 × 65). The loss is the negative log likelihood, implemented as cross_entropy. One subtlety: cross-entropy wants channels as the second dimension, so we reshape (B,T,C) into (B·T, C) and the targets into (B·T,).

Expected initial loss
−ln(1 / vocab_size) = −ln(1 / 65) ≈ 4.17
A random model should score about 4.17. We get 4.87 at init — the predictions are not perfectly diffuse, but we can already evaluate the loss correctly.
7 · Generation

🔮 Sampling One Character at a Time

The generate function takes the current context (B, T) and extends it to (B, T+1), (B, T+2), … for max_new_tokens steps. At each step we take the logits for the last time step, convert them to probabilities with softmax, sample one index with torch.multinomial, and append it.

generate — autoregressive samplingdef generate(self, idx, max_new_tokens): for _ in range(max_new_tokens): logits, loss = self(idx) # (B, T, C) logits = logits[:, -1, :] # last step -> (B, C) probs = F.softmax(logits, dim=-1) idx_next = torch.multinomial(probs, num_samples=1) # (B, 1) idx = torch.cat((idx, idx_next), dim=1) # (B, T+1) return idx # kick off generation from a single newline character (index 0) idx = torch.zeros((1, 1), dtype=torch.long) print(decode(m.generate(idx, max_new_tokens=100)[0].tolist()))
The function is written to feed the whole history into the model even though the bigram only needs the last character. That looks wasteful now, but we keep it general so the exact same code works later when attention actually uses the history.
8 · Training the Model

🏋️ The Optimization Loop

We use the AdamW optimizer — more advanced and popular than plain SGD. For tiny networks a high learning rate like 1e-3 works fine. The loop is standard: sample a batch, evaluate the loss, zero the gradients, backprop, and step.

Training loopoptimizer = torch.optim.AdamW(m.parameters(), lr=1e-3) for step in range(10000): xb, yb = get_batch('train') logits, loss = m(xb, yb) optimizer.zero_grad(set_to_none=True) loss.backward() optimizer.step()

The bigram model drops from ~4.7 down to about 2.5. The output is no Shakespeare, but it is recognizably structured text — clearly better than the random garbage we started with. The fundamental limitation is obvious: the tokens are not talking to each other.

📊 Estimating Loss Robustly

Printing the loss of a single batch is noisy. A cleaner approach averages the loss over many batches for both splits, wrapped in @torch.no_grad() (so PyTorch skips storing intermediates) and with the model set to eval() mode and back to train() afterwards — good practice even when, as here, there are no Dropout or BatchNorm layers yet.

estimate_loss@torch.no_grad() def estimate_loss(): out = {} model.eval() for split in ['train', 'val']: losses = torch.zeros(eval_iters) for k in range(eval_iters): X, Y = get_batch(split) logits, loss = model(X, Y) losses[k] = loss.item() out[split] = losses.mean() model.train() return out
9 · The Math Trick Behind Self-Attention

🧮 Weighted Averages via Matrix Multiply

Before self-attention, we need one mathematical trick. We want each token to gather information only from itself and the past (never the future — we are predicting the future). The simplest "communication" is to average all preceding tokens. The naive version is a slow double loop; the trick is to do it with a single matrix multiplication.

Lower-triangular weights → weighted averagea = tril(ones(3,3)) b = [[2,7], c = a @ b a / a.sum(1, keepdim) [6,4], = [[2.0, 7.0 ], [6,5]] [4.0, 5.5 ], a = [[1.00, 0, 0 ], [4.67, 5.33]] [0.50, 0.50, 0 ] each row of c is the [0.33, 0.33, 0.33]] running average of b's rows

A lower-triangular matrix of ones (torch.tril), normalized so each row sums to 1, performs an incremental average when multiplied by any matrix. Row t of the result is the average of rows 0…t of the input — exactly "average my past".

Applied to our (B, T, C) tensor, PyTorch broadcasts a (T, T) weight matrix into a batched matrix multiply, producing the same per-batch averaging in parallel. Three versions are equivalent — the third uses softmax and is the one that generalizes to attention:

Version 3 — softmax form (the one we keep)tril = torch.tril(torch.ones(T, T)) wei = torch.zeros((T, T)) wei = wei.masked_fill(tril == 0, float('-inf')) # block the future wei = F.softmax(wei, dim=-1) # normalize -> averages xbow = wei @ x
The wei matrix starts at zero (uniform averaging), but these weights are really affinities — how interesting each past token is. Setting future positions to -inf before softmax means "do not aggregate from the future." Soon these affinities will become data-dependent instead of constant — and that is self-attention.
10 · A Single Self-Attention Head

🔑 Queries, Keys & Values

We don't want uniform averaging — different tokens find different past tokens more or less interesting, and that should be data-dependent. A vowel might want to look back for consonants. This is the problem self-attention solves.

Every token emits two vectors: a query ("what am I looking for") and a key ("what do I contain"). The affinity between two tokens is the dot product of one's query with the other's key. Aligned query/key pairs interact strongly, so that token's information flows in more.

Scaled dot-product attention
Attention(Q, K, V) = softmax( (Q · KT) / √dk ) · V
Q queries · K keys · V values · dk head size · the mask sets future positions to −∞ before softmax.
Single head of self-attentionclass Head(nn.Module): def __init__(self, head_size): super().__init__() self.key = nn.Linear(n_embd, head_size, bias=False) self.query = nn.Linear(n_embd, head_size, bias=False) self.value = nn.Linear(n_embd, head_size, bias=False) self.register_buffer('tril', torch.tril(torch.ones(block_size, block_size))) def forward(self, x): B, T, C = x.shape k = self.key(x) # (B, T, head_size) q = self.query(x) # (B, T, head_size) # compute affinities, scaled by 1/sqrt(head_size) wei = q @ k.transpose(-2, -1) * C**-0.5 # (B, T, T) wei = wei.masked_fill(self.tril[:T, :T] == 0, float('-inf')) wei = F.softmax(wei, dim=-1) v = self.value(x) # (B, T, head_size) return wei @ v # (B, T, head_size)

We aggregate values v, not the raw token vectors x. Think of x as private to the token; v is "here's what I'll tell you if you find me interesting." The √dk scaling keeps the variance of wei around 1 so the softmax stays diffuse at init — without it, large values make softmax collapse toward a one-hot vector, aggregating from a single node.

📐 Six Notes on Attention

  • Communication mechanism. Attention is a directed graph where each node aggregates a weighted sum of the nodes pointing to it, data-dependently.
  • No notion of space. Attention acts on a set of vectors; it has no idea of position, which is why we add positional embeddings.
  • Batch elements are independent. The batch dimension never talks to itself — it's a batched matrix multiply.
  • Decoder vs encoder. The triangular mask makes this a decoder block (autoregressive). Delete the mask and all nodes talk freely — an encoder block (e.g. for sentiment analysis).
  • Self vs cross attention. Here Q, K, V all come from the same x (self-attention). In cross-attention the keys/values come from a separate source (e.g. an encoder).
  • Positional encoding required. Because attention is permutation-invariant, position must be injected explicitly.

We also add a second embedding table for positions, and feed the sum of token and positional embeddings into the head:

Token + position embeddingsself.token_embedding_table = nn.Embedding(vocab_size, n_embd) self.position_embedding_table = nn.Embedding(block_size, n_embd) tok_emb = self.token_embedding_table(idx) # (B, T, C) pos_emb = self.position_embedding_table(torch.arange(T)) # (T, C) x = tok_emb + pos_emb # (B, T, C)

With one attention head, the validation loss improves from ~2.5 to about 2.4. The head is doing useful communication, but there is a long way to go.

11 · Multi-Head Attention

🧠 Many Communication Channels in Parallel

Multi-head attention simply runs multiple attention heads in parallel and concatenates their results over the channel dimension. With n_embd = 32, four heads of size 8 each concatenate back to 32 — like a group convolution. Multiple heads give multiple independent channels: one can hunt for vowels, another for consonants at certain positions, and so on.

Multi-head attentionclass MultiHeadAttention(nn.Module): def __init__(self, num_heads, head_size): super().__init__() self.heads = nn.ModuleList([Head(head_size) for _ in range(num_heads)]) self.proj = nn.Linear(n_embd, n_embd) # projection back to residual pathway def forward(self, x): out = torch.cat([h(x) for h in self.heads], dim=-1) return self.proj(out)

Four communication channels bring the validation loss down to about 2.28.

12 · Feed-Forward Layers

⚙️ Time to Think

After attention does the communication, the tokens jump too quickly to the logits — they gather information but never think about it. The feed-forward network is a simple per-token MLP (linear → ReLU → linear) that lets each token process what it gathered, independently. The paper uses an inner expansion factor of 4×.

Feed-forward (per-token MLP)class FeedForward(nn.Module): def __init__(self, n_embd): super().__init__() self.net = nn.Sequential( nn.Linear(n_embd, 4 * n_embd), nn.ReLU(), nn.Linear(4 * n_embd, n_embd), # projection back ) def forward(self, x): return self.net(x)

Calling feed-forward right after self-attention nudges the validation loss down to ~2.24. The pattern emerging is communicate, then compute — repeated.

🔁 Interspersing Communication and Computation

🛣️ Residual Connections

Residual (skip) connections add the input back after each sub-layer. You fork off the residual pathway, do some computation, and add the result back. During backprop, addition distributes gradients equally to both branches, creating a gradient super-highway from the loss straight to the input. The residual blocks start out contributing almost nothing and "come online" during training.

📏 LayerNorm (Pre-Norm)

LayerNorm normalizes each token's feature vector to zero mean and unit variance (per row), with learnable gamma and beta. Unlike BatchNorm, it needs no running buffers and no train/test distinction. Modern practice applies it before each sub-layer — the pre-norm formulation, a small deviation from the original paper.

Transformer block (pre-norm + residual)class Block(nn.Module): def __init__(self, n_embd, n_head): super().__init__() head_size = n_embd // n_head self.sa = MultiHeadAttention(n_head, head_size) self.ffwd = FeedForward(n_embd) self.ln1 = nn.LayerNorm(n_embd) self.ln2 = nn.LayerNorm(n_embd) def forward(self, x): x = x + self.sa(self.ln1(x)) # communicate (+ residual) x = x + self.ffwd(self.ln2(x)) # compute (+ residual) return x

With residuals and the 4× feed-forward we reach ~2.08, and adding LayerNorm (plus a final LayerNorm before the output linear layer) brings it to ~2.06. The training loss now slightly leads validation — a first hint of overfitting.

14 · Dropout & Scaling Up

💧 Regularization Before Going Big

Dropout randomly zeroes a subset of activations on every forward/backward pass, effectively training an ensemble of sub-networks that merge at test time. We add it before each residual re-entry and after the softmax affinities. It's a regularizer we switch on because scaling up risks overfitting.

15 · Decoder-Only vs Encoder-Decoder

🏗️ Why Our Architecture Looks Different

We built a decoder-only Transformer — exactly what GPT uses. The original paper has an encoder-decoder architecture because it solves machine translation. The differences:

Decoder (ours / GPT)
  • Triangular mask → autoregressive
  • Future cannot see past
  • Unconditioned generation
  • Self-attention + feed-forward only
Encoder-Decoder (translation)
  • Encoder: no mask, all tokens talk
  • Encodes a source (e.g. French)
  • Cross-attention feeds encoder → decoder
  • Conditions output on the source

We have nothing to condition on — just a text file to imitate — so we drop the encoder and the cross-attention block entirely. The triangular mask is what makes ours a decoder usable for language modeling.

16 · nanoGPT & The Road to ChatGPT

📦 nanoGPT

The production-quality version of this is nanoGPT — two files: model.py (the Transformer, nearly identical to ours) and train.py (checkpointing, learning-rate decay, distributed training). The main difference is that nanoGPT batches all heads into a single tensor (adding a 4th "head" dimension) for efficiency, and uses GELU instead of ReLU to match OpenAI's checkpoints.

🪜 Pre-training vs Fine-Tuning

Our model is a tiny pre-training step. Real ChatGPT is built in two stages:

Fine-tuning has roughly three steps: (1) supervised fine-tuning on question→answer documents; (2) raters rank model responses to train a reward model; (3) PPO reinforcement learning optimizes the policy to score high reward. This data is largely internal to OpenAI and much harder to replicate — nanoGPT focuses on stage 1.
