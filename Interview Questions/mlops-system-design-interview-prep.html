Interview Questions · MLOps · System Design

MLOps / System Design Interview Prep

The round where breadth and structure matter more than depth on any one piece — a repeatable framework, the key MLOps and LLM concepts, three fully worked designs, and what interviewers are actually scoring.

1 · The Framework

🧭 Your Repeatable Seven-Step Script

Memorize this flow and narrate it out loud every time. The discipline of walking it signals seniority even if you're light on a specific tool.

  1. Clarify Requirements Always start here; never jump to the model.
    • Functional: What exactly does the system do? Who's the user? What's the output?
    • Non-functional: Latency (p50/p99), throughput (QPS), scale (users, items, requests/day), availability SLA, cost budget.
    • Scoping questions: "Is this real-time or can it be batch?" "What's the acceptable latency?" "What's the scale — thousands or billions of items?" "Is there a human in the loop?"
  2. Data — sources, volume, labels, quality, drift
    • Where does data come from (logs, clickstream, transactions, user-generated)?
    • How are labels obtained (explicit feedback, implicit signals, human annotation, weak supervision)?
    • Class imbalance, label delay, data freshness, privacy/PII concerns.
  3. Features — engineering + online/offline store
    • Offline store (training, batch, historical) vs online store (low-latency serving).
    • The critical concept: train-serve skew — features must be computed identically in both, which is why feature stores exist.
    • Point-in-time correctness to avoid label leakage.
  4. Model — baseline first, then complexity
    • Always state: "I'd start with a simple baseline" (logistic regression, popularity-based, heuristic) to establish a reference and validate the pipeline end-to-end.
    • Then iterate to complex (gradient-boosted trees, deep models, two-tower, transformers).
    • Training infra: distributed training, hardware, frequency.
  5. Serving — match the pattern to the latency budget
    • Batch: precompute predictions offline (e.g., daily recommendations). Cheapest, highest latency tolerance.
    • Real-time / online: compute on request (fraud, ad CTR). Needs low-latency feature fetch + fast model.
    • Streaming: continuous processing of event streams.
    • Mention A/B testing, shadow deployment, and canary releases for safe rollout.
  6. Monitoring — name all four layers explicitly
    • Data drift — input distribution changes (PSI, KS test).
    • Model / concept drift — relationship between X and y changes (performance decay).
    • Operational — latency, error rate, throughput.
    • Business metrics — the metric that actually matters (revenue, fraud caught, engagement).
  7. Iteration — close the loop Retraining cadence (scheduled vs triggered by drift), feedback collection, continuous evaluation, and the experiment → deploy → monitor cycle.
Pro move: Open every answer with — "Let me clarify requirements first, then I'll walk through data, features, modeling, serving, monitoring, and iteration." This frames you as systematic before you've said anything technical.
2 · Key MLOps Concepts

🧰 Talking Points You Should Have Ready

Experiment tracking (MLflow, Weights & Biases)
Log hyperparameters, metrics, artifacts, and code versions for every run so experiments are reproducible and comparable. Answers "which config produced this model?"
Feature store (Feast, Tecton)
Centralized system serving consistent features to both training (offline) and inference (online), solving train-serve skew and enabling feature reuse across teams.
Model registry
Versioned catalog of trained models with stage tags (staging/production/archived), lineage, and metadata. Enables controlled promotion and rollback.
Data versioning (DVC)
Version control for datasets and pipelines so you can reproduce exactly which data trained a model — critical for debugging and compliance.
CI/CD for ML
Beyond code, it includes data validation, automated retraining, model testing (not just unit tests but model quality gates), and automated deployment. Often called CT (continuous training).
Deployment strategies
  • Shadow deployment: new model runs alongside production receiving real traffic, but its predictions aren't served — you compare silently against the live model with zero user risk.
  • Canary release: route a small % of traffic to the new model, monitor, then gradually ramp up.
  • A/B testing: split traffic to measure causal impact on business metrics.
  • Blue-green: maintain two environments, switch traffic instantly, roll back fast.
Drift detection
  • PSI (Population Stability Index): measures distribution shift between two periods; common thresholds — <0.1 stable, 0.1–0.25 moderate, >0.25 significant shift.
  • KS test (Kolmogorov–Smirnov): statistical test comparing two distributions for continuous features.
  • Concept drift: track prediction quality over time as labels arrive.
Observability
Logging, metrics, tracing, and alerting across the full pipeline — knowing why something broke, not just that it broke.
3 · LLM-Specific MLOps

✨ The Hot Section — Emphasize This

LLM systems break some traditional MLOps assumptions (no labels, non-deterministic outputs, expensive inference). Lead with these.

Prompt versioning & management
Treat prompts as versioned artifacts (like code) since small prompt changes drastically alter behavior. Track which prompt version produced which output.
Eval pipelines
Since outputs are open-ended, use a mix of automated metrics, LLM-as-judge, golden test sets / regression suites, and human eval. Run evals in CI before deploying prompt or model changes to catch regressions.
Guardrails
Input filtering (block injection, PII, toxic input) and output filtering (block hallucinations, unsafe content, PII leakage, off-topic responses). Often a separate validation layer or smaller classifier model.
Semantic caching
Cache responses keyed by semantic similarity of queries (via embeddings), not exact match, so paraphrased questions hit the cache — cutting cost and latency significantly.
Cost tracking
Monitor token usage per request / user / feature; LLM costs scale with tokens, so observability on input/output token counts is essential for unit economics.
Rate limiting
Protect against abuse and runaway costs; manage provider quotas.
Fallback chains
If the primary model fails, times out, or hits rate limits, gracefully degrade to a cheaper model, cached response, or a default — ensuring reliability.
4 · Worked Design A — RAG Chatbot for Enterprise Docs

📚 The Most Likely LLM Question

Requirements
Functional — employees ask natural-language questions, get accurate answers grounded in internal docs with citations. Non-functional — p99 latency a few seconds, handle concurrent users, strict access control (answers must respect document permissions), no hallucination of sensitive policy.
Data
Sources are wikis, PDFs, tickets, Slack, Confluence. Ingestion pipeline handles parsing, deduplication, and chunking. Chunking strategy matters most — overlapping chunks of a few hundred tokens, respecting semantic/section boundaries. Attach metadata (source, permissions, timestamp) to each chunk.
Features / Indexing
Embed chunks with an embedding model, store in a vector DB (Pinecone, Weaviate, pgvector). Consider hybrid search (dense vectors + BM25 keyword) for better recall on exact terms like error codes.
Model / Retrieval
Retriever pulls top-k chunks → optional reranker (cross-encoder) reorders for relevance → augment prompt with retrieved context + citations → LLM generates grounded answer. Baseline first: simple top-k vector search with an off-the-shelf LLM, then add reranking and query rewriting.
Serving
Real-time. Pipeline: query → (rewrite) → embed → retrieve → rerank → generate. Apply access control by filtering retrieval to documents the user can see. Add semantic caching for repeated questions and a fallback model if the primary is down.
Monitoring
Retrieval quality (are relevant chunks retrieved?), groundedness/faithfulness (does the answer stick to sources?), hallucination rate, latency, cost per query, and user feedback (thumbs up/down). Run an eval suite on a golden Q&A set before any prompt/model change.
Iteration
Re-embed and re-index on doc updates (incremental), refine chunking and prompts based on failure analysis, fine-tune retriever/reranker on collected feedback. Address freshness via scheduled or event-triggered re-ingestion.
Top trap to mention: RAG quality is usually a retrieval problem, not a generation problem — if the right chunk isn't retrieved, the LLM can't answer. So invest in chunking, hybrid search, and reranking first.
5 · Worked Design B — Fraud Detection

🛡️ Real-Time, Imbalanced, Adversarial

Requirements
Functional — flag fraudulent transactions in real time, allow/block/review decision. Non-functional — very low latency (block before transaction completes, ~tens of ms), extreme class imbalance (fraud is <1%), high cost of false negatives (missed fraud) and false positives (blocking legit customers), adversarial (fraudsters adapt).
Data
Transaction logs, user history, device/IP, merchant data. Labels are delayed and noisy — chargebacks arrive weeks later, and many frauds go unlabeled. Use confirmed fraud + chargebacks as positives; handle label delay explicitly.
Features
Heavy feature engineering — velocity features (transactions per hour), aggregations (avg spend, deviation from normal), device/location anomalies. Online feature store is essential for real-time low-latency lookups; offline store computes historical aggregates. Train-serve consistency is critical here.
Model
Baseline — logistic regression or rules. Then gradient-boosted trees (XGBoost) which dominate tabular fraud, possibly with graph features or anomaly detection for novel patterns. Handle imbalance via class weighting, focal loss, or careful sampling — and evaluate with PR-AUC, not accuracy.
Serving
Real-time scoring within the transaction flow. Tiered approach: fast rules block obvious fraud, model scores the rest, ambiguous cases go to manual review queue. Set the decision threshold by business cost tradeoff, not a default 0.5.
Monitoring
Precision/recall over time (with label delay accounting), drift in transaction patterns (PSI on key features), fraud rate by segment, false-positive rate (customer friction), and dollar impact. Alert on sudden distribution shifts (could be an attack).
Iteration
Frequent retraining because fraud is adversarial and concept drift is fast. Feedback loop from the manual review team provides fresh labels. Watch for feedback bias — you only get labels on transactions you allowed, so blocked-transaction labels are missing (selection bias).
6 · Worked Design C — Ad CTR / Recommendation Ranking

🎯 Massive Scale, Tight Latency, Calibration

Requirements
Functional — predict probability a user clicks an ad/item, rank candidates. Non-functional — extremely high QPS, very tight latency budget (single-digit ms for ranking), billions of items and users, must update fast as new items/ads appear.
Data
Impression and click logs (implicit feedback). Massive scale, sparse high-cardinality categorical features (user IDs, ad IDs). Positive label = click; handle that non-clicks aren't necessarily negatives (position bias — top items get more clicks regardless of relevance).
Features
User features, item/ad features, context (time, device), and crosses/interactions. Embeddings for high-cardinality IDs. Online store for real-time features, offline for training. Point-in-time correctness essential.
Model
Standard architecture is candidate generation → ranking. Retrieval narrows billions to hundreds (two-tower model, ANN search); ranking scores those precisely (gradient-boosted trees or deep models like Wide & Deep, DLRM). Baseline: logistic regression on engineered features — historically strong for CTR.
Serving
Two-stage real-time. Retrieval via approximate nearest neighbor; ranking model scores the shortlist under the latency budget. Precompute embeddings where possible. A/B test every change against business metrics.
Monitoring
CTR/conversion (business metric), AUC/log-loss (model metric), calibration (predicted probabilities must match observed rates — critical for ad auctions/pricing), latency, and feature/prediction drift. Calibration is a CTR-specific must-mention.
Iteration
Continuous/frequent retraining (often daily or online) because trends and inventory shift constantly. Address position/selection bias with techniques like inverse propensity weighting. Online learning for freshness.
7 · What Interviewers Are Scoring

📝 The Signals That Differentiate Candidates

  • Did you clarify before designing? Jumping to "I'll use a transformer" is the #1 junior signal. Ask about scale, latency, and labels first.
  • Did you propose a baseline before complexity? This is a strong seniority signal — it shows you value iteration and validation over hype.
  • Did you reason about tradeoffs out loud? Batch vs real-time, precision vs recall, cost vs latency, accuracy vs interpretability. There's rarely one right answer; they want your reasoning.
  • Did you cover monitoring and iteration? Most candidates stop at "deploy the model." Talking about drift, retraining, and feedback loops sets you apart.
  • Did you name the domain-specific trap? Calibration for CTR, retrieval-not-generation for RAG, label delay / feedback bias for fraud, cold-start for recommendations. One sharp domain insight signals real experience.
